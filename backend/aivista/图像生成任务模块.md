# AiVista 图像生成任务模块

> 对应迭代：迭代 2——文生图与局部重绘任务  
> 当前状态：详细设计中  
> 本文档只记录已确认的设计决策与待确认项；未实现内容不得视为已交付。

## 1. 目标与范围

本模块负责创建、执行、追踪和通知 AI 图像生成任务，并将生成结果可靠存储到项目自有的阿里云 OSS。

当前优先实现的能力：

- 用户创建普通文生图任务。
- 支持用户创建、切换和恢复多个持久化生成会话。
- 同一会话同一时间仅允许一个活动生成任务；不同会话可独立排队或生成。
- 持久化会话内的对话历史，并为后续模型调用保留上下文记忆。
- 后端异步调用阿里云百炼 Qwen-Image。
- 一次任务可生成的图片数量由实际调用模型的能力决定。
- 将图片从服务商临时地址转存至私有 OSS。
- 用户通过 SSE 获得任务状态变化通知，并通过 REST 查询任务详情。

后续在同一任务模型中扩展：

- 局部重绘任务。
- 用户主动选择的 Agent / 工作流执行模式。

不属于当前实现范围：

- 局部重绘具体输入、遮罩和编辑 API。
- Agent / 工作流定义、编排、版本管理和界面。
- 社区发布、收藏、点赞、关注和完整生成历史浏览。
- 用户计费、订单和付费配额。

## 2. 已确认设计决策

| 主题 | 决策 |
| --- | --- |
| 图像服务 | 使用阿里云百炼 Qwen-Image API。 |
| 默认模型 | 当前固定使用 `qwen-image-2.0-pro`；每个任务保存实际调用的模型标识。 |
| 存储 | 服务商返回的临时图片 URL 由后端下载并转存至私有阿里云 OSS，不作为长期资产地址。 |
| 任务类型 | `TEXT_TO_IMAGE`、`INPAINT`；当前仅开放 `TEXT_TO_IMAGE`。 |
| 执行模式 | `DIRECT`、`WORKFLOW`；当前仅开放 `DIRECT`。`WORKFLOW` 未来由用户主动选择。 |
| 执行方式 | 所有模式统一由后端异步任务系统执行；`DIRECT` 仅表示直接调用模型服务，不表示 HTTP 请求同步等待结果。 |
| 会话规则 | 用户可拥有多个持久化生成会话；同一会话同时只允许一个活动任务，多个会话可独立排队或生成。 |
| 历史与上下文 | 当前 `DIRECT` 文生图的一轮历史由用户提示词消息和 AI 生成结果消息组成；结果消息不含文本，通过关联任务取得状态和图片。模型调用前由应用基于会话历史组装上下文。模型接口本身不保留多轮记忆，具体上下文编排策略待单独设计。 |
| 实时通知 | 使用 SSE 将任务状态变化主动推送给在线用户；REST 接口负责详情查询和断线后的状态对账。 |
| 首版部署 | 首版按单个 Spring Boot 应用实例设计：同一进程处理 HTTP/SSE 请求并消费 RabbitMQ 任务，应用内直接向 SSE 连接推送事件，不启用 Redis Pub/Sub；扩展为多个应用实例或独立工作器时，再通过 Redis Pub/Sub 跨实例分发事件。 |
| 任务真相来源 | MySQL 任务与结果记录是权威状态；SSE 仅为实时通知通道。 |
| 服务商密钥 | 百炼 API Key 仅保存在服务端配置中，绝不下发客户端或记录到日志。 |

阿里云 Qwen-Image 2.0 Pro 支持文生图与图像编辑，可生成 1～6 张 PNG。前端向用户提供 1～6 张的数量选择；服务端也必须校验该请求范围，防止绕过前端直接调用 API 传入非法参数。模型调用本身为同步方式，因此由后台工作器包装为异步任务执行。详见[阿里云 Qwen-Image API 文档](https://help.aliyun.com/zh/model-studio/qwen-image-api)。

### 2.1 千问 Qwen-Image 调用参考

> 官方文档：[千问 Qwen-Image 文生图 API](https://help.aliyun.com/zh/model-studio/qwen-image-api)（访问与实现前应再次核对模型可用地域、版本、限流和计费规则）。

当前 `DIRECT` 执行器调用百炼的同步多模态生成接口；项目层面的异步由 RabbitMQ 工作线程负责，不应将本项目任务状态机与百炼异步接口混用。

| 项目 | 约定 |
| --- | --- |
| 默认模型 | `qwen-image-2.0-pro`；当前固定使用该模型，每次调用均保存实际模型标识。该模型支持同步接口。 |
| 北京地域端点 | `POST https://{WorkspaceId}.cn-beijing.maas.aliyuncs.com/api/v1/services/aigc/multimodal-generation/generation` |
| 鉴权 | 请求头 `Authorization: Bearer ${DASHSCOPE_API_KEY}`；API Key 与地域对应，严禁写入客户端、代码仓库或日志。 |
| 请求头 | `Content-Type: application/json`。 |
| 正向提示词 | `input.messages` 必须且只能包含一条 `user` 消息；其 `content` 必须且只能包含一个 `text`。`qwen-image-2.0` 系列上限为 1300 Token，服务端应在提交前校验项目自身长度限制。 |
| 反向提示词 | `parameters.negative_prompt` 可选，最长 500 字符；项目接口字段 `negativePrompt` 映射至该参数。 |
| 尺寸 | `parameters.size` 格式为 `宽*高`。`qwen-image-2.0` 系列总像素范围为 512×512 至 2048×2048；本项目仅允许预定义的宽高比映射值。 |
| 生成数量 | `parameters.n` 为 1～6；前端提供该范围选择，服务端执行同范围校验。 |
| 提示词改写 | `parameters.prompt_extend` 官方默认 `true`；当前普通模式固定传 `false`，以保证保存的原始提示词与实际输入一致。 |
| 返回图片 | 服务商返回临时公开 URL；工作线程必须下载、校验并转存至私有 OSS 后，才可将任务置为 `SUCCEEDED`。 |

同步请求体结构如下。实现时只替换模型、提示词和已校验的业务参数：

```json
{
  "model": "qwen-image-2.0-pro",
  "input": {
    "messages": [
      {
        "role": "user",
        "content": [{ "text": "正向提示词" }]
      }
    ]
  },
  "parameters": {
    "negative_prompt": "反向提示词",
    "size": "2048*2048",
    "n": 1,
    "prompt_extend": false,
    "watermark": false
  }
}
```

注意：百炼的异步文生图接口当前仅适用于 `qwen-image-plus` 和 `qwen-image`，不适用于 `qwen-image-2.0-pro`。若未来切换到百炼异步接口，需重新设计服务商任务 ID 的轮询、超时与重试策略，不复用当前同步执行器的调用逻辑。

## 3. 核心架构

```text
创建任务 API
    ↓
MySQL：任务记录 + Outbox 事件（同一事务）
    ↓
RabbitMQ：生成任务队列
    ↓
任务工作器：DIRECT 执行器调用百炼 Qwen-Image
    ↓
下载临时结果 → 上传私有 OSS → 写入结果图片与任务状态
    ↓
MySQL：Outbox 状态事件（同一事务）
    ↓
Redis Pub/Sub：在线通知分发
    ↓
SSE：向所属用户推送任务状态变化
```

组件职责：

| 组件 | 职责 |
| --- | --- |
| MySQL | 保存任务、图片、Outbox 事件；是唯一权威状态来源。 |
| RabbitMQ | 承载生成任务、确认、重试、延迟与死信处理。 |
| 阿里云百炼 | 生成图像。 |
| 阿里云 OSS | 保存项目拥有的生成结果文件。 |
| Redis Pub/Sub | 向持有 SSE 连接的应用节点分发实时通知；首版单实例不启用，多实例部署时启用；不作为持久化消息来源。 |
| SSE | 向浏览器单向推送任务变化。 |

采用 Transactional Outbox：任务状态与待分发事件必须在同一数据库事务内写入。事件分发失败可重试，避免“任务已成功但用户永远收不到通知”的状态不一致。

## 4. 任务模型

### 4.1 类型与模式

```text
task_type
- TEXT_TO_IMAGE
- INPAINT

execution_mode
- DIRECT
- WORKFLOW
```

当前允许的组合只有 `TEXT_TO_IMAGE + DIRECT`。未来扩展组合时不修改任务基础生命周期与结果模型。

### 4.2 会话与任务规则

- 会话是用户可持久化恢复的生成对话单元；用户离开页面、关闭浏览器或切换会话不得中断已创建的后台任务。
- 每个生成任务必须归属于一个会话；同一会话在 `QUEUED`、`RUNNING` 或 `CANCEL_REQUESTED` 状态下，不得创建第二个活动任务。
- 不同会话可分别创建任务。受用户级并发与工作器容量影响，任务可并行执行或在队列中等待，但会话切换不影响任务执行。
- 会话历史是对话记忆的权威来源。当前 `DIRECT` 文生图的一轮历史由用户提示词消息和 AI 生成结果消息构成；结果消息关联任务，由任务再关联状态和结果图片。每次调用模型时，应用需要保存该次任务使用的上下文快照和最终调用提示词，以便历史恢复和结果追溯。
- 当前文生图模型调用为单轮调用，不会天然记住上一轮内容，也不产生 AI 文本内容；AI 生成结果消息只承载任务引用和图片结果。“上下文记忆”由应用层组织，具体采用历史拼接、摘要还是独立文本模型编排，待数据模型与提示词策略一并确认。
- Agent / 工作流模式未来可产生思考过程或文本回复；相关消息类型和内容在该模式落地时启用，不影响当前 `DIRECT` 文生图的记录方式。

### 4.3 状态机

```text
QUEUED
  ↓
RUNNING
  ├── SUCCEEDED
  ├── PARTIALLY_SUCCEEDED
  ├── FAILED
  └── CANCEL_REQUESTED → CANCELLED / CANCELLED_WITH_RESULTS
```

规则：

- 状态只允许按既定方向迁移，终态不可再次执行。
- 请求数量的全部图片均成功转存 OSS 时，任务进入 `SUCCEEDED`；至少一张成功转存、但未全部转存时进入 `PARTIALLY_SUCCEEDED`；没有任何图片成功转存时进入 `FAILED`。
- 用户取消时先进入 `CANCEL_REQUESTED`。若尚无图片完成转存，进入 `CANCELLED`；若取消前已有至少一张图片完成转存，保留已完成图片并进入 `CANCELLED_WITH_RESULTS`，后续未转存结果不再处理。模型调用可能已经产生费用。
- 仅“确认请求尚未发送到服务商”的网络失败可自动重试。请求可能已发出但发生超时、连接中断等结果未知情况时，不得自动重试，以避免重复生成与重复计费；任务进入 `FAILED`，内部失败码记为 `PROVIDER_RESULT_UNKNOWN`，对用户仅返回安全的失败说明。
- 参数错误、内容违规等不可重试失败直接进入 `FAILED`。
- 每次状态变化递增 `task_version`，客户端仅接受比本地版本新的事件。

### 4.4 核心数据表

以下资源主键统一使用 `BIGINT UNSIGNED` 自增；对外 HTTP 响应统一序列化为字符串。所有时间使用 `DATETIME`，创建和更新时间由数据库维护。

#### `generation_sessions`：生成会话

| 字段 | 规则与职责 |
| --- | --- |
| `id` | 会话 ID。 |
| `user_id` | 所属用户 ID，外键关联 `users.id`。 |
| `title` | 会话名称，`VARCHAR(100)`；首版由用户创建时提供或使用默认名称。 |
| `context_summary` | 当前会话的上下文摘要，`LONGTEXT`，可空；具体内容格式随上下文编排策略确定。 |
| `context_version` | 上下文版本号，`INT UNSIGNED`，用于识别摘要更新。 |
| `active_task_id` | 当前活动任务 ID，可空；创建任务时原子设置，任务进入终态时清空，用于保证同一会话只有一个活动任务。 |
| `last_message_at` | 最近一条消息时间，用于用户会话列表排序。 |
| `created_at` / `updated_at` | 会话创建与更新时间。 |

索引与约束：`idx_generation_sessions_user_last_message(user_id, last_message_at)`；`active_task_id` 仅可指向该会话的活动任务，具体外键与事务实现随迁移脚本确定。

#### `generation_messages`：会话历史消息

| 字段 | 规则与职责 |
| --- | --- |
| `id` | 消息 ID。 |
| `session_id` | 所属生成会话 ID，外键关联 `generation_sessions.id`。 |
| `sequence_no` | 会话内递增序号，`INT UNSIGNED`；用于稳定恢复消息顺序。 |
| `role` | 当前 `DIRECT` 模式使用 `USER` 和 `ASSISTANT`：前者表示用户提示词，后者表示 AI 生成结果；`SYSTEM` 为系统消息预留。 |
| `message_type` | 当前使用 `PROMPT` 和 `GENERATION_RESULT`；`TEXT`、`TASK_STATUS` 等类型为后续 Agent / 工作流预留。 |
| `content` | 文本内容，`LONGTEXT`，可空；`PROMPT` 保存用户提示词，当前 `GENERATION_RESULT` 可为空。Agent / 工作流模式的 AI 文本回复未来保存在此字段。 |
| `task_id` | 关联生成任务 ID，可空；当前 AI 生成结果消息必须关联一个任务。图片 ID 通过 `task_id → generation_images` 获取，不重复存储。 |
| `created_at` | 消息创建时间；历史消息创建后不修改。 |

索引与约束：唯一索引 `uk_generation_messages_session_sequence(session_id, sequence_no)`；索引 `idx_generation_messages_session_created(session_id, created_at)`。

创建任务时，同一事务中写入用户 `PROMPT` 消息、关联任务和一条 `GENERATION_RESULT` AI 消息。结果消息在生成中即存在，客户端通过其 `task_id` 展示任务状态；任务完成后，通过 `task_id → generation_images` 取得 1～6 张图片。消息表不保存 OSS 签名 URL；图片记录只保存私有对象定位信息，客户端再按权限获取短期 URL。

#### `generation_tasks`：异步生成任务

| 字段 | 规则与职责 |
| --- | --- |
| `id` | 任务 ID。 |
| `user_id` / `session_id` | 所属用户与会话；`session_id` 关联 `generation_sessions.id`。 |
| `source_message_id` | 触发本次生成的用户提示词消息 ID。 |
| `task_type` / `execution_mode` | 图像任务类型与执行模式。 |
| `provider` / `model` | 服务商标识与实际调用模型标识。 |
| `status` / `task_version` | 当前任务状态，以及乐观锁和 SSE 事件去重版本。 |
| `original_prompt` / `negative_prompt` | 用户本次原始正向、负向提示词快照。 |
| `context_snapshot` | 本次生成使用的会话上下文快照，`JSON`；具体结构待上下文编排策略确定。 |
| `final_prompt` | 最终传给图像模型的正向提示词，`LONGTEXT`；用于生成追溯。 |
| `width` / `height` | 实际请求的输出尺寸。 |
| `requested_image_count` / `completed_image_count` | 请求生成数量与实际成功转存数量。 |
| `provider_request_id` | 服务商请求追踪标识。 |
| `idempotency_key` / `request_fingerprint` | 创建请求幂等键与请求参数摘要。 |
| `failure_code` / `failure_message` | 对用户安全的失败分类与说明。 |
| `created_at` / `updated_at` / `started_at` / `completed_at` | 任务生命周期时间。 |

索引与约束：唯一索引 `uk_generation_tasks_user_idempotency(user_id, idempotency_key)`；索引 `idx_generation_tasks_session_created(session_id, created_at)` 和 `idx_generation_tasks_user_status(user_id, status)`。

#### `generation_images`：生成结果图片

| 字段 | 规则与职责 |
| --- | --- |
| `id` | 图片 ID。 |
| `task_id` / `user_id` | 所属任务与用户。会话归属通过任务关联获得，不重复存储 `session_id`。 |
| `oss_bucket` / `object_key` | 私有 OSS 对象定位信息，永不保存短期签名 URL。 |
| `content_type` / `file_size` | 文件元数据。 |
| `width` / `height` | 实际图像尺寸。 |
| `sort_order` | 同一任务内的结果顺序。 |
| `created_at` | 保存时间。 |

索引与约束：唯一索引 `uk_generation_images_task_sort(task_id, sort_order)`；索引 `idx_generation_images_user_created(user_id, created_at)`，为迭代 3 的个人资产查询提供基础。

#### `outbox_events`：可靠事件分发

| 字段 | 规则与职责 |
| --- | --- |
| `id` | 事件 ID。 |
| `aggregate_type` / `aggregate_id` | 聚合类型与 ID；本模块主要为生成任务。 |
| `event_type` | 事件类型，例如任务状态更新。 |
| `payload` | 事件载荷，`JSON`。 |
| `status` / `retry_count` | 分发状态与重试次数。 |
| `available_at` / `published_at` | 下次可分发时间与成功分发时间。 |
| `created_at` | 事件创建时间。 |

## 5. 普通文生图参数

当前创建任务接口只接受以下业务参数：

| 参数 | 规则 | 服务商映射 |
| --- | --- | --- |
| `prompt` | 必填，最多 1000 个字符。 | 正向提示词。 |
| `negativePrompt` | 可选，最多 500 个字符。 | `negative_prompt`。 |
| `aspectRatio` | 必填，默认 `1:1`。 | 映射为固定 `size`。 |
| `imageCount` | 必填；前端仅提供 `1`～`6` 的选择，服务端校验后映射至 `n`。 | `n`。 |

尺寸映射：

| `aspectRatio` | `size` |
| --- | --- |
| `1:1` | `2048*2048` |
| `16:9` | `2688*1536` |
| `9:16` | `1536*2688` |
| `4:3` | `2368*1728` |
| `3:4` | `1728*2368` |

普通模式固定 `prompt_extend = false`，以使已保存的原始提示词与实际普通模式输入一致。工作流模式未来如改写提示词，必须额外保存原始提示词、最终提示词与工作流版本。

当前不向用户开放模型选择、随机种子、水印开关、任意宽高、自由工作流配置等参数。

## 6. OSS 存储与访问

生成图片必须在服务端完成转存：

1. 工作器从百炼返回的临时 URL 下载图像。
2. 校验 HTTP 响应成功、响应类型为图片、文件大小和图片元数据，避免临时 URL 失效或异常时将错误响应误存为图片。
3. 上传至私有 OSS。
4. 成功后写入 `generation_images`；全部图片转存成功时置为 `SUCCEEDED`，至少一张成功但未全部成功时置为 `PARTIALLY_SUCCEEDED`。

对象键规范：

```text
users/{userId}/generations/{taskId}/{imageId}.png
```

迭代 2 只保存私有 OSS 原图，客户端不直接获得永久 OSS 地址，任务详情接口按权限返回短期签名 URL。迭代 4 用户发布图片时，基于私有原图生成公开展示副本或缩略图，保存到公开 Bucket 或公开前缀，并通过固定 CDN/OSS URL 供灵感社区浏览；撤销发布时移除或禁止该公开副本访问，不影响私有原图。

## 7. API 与认证契约

所有接口使用 `/api` 前缀、统一响应结构和 Bearer Access Token 认证。

| 功能 | 方法与路径 | 认证 | 说明 |
| --- | --- | --- | --- |
| 创建普通文生图任务 | `POST /api/generation-tasks` | Bearer Token | 任务必须关联一个会话；要求 `Idempotency-Key` 请求头；返回 `202 Accepted` 与任务概要。具体会话参数待数据模型确认。 |
| 查询任务详情 | `GET /api/generation-tasks/{taskId}` | Bearer Token | 仅任务所属用户可查询；成功结果返回短期 OSS 签名 URL。 |
| 查询活动任务 | `GET /api/generation-tasks?active=true` | Bearer Token | 用于页面初始化与 SSE 重连后的状态对账，不替代后续完整历史资产查询。 |
| 取消任务 | `POST /api/generation-tasks/{taskId}/cancel` | Bearer Token | 仅允许取消非终态任务。 |
| 建立事件流 | `GET /api/events` | Bearer Token | `text/event-stream`；一个用户维持一条连接。 |

SSE 事件示例：

```text
event: generation.task.updated
id: 928
data: {"sessionId":"456","taskId":"123","taskVersion":4,"status":"SUCCEEDED"}
```

事件不携带图片签名 URL 或完整任务信息。客户端收到事件后请求任务详情；首次连接和重连后查询活动任务完成状态对账。

浏览器侧不得把 Access Token 写入 SSE URL 查询参数。由于当前认证使用 Bearer Token，事件流应使用支持 `Authorization` 请求头的 Fetch 流式 SSE 实现。Access Token 即将到期时，客户端刷新 Token 并重新连接事件流。

幂等规则：同一用户使用相同 `Idempotency-Key` 且请求参数摘要一致时，返回首次创建的任务；若摘要不同，返回 `409 Conflict`，不得创建新任务。该约束用于避免网络重试、用户重复点击等情形创建或执行重复任务。

## 8. 权限、可靠性与安全

- 用户只能创建、查询和取消自己的任务，不能读取其他用户的任务、结果或 OSS 签名 URL。
- 创建请求必须使用幂等键；同一用户与同一幂等键只允许对应一个任务。相同键但不同请求参数必须拒绝。
- 任务工作器以状态条件更新方式领取任务，防止消息重复投递时重复执行。
- 限制每个用户的并行任务数、创建频率和提示词长度；`imageCount` 仅校验当前模型支持的 1～6 范围，不额外施加业务数量上限；全局工作器并发数由服务端配置。
- 服务商 API Key、OSS 密钥、临时图片 URL 和原始服务商错误不得写入客户端响应或普通业务日志。
- Nginx / 网关必须关闭 SSE 响应缓冲并设置足够长的读取超时。
- SSE 定期发送心跳；Spring MVC 通过 `SseEmitter` 支持 SSE，断线恢复依赖任务状态对账而不是假设事件绝不丢失。参见[Spring SSE 文档](https://docs.spring.io/spring-framework/reference/web/webmvc/mvc-ann-async.html)。

## 9. 实施顺序与验收

实施顺序：

1. 确认会话、消息、任务、图片之间的数据关系，以及 MySQL 表、索引、Flyway 迁移与 Outbox 事件模型。
2. 配置私有 OSS，完成上传、签名 URL 与文件校验。
3. 配置 RabbitMQ 和任务/事件分发基础设施；首版单实例不部署 Redis Pub/Sub，多实例部署时再补充。
4. 实现 `DIRECT` 执行器与百炼 Qwen-Image 调用。
5. 实现会话内单活动任务约束、任务状态机、幂等、重试、取消和失败分类。
6. 实现会话历史恢复、上下文快照保存，以及 SSE 连接、心跳、事件分发与断线状态对账。
7. 补充单元测试、MySQL 集成测试、RabbitMQ/Redis/OSS 集成测试和真实百炼联调验收。

验收标准：

- 同一幂等键不会创建或执行重复任务。
- 用户可创建 1～6 张的普通文生图任务，并仅能读取自己的结果。
- 用户可创建多个会话；同一会话不会同时存在两个活动任务，切换或关闭会话页面不影响已创建任务的执行。
- 用户重新打开页面后可恢复会话历史、任务状态和已生成图片；SSE 事件可定位到对应会话与任务。
- 每次生成任务均保留其上下文快照与最终调用提示词，满足模型生成上下文的可追溯性。
- 所有请求图片均成功转存私有 OSS 后，任务进入 `SUCCEEDED`；至少一张成功转存但未全部成功时，任务进入 `PARTIALLY_SUCCEEDED`；没有任何图片成功转存时进入 `FAILED`。
- 任务状态变化可通过 SSE 通知在线客户端；断线重连后客户端能通过活动任务查询恢复正确状态。
- 可重试与不可重试失败得到不同处理；最终失败对用户返回安全的失败说明。
- 调用结果未知时不会自动重试，避免重复生成与重复计费。
- 同一幂等键且参数相同不会创建重复任务；相同幂等键但参数不同返回冲突。
- 任务、图片、Outbox 事件和任务状态变更具备可追溯性。
