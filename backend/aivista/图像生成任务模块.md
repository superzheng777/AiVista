# AiVista 图像生成任务模块

> 对应迭代：迭代 2——文生图与局部重绘任务  
> 当前状态：详细设计中  
> 本文档只记录已确认的设计决策与待确认项；未实现内容不得视为已交付。

## 1. 目标与范围

本模块负责创建、执行、追踪和通知 AI 图像生成任务，并将生成结果可靠存储到项目自有的阿里云 OSS。

当前优先实现的能力：

- 用户创建普通文生图任务。
- 支持用户创建、切换和恢复多个持久化生成会话。
- 同一会话同一时间仅允许一个活动生成任务；不同会话可独立排队或生成。
- 持久化会话内的对话历史，并为后续模型调用保留上下文记忆。
- 后端异步调用阿里云百炼 Qwen-Image。
- 一次任务可生成的图片数量由实际调用模型的能力决定。
- 将图片从服务商临时地址转存至私有 OSS。
- 用户通过 SSE 获得任务状态变化通知，并通过 REST 查询任务详情。

后续在同一任务模型中扩展：

- 局部重绘任务。
- 用户主动选择的 Agent / 工作流执行模式。

不属于当前实现范围：

- 局部重绘具体输入、遮罩和编辑 API。
- Agent / 工作流定义、编排、版本管理和界面。
- 社区发布、收藏、点赞、关注和完整生成历史浏览。
- 用户计费、订单和付费配额。

## 2. 已确认设计决策

| 主题 | 决策 |
| --- | --- |
| 图像服务 | 使用阿里云百炼 Qwen-Image API。 |
| 默认模型 | 当前调用百炼时固定使用 `qwen-image-2.0`；每个任务的 `model` 保存完整模型标识 `bailian/qwen-image-2.0`。 |
| 存储 | 服务商返回的临时图片 URL 由后端下载并转存至私有阿里云 OSS，不作为长期资产地址。 |
| 图片格式 | 首版仅接收和保存 PNG，不做格式转换。下载时响应 `Content-Type`、文件魔数和实际解析格式必须一致且均为 PNG；不符合时视为该图片转存失败。 |
| 百炼 HTTP 调用 | 前端创建任务后立即获得 `202 Accepted`；后台工作器再通过 HTTP 同步调用百炼。工作器连接百炼超过 5 秒仍未建立连接时，视为确认未发出请求，最多自动重试 5 次；已建立连接但等待结果超过 300 秒时，视为服务商结果未知，任务进入 `FAILED + PROVIDER_RESULT_UNKNOWN`，绝不自动重试。全局最多同时执行 30 个模型调用；真正调用百炼前还必须取得全局出口令牌，初始限制为每秒 2 次。 |
| 第三方数据处理同意 | 用户首次创建生成任务前，必须读取并明确同意“本次提示词及生成所需会话上下文将发送至阿里云百炼处理”的当前规则版本；后端返回当前版本、完整文案和用户同意状态，并记录用户实际确认的版本、文案摘要和同意时间。确认请求必须携带版本，且只接受当前有效版本；规则升级后必须重新确认。 |
| 首版执行范围 | 仅支持普通文生图；由后端异步任务系统直接调用模型服务，不在任务表保存任务类型或执行模式。局部重绘与工作流模式在对应迭代通过数据库迁移单独设计。 |
| 会话规则 | 会话列表展示用户所有已提交内容的持久化会话，不限制数量。进入生成页默认打开一个不持久化的空白创作框；首次提交时原子创建会话和任务，成功提交后才可再从侧栏开启下一个空白创作框。默认标题截取首条正向提示词，用户可后续修改。单个会话只允许一个未完成任务；单个用户跨所有会话和浏览器页面最多 4 个未完成任务。 |
| 网关防护 | 请求级防刷、DDoS 防护和 IP 限制由 Nginx / API 网关等部署层独立配置，不作为本业务模块的固定规则或错误码。 |
| 百炼出口限流 | 首版单实例使用应用内全局令牌桶，初始每秒最多调用百炼 2 次。任务可以先进入 RabbitMQ；工作器取得出口令牌后才调用百炼，不得阻塞 MQ 消费线程。仅官方定义的短时限流 `429 Throttling`、`Throttling.RateQuota`、`Throttling.BurstRate` 可自动重试，最多 5 次；每次按指数退避并加入随机抖动重新排队，不重复占用用户额度。 |
| 每日额度时区 | 每日 12 张图片额度按 `Asia/Shanghai`（北京时间）自然日统计，每日 00:00 重置；数据库生命周期时间仍统一以 UTC 存储。 |
| 历史与上下文 | 当前普通文生图的一轮历史仅保存用户提交的正向、负向提示词消息；生成结果通过任务及其图片记录关联，不创建 AI 结果占位消息。同一会话继承历史正向与负向提示词；模型调用前分别按既定长度规则拼接两类历史与本次输入。前端必须提示用户一个会话只围绕一个图片主题创作，更换主题时新建会话。 |
| 实时通知 | 使用 SSE 将任务状态变化主动推送给在线用户；REST 接口负责详情查询和断线后的状态对账。 |
| 执行队列 | 首版使用 Durable Quorum Queue `generation.task.execute`，声明时显式设置 `x-queue-type=quorum`。单节点部署时以 1 个成员运行；需要节点级高可用时扩展为 3 节点、3 成员。消费者并发初始为 30、每消费者 `prefetch=1`、使用手动 ACK。 |
| 排队超时 | 任务处于 `QUEUED` 超过 5 分钟时，由应用主动收敛为 `FAILED + QUEUE_TIMEOUT` 并通知客户端；不使用 RabbitMQ 消息 TTL，避免消息过期后任务真相无法同步。 |
| 数据保留与备份 | 用户生成图片、会话、提示词和任务记录首版不自动过期。服务商临时 URL 快照全部转存后立即清除，最长保留 24 小时；已发布 Outbox 事件保留 7 天，失败 Outbox 事件保留 14 天。生产数据库必须具备加密备份与可验证恢复能力，备份保留 7 天；使用 RDS 时采用其自动备份，自建 MySQL 时采用定时全量备份加二进制日志或等效方案，并在正式上线前完成至少一次恢复演练。 |
| 首版部署 | 首版按单个 Spring Boot 应用实例设计：同一进程处理 HTTP/SSE 请求并消费 RabbitMQ 任务，应用内直接向 SSE 连接推送事件，并使用应用内百炼出口限流器，不启用 Redis。迁移到 Redis 等共享限流和跨实例事件分发能力之前，不得水平扩容或拆分独立工作器。请求级防护由部署层网关承担。 |
| 任务真相来源 | MySQL 任务与结果记录是权威状态；SSE 仅为实时通知通道。 |
| 服务商密钥 | 本地开发可在未提交的 `application-local.yaml` 填写开发密钥；环境变量始终可覆盖配置。首版生产环境统一通过环境变量注入百炼 API Key 与 OSS 凭证，绝不下发客户端、写入代码仓库或记录到日志。 |

首版固定使用 Qwen-Image 2.0，向用户提供 1～6 张的数量选择和五种预定义尺寸；服务端也必须按当前模型能力校验请求范围和尺寸映射，防止绕过前端直接调用 API 传入非法参数。不提供模型选择，也不自动降级或切换模型。项目仍由后台工作器包装为异步任务执行。编码前必须在目标地域和业务空间完成真实百炼联调，确认模型可用性、单次最大图片数、支持尺寸、`negative_prompt`、`prompt_extend=false`、每秒 2 次出口限制、计费口径和错误码分类；若结果与本文档不一致，应先更新配置与文档再实施。详见[阿里云 Qwen-Image API 文档](https://help.aliyun.com/zh/model-studio/qwen-image-api)。

模型能力（正、负提示词长度，单次图片数量和尺寸）由应用层按完整模型标识配置并校验，数据库提示词字段不绑定当前模型上限。首版配置 `bailian/qwen-image-2.0`；后续接入模型时扩展能力配置和调用适配器，不需要为提示词字段修改数据库结构。

### 2.1 千问 Qwen-Image 调用参考

> 官方文档：[千问 Qwen-Image 文生图 API](https://help.aliyun.com/zh/model-studio/qwen-image-api)（访问与实现前应再次核对模型可用地域、版本、限流和计费规则）。

当前首版执行器调用百炼的同步多模态生成接口；项目层面的异步由 RabbitMQ 工作线程负责，不应将本项目任务状态机与百炼异步接口混用。

| 项目 | 约定 |
| --- | --- |
| 默认模型 | 百炼请求模型为 `qwen-image-2.0`；任务记录保存 `bailian/qwen-image-2.0`。首版仍通过后台工作器调用服务商接口。 |
| 北京地域端点 | `POST https://{WorkspaceId}.cn-beijing.maas.aliyuncs.com/api/v1/services/aigc/multimodal-generation/generation` |
| 鉴权 | 请求头 `Authorization: Bearer ${DASHSCOPE_API_KEY}`；API Key 与地域对应，严禁写入客户端、代码仓库或日志。 |
| 请求头 | `Content-Type: application/json`。 |
| 正向提示词 | `input.messages` 必须且只能包含一条 `user` 消息；其 `content` 必须且只能包含一个 `text`。`qwen-image-2.0` 系列上限为 1300 Token，服务端应在提交前校验项目自身长度限制。 |
| 反向提示词 | `parameters.negative_prompt` 可选，最长 500 字符；项目接口字段 `negativePrompt` 映射至该参数。 |
| 尺寸 | `parameters.size` 格式为 `宽*高`。`qwen-image-2.0` 系列总像素范围为 512×512 至 2048×2048；本项目仅允许预定义的宽高比映射值。 |
| 生成数量 | `parameters.n` 为 1～6；前端提供该范围选择，服务端执行同范围校验。 |
| 提示词改写 | `parameters.prompt_extend` 官方默认 `true`；当前普通模式固定传 `false`，以保证保存的原始提示词与实际输入一致。 |
| 返回图片 | 服务商返回临时公开 URL；工作线程必须下载、校验并转存至私有 OSS 后，才可将任务置为 `SUCCEEDED`。 |

同步请求体结构如下。实现时只替换模型、提示词和已校验的业务参数：

```json
{
  "model": "qwen-image-2.0",
  "input": {
    "messages": [
      {
        "role": "user",
        "content": [{ "text": "正向提示词" }]
      }
    ]
  },
  "parameters": {
    "negative_prompt": "反向提示词",
    "size": "2048*2048",
    "n": 1,
    "prompt_extend": false,
    "watermark": false
  }
}
```

注意：首版项目异步性由 RabbitMQ 工作器提供，当前不直接使用服务商异步任务协议。若未来切换为服务商异步接口，需重新设计服务商任务 ID 的轮询、超时与重试策略，不复用当前执行器的调用逻辑。

### 2.2 百炼错误分类与自动重试

错误码分类以[阿里云百炼官方错误码](https://www.alibabacloud.com/help/zh/model-studio/error-code)为准；实现时必须同时读取 HTTP 状态、百炼 `code` 和安全化后的 `message`，不得仅按 HTTP 状态码决定重试。该页面与真实百炼联调结果是后续补充或调整映射的参考来源。

| 识别结果 | 任务处理 | 自动重试 | 稳定失败分类 |
| --- | --- | --- | --- |
| TCP/TLS 连接未建立，能够确认 HTTP 请求尚未发出 | 清空调用开始标记后重新排队 | 最多 5 次，指数退避加抖动 | 超限后 `PROVIDER_CONNECTION_FAILED` |
| `429 Throttling`、`Throttling.RateQuota`、`Throttling.BurstRate` | 重新排队 | 最多 5 次，指数退避加抖动 | 超限后 `PROVIDER_RATE_LIMITED` |
| `429 Throttling.AllocationQuota`，以及 `CommodityNotPurchased`、预付/后付账单逾期、免费额度耗尽 | 直接失败 | 不重试 | `PROVIDER_QUOTA_UNAVAILABLE` |
| `400 DataInspectionFailed` | 直接失败 | 不重试 | `PROVIDER_CONTENT_REJECTED` |
| 其他 `400` 参数、输入格式、尺寸等错误；`401`、`403`、`404` 的鉴权、业务空间、模型、权限或端点错误 | 直接失败，并记录可排障的安全摘要 | 不重试 | `PROVIDER_CONFIGURATION_ERROR` |
| 已收到百炼 `500`、`RequestTimeOut` 等服务端错误；本地读取超时、连接中断、网关 `502/504` 或其他可能已发出请求的异常 | 直接失败 | 不重试 | `PROVIDER_RESULT_UNKNOWN` |

官方文档对部分 `500` 建议稍后重试；首版仍不自动重试，因为同步调用没有项目可验证的服务商幂等键或结果查询依据，无法证明模型未开始处理。该取舍优先避免重复生成与重复计费；用户可在任务失败后主动创建一项新任务。

## 3. 核心架构

```text
创建任务 API
    ↓
MySQL：任务记录 + Outbox 事件（同一事务）
    ↓
RabbitMQ：生成任务队列
    ↓
任务工作器：调用百炼 Qwen-Image
    ↓
模型返回临时 URL → MySQL：保存转存快照
    ↓
下载临时结果 → 上传私有 OSS
    ↓
MySQL：图片记录 + 任务终态 + SSE Outbox 事件（同一事务）
    ↓
RabbitMQ：工作器 ACK 执行消息
    ↓
应用内 SSE 分发：向所属用户推送任务状态变化
    ↓
多实例部署时：Redis Pub/Sub 跨实例分发
```

组件职责：

| 组件 | 职责 |
| --- | --- |
| MySQL | 保存任务、图片、Outbox 事件；是唯一权威状态来源。 |
| RabbitMQ | 使用 Durable Quorum Queue 承载生成任务、确认与有限重试。执行队列显式声明 `x-queue-type=quorum`，消费者并发初始为 30、每消费者 `prefetch=1`、使用手动 ACK。 |
| 阿里云百炼 | 生成图像。 |
| 阿里云 OSS | 保存项目拥有的生成结果文件。 |
| Redis Pub/Sub | 向持有 SSE 连接的应用节点分发实时通知；首版单实例不启用，多实例部署时启用；不作为持久化消息来源。 |
| SSE | 向浏览器单向推送任务变化。 |

采用 Transactional Outbox：任务状态与待分发事件必须在同一数据库事务内写入。创建任务时记录 `TASK_EXECUTE`；任务状态变化时记录 `TASK_STATUS_CHANGED`。事件表不保存通用 JSON 载荷；分发器根据事件类型路由，发布的消息仅携带 `eventId`、`taskId` 与 `taskVersion`。工作器或 SSE 分发器按 `taskId` 从 MySQL 读取所需当前状态，消息不得包含提示词、临时 URL、签名 URL或密钥。任务投递事件采用有限自动重试；超过上限则将对应任务收敛为 `FAILED`，提示用户重新发起，不引入人工重放或运维处理链路。SSE 事件发送失败不改变任务真相，客户端通过 REST 状态对账恢复。

工作器使用手动 ACK：模型返回的临时 URL 快照、图片记录、任务终态和 SSE Outbox 事件均已提交后，才 ACK RabbitMQ 执行消息。工作器在此之前崩溃时，RabbitMQ 重投同一任务；新工作器读取任务状态，已有快照则只恢复 OSS 转存，终态任务则直接 ACK，不重复调用模型。

## 4. 任务模型

### 4.1 首版任务范围

首版仅创建普通文生图任务。局部重绘与工作流模式尚未落库；开始对应迭代时，再根据实际输入、执行方式与结果关系通过 Flyway 迁移新增字段或扩展表。

### 4.2 会话与任务规则

- 会话是用户可持久化恢复的生成对话单元；用户离开页面、关闭浏览器或切换会话不得中断已创建的后台任务。
- 每个生成任务必须归属于一个会话。会话列表展示全部已提交内容的会话，不限制数量。页面初始空白创作框不落库；其首次提交在同一事务中创建会话、任务、消息与任务 Outbox 事件。仅当该空白创作框成功提交后，前端才允许侧栏“新建会话”开启下一个空白创作框，避免无内容会话。
- 首次提交时由后端截取首条正向提示词生成默认会话标题；用户可后续修改标题。会话列表仅展示已提交内容的会话，按最近消息时间倒序。
- 同一会话在 `QUEUED`、`RUNNING` 或 `CANCEL_REQUESTED` 状态下，不得创建第二个未完成任务；单个用户跨所有会话、浏览器页面和设备最多 4 个未完成任务。前端默认仅保留一个空白新对话，且该对话成功提交后才允许新建下一个空白对话；后端仍必须以数据库约束为准，不能仅依赖前端按钮状态。创建任务时，在同一事务内锁定当前 `users` 行、检查未完成任务数并占用当日图片额度，再创建任务与执行 Outbox 事件。达到用户总量上限时返回 `409 Conflict` 与错误码 `USER_GENERATION_CONCURRENCY_LIMIT`。
- 会话历史是对话记忆的权威来源。当前普通文生图的一轮历史仅由一条用户提示词消息构成，其中保存该轮正向、负向提示词；任务通过唯一的 `source_message_id` 关联这条消息，任务再关联状态和结果图片。消息查询以“用户消息 + 生成结果”作为一个展示单元组装返回，不新增 AI 消息表。每次调用模型时，应用保存最终调用的正向、负向提示词，以便结果追溯。
- 当前文生图模型调用为单轮调用，不会天然记住上一轮内容，也不产生 AI 文本内容。每次生成分别拼接会话中的历史正向提示词与本次正向提示词、历史非空负向提示词与本次负向提示词；两类内容超过各自模型限制时，分别从最早的历史内容开始排除，绝不删除历史消息或截断本次输入。当前正向或负向提示词自身超长时，直接返回参数校验错误，不创建任务、不调用模型。
- Agent / 工作流模式未来如需产生文本消息，应在对应迭代新增消息类型或独立消息模型，不影响当前普通文生图的简化记录方式。

### 4.3 状态机

```text
QUEUED
  ├── CANCELLED
  └── RUNNING
        ├── SUCCEEDED
        ├── PARTIALLY_SUCCEEDED
        ├── FAILED
        └── CANCEL_REQUESTED → CANCELLED
```

规则：

- 状态只允许按既定方向迁移，终态不可再次执行。工作器领取与所有状态变更均使用带前置状态和 `task_version` 的数据库条件更新；受影响行数为 `1` 才代表成功，受影响行数为 `0` 的工作器不得执行模型调用。不得将悲观锁或数据库事务跨越百炼调用、图片下载或 OSS 上传。
- 请求数量的全部图片均成功转存 OSS 时，任务进入 `SUCCEEDED`；至少一张成功转存、但未全部转存时进入 `PARTIALLY_SUCCEEDED`；没有任何图片成功转存时进入 `FAILED`。
- 用户取消时：仍在队列中、工作器尚未领取或 `provider_call_started_at` 为空的任务直接进入 `CANCELLED`；百炼调用已开始时进入 `CANCEL_REQUESTED`，且仍占用实际调用资源。待调用结束、超时或失败后统一进入 `CANCELLED`；此后返回的任何图片均不得下载、转存或记录。取消与工作器领取通过带前置状态、`task_version` 和 `provider_call_started_at` 的条件更新竞争。重复取消 `CANCEL_REQUESTED` 或 `CANCELLED` 属于幂等成功；对 `SUCCEEDED`、`PARTIALLY_SUCCEEDED` 或 `FAILED` 再取消返回 `409 TASK_ALREADY_FINISHED`。取消不返还每日额度。
- 仅“确认请求尚未发送到服务商”的连接失败，以及官方明确的短时限流 `429 Throttling`、`Throttling.RateQuota`、`Throttling.BurstRate` 可自动重试，均最多 5 次并按指数退避加抖动重新排队。请求可能已发出但发生超时、连接中断、网关错误或百炼 `500` 等结果未知情况时，不得自动重试，以避免重复生成与重复计费；任务进入 `FAILED`，内部失败码记为 `PROVIDER_RESULT_UNKNOWN`，对用户仅返回安全的失败说明。
- `DataInspectionFailed` 内容安全、参数错误、计费/配额、鉴权、模型、业务空间和端点错误均为不可重试失败，直接进入 `FAILED`；完整映射见“2.2 百炼错误分类与自动重试”。
- 每次状态变化递增 `task_version`，客户端仅接受比本地版本新的事件。

### 4.4 核心数据表

以下资源主键统一使用 `BIGINT UNSIGNED` 自增；对外 HTTP 响应统一序列化为字符串。除 `user_generation_daily_usage.usage_date` 外，所有生命周期时间使用 UTC `DATETIME(3)`，`usage_date` 使用服务端按 `Asia/Shanghai` 计算的 `DATE`。

技术生命周期时间由数据库维护：`created_at` 使用 `DEFAULT CURRENT_TIMESTAMP(3)`，`updated_at` 使用 `DEFAULT CURRENT_TIMESTAMP(3) ON UPDATE CURRENT_TIMESTAMP(3)`。业务事件时间由应用在对应动作中显式写入，包括 `last_message_at`、`started_at`、`completed_at`、`provider_call_started_at`、`consented_at`、`available_at`、`locked_at` 与 `published_at`；它们不得因无关字段更新而自动变化。

#### `generation_sessions`：生成会话

| 字段 | 规则与职责 |
| --- | --- |
| `id` | `BIGINT UNSIGNED` 自增会话 ID。 |
| `user_id` | `BIGINT UNSIGNED NOT NULL` 所属用户 ID，外键关联 `users.id`。 |
| `title` | `VARCHAR(100) NOT NULL` 会话名称；首次提交时由后端截取首条正向提示词生成默认标题，用户可修改。 |
| `last_message_at` | UTC `DATETIME(3) NOT NULL` 最近一次用户成功提交提示词消息的时间，用于会话列表排序；新建任务写入该轮消息时由应用更新。 |
| `created_at` / `updated_at` | UTC `DATETIME(3) NOT NULL` 会话创建与更新时间；`updated_at` 在标题或 `last_message_at` 变化时由数据库自动更新。 |

索引与约束：`idx_generation_sessions_user_last_message(user_id, last_message_at)`。创建已有会话的任务时，事务必须先锁定该会话行，再查询是否存在 `QUEUED`、`RUNNING` 或 `CANCEL_REQUESTED` 任务；存在任意一个即拒绝创建，以保证同一会话只有一个未完成任务。

#### `generation_messages`：会话历史消息

| 字段 | 规则与职责 |
| --- | --- |
| `id` | `BIGINT UNSIGNED` 自增消息 ID。 |
| `session_id` | `BIGINT UNSIGNED NOT NULL` 所属生成会话 ID，外键关联 `generation_sessions.id`。 |
| `sequence_no` | 会话内递增序号，`INT UNSIGNED`，从 `1` 开始；创建任务时在已锁定的会话行内分配，用于稳定恢复消息顺序。 |
| `prompt` | 本轮用户正向提示词，`TEXT`。 |
| `negative_prompt` | 本轮用户负向提示词，`TEXT`，可空。 |
| `created_at` | UTC `DATETIME(3)` 消息创建时间；历史消息创建后不修改，不设置 `updated_at`。 |

索引与约束：唯一索引 `uk_generation_messages_session_sequence(session_id, sequence_no)`；该索引同时满足按会话顺序读取消息，不再额外建立时间索引。

创建任务时，同一事务中写入一条提示词消息和关联任务；任务以唯一的 `source_message_id` 指向该消息，保证一条消息至多对应一个任务。任务完成后，通过 `task_id → generation_images` 取得 1～6 张图片。消息表不保存 OSS 签名 URL；图片记录只保存私有对象定位信息，客户端再按权限获取短期 URL。

#### `generation_tasks`：异步生成任务

| 字段 | 规则与职责 |
| --- | --- |
| `id` | `BIGINT UNSIGNED` 自增任务 ID。 |
| `user_id` / `session_id` / `source_message_id` | `BIGINT UNSIGNED NOT NULL`；所属用户、会话与触发本次生成的提示词消息。 |
| `model` | `VARCHAR(128) NOT NULL`，带服务商命名空间的完整模型标识；首版固定为 `bailian/qwen-image-2.0`。调用适配器据此前缀选择服务商，并解析出百炼请求模型名 `qwen-image-2.0`。 |
| `status` / `task_version` | `VARCHAR(32) NOT NULL DEFAULT 'QUEUED'` 的当前任务状态（由 Java 枚举及状态迁移逻辑校验，不使用 MySQL `ENUM` 或 `CHECK`）与 `INT UNSIGNED NOT NULL DEFAULT 0` 的乐观锁、SSE 事件去重版本。 |
| `attempt_count` | `TINYINT UNSIGNED NOT NULL DEFAULT 0`；已发生的可重试调用次数，百炼明确限流时最多自动重试 5 次，不直接暴露给客户端。 |
| `provider_call_started_at` | UTC `DATETIME(3)`，可空；百炼调用开始前写入，非空表示请求可能已发出。消息重投时如未保存结果快照，则直接收敛为 `FAILED + PROVIDER_RESULT_UNKNOWN`，不得再次调用。 |
| `final_prompt` / `final_negative_prompt` | `TEXT`；最终传给图像模型的正向提示词非空，负向提示词可空。任务创建时按会话历史计算并保存，用于生成追溯。 |
| `width` / `height` | `INT UNSIGNED NOT NULL`，实际请求的输出尺寸。 |
| `requested_image_count` / `completed_image_count` | `TINYINT UNSIGNED NOT NULL`，前者为请求数量，后者默认 `0` 并记录实际成功转存数量；接口根据任务状态计算失败或取消数量，不预创建失败图片槽位。 |
| `provider_request_id` | `VARCHAR(128)`，可空；服务商请求追踪标识。 |
| `provider_result_snapshot` | `JSON`，可空；服务商成功响应的短期转存快照：整次 `provider_request_id`、临时 URL 列表及各 URL 的内部 `source_index`。仅用于 OSS 转存恢复；仅允许后端读写、不返回客户端、不写入普通日志，全部转存后立即清除，最长不超过服务商 URL 的 24 小时有效期。生产 RDS 必须开启静态加密，并使用最小权限数据库账号与网络隔离。 |
| `idempotency_key` / `request_fingerprint` | `CHAR(36) NOT NULL` 的 UUID v4 幂等键与 `CHAR(64) NOT NULL` 的 SHA-256 十六进制请求指纹；指纹覆盖规范化后的 `sessionId`（包括未传状态）、`prompt`、`negativePrompt`、`aspectRatio`、`imageCount`。两者随任务长期保存。 |
| `failure_code` | `VARCHAR(64)`，可空；稳定失败分类。接口按该错误码生成安全的用户提示，不持久化 `failure_message`。 |
| `created_at` / `updated_at` / `started_at` / `completed_at` | UTC `DATETIME(3)`；前两者非空，后两者可空。`started_at` 在首次进入 `RUNNING` 时写入，429 重排队不清空；`completed_at` 仅在终态写入。 |

索引与约束：唯一索引 `uk_generation_tasks_user_idempotency(user_id, idempotency_key)` 与 `uk_generation_tasks_source_message(source_message_id)`；索引 `idx_generation_tasks_session_created(session_id, created_at)` 和 `idx_generation_tasks_user_status(user_id, status)`。前者保证幂等创建，后者保证一条用户提示词消息至多关联一个生成任务。创建任务时锁定 `users` 行并按状态统计未完成任务数，保证单用户最多四个未完成任务。

#### `generation_images`：生成结果图片

| 字段 | 规则与职责 |
| --- | --- |
| `id` | `BIGINT UNSIGNED` 自增图片 ID。 |
| `task_id` / `user_id` | `BIGINT UNSIGNED NOT NULL`；所属任务与用户。会话归属通过任务关联获得，不重复存储 `session_id`。 |
| `object_key` | `VARCHAR(512) NOT NULL` 私有 OSS 对象键；首版 Bucket 由服务端配置统一指定，永不保存短期签名 URL。 |
| `content_type` / `file_size` | `VARCHAR(64) NOT NULL` 与 `BIGINT UNSIGNED NOT NULL` 文件元数据；首版 `content_type` 固定为 `image/png`。 |
| `width` / `height` | `INT UNSIGNED NOT NULL`，实际图像尺寸。 |
| `source_index` | `TINYINT UNSIGNED NOT NULL`；服务商结果的内部来源标识，仅用于崩溃恢复时避免重复转存；不作为用户可见的失败槽位或顺序。 |
| `created_at` | UTC `DATETIME(3) NOT NULL` 保存时间。 |

索引与约束：唯一索引 `uk_generation_images_task_source(task_id, source_index)`；索引 `idx_generation_images_user_created(user_id, created_at)`，为迭代 3 的个人资产查询提供基础。对象键由服务端生成，不额外建立其唯一索引。仅在图片成功校验并转存 OSS 后创建记录；单张失败仅汇总至任务字段。

#### `outbox_events`：可靠事件分发

| 字段 | 规则与职责 |
| --- | --- |
| `id` | `BIGINT UNSIGNED` 自增事件 ID。 |
| `event_type` | `VARCHAR(32) NOT NULL`；应用层仅允许 `TASK_EXECUTE` 或 `TASK_STATUS_CHANGED`，同时决定分发目的地。 |
| `task_id` / `task_version` | `BIGINT UNSIGNED NOT NULL` 与 `INT UNSIGNED NOT NULL`；关联任务与其状态版本，分发时作为最小消息标识。 |
| `status` / `retry_count` | `VARCHAR(16) NOT NULL DEFAULT 'PENDING'` 的分发状态（应用层仅允许 `PENDING`、`PROCESSING`、`PUBLISHED`、`FAILED`）与 `TINYINT UNSIGNED NOT NULL DEFAULT 0` 的重试次数。 |
| `available_at` / `locked_at` / `published_at` | UTC `DATETIME(3)`；`available_at` 非空，初始等于创建时间、重试时更新为下次投递时间；其余两项可空，分别记录分发器领取和成功分发时间。 |
| `last_error` | `VARCHAR(512)`，可空；最近一次分发失败的内部错误摘要，不包含密钥、临时 URL 或完整提示词。 |
| `created_at` | UTC `DATETIME(3) NOT NULL` 事件创建时间；不设置 `updated_at`。 |

保留规则：已发布事件保留 7 天，失败事件保留 14 天后由定时清理任务删除；任务和图片记录仍是业务追溯来源，不依赖 Outbox 长期保存。

索引与约束：索引 `idx_outbox_events_status_available(status, available_at)`，用于分发器扫描待投递事件。`locked_at` 用于回收工作线程异常退出后遗留的 `PROCESSING` 事件。

创建任务时在短事务内对当前用户执行 `SELECT id FROM users WHERE id = ? FOR UPDATE`，以串行化同一用户的任务创建。随后统计该用户处于 `QUEUED`、`RUNNING` 或 `CANCEL_REQUESTED` 的任务数；少于 4 才允许创建新任务，否则返回 `409 USER_GENERATION_CONCURRENCY_LIMIT`。工作器只需以带前置状态的条件更新将任务从 `QUEUED` 领取为 `RUNNING`；用户未完成任务总数已限制为 4，因此不再维护额外的排队或执行槽位。

#### `user_generation_daily_usage`：用户每日图片用量

每个用户每天一行，用于原子控制每日请求图片额度。

| 字段 | 规则与职责 |
| --- | --- |
| `user_id` / `usage_date` | `BIGINT UNSIGNED NOT NULL` 用户 ID 与 `DATE NOT NULL` 的按 `Asia/Shanghai` 计算业务自然日，组成联合主键。 |
| `requested_image_count` | `TINYINT UNSIGNED NOT NULL DEFAULT 0`，当日已创建任务的请求图片数量总和；每日额度固定为 12。 |
| `updated_at` | UTC `DATETIME(3) NOT NULL`，最近一次额度占用时间。 |

创建任务时在同一事务中按 `imageCount` 原子占用额度，条件为占用后不超过 12。`usage_date` 由服务端按 `Asia/Shanghai` 计算，不直接使用数据库 UTC 日期。任务创建成功即占用，后续成功、失败、部分成功或取消均不返还；同一幂等键重放不重复占用。

#### `user_consents`：用户规则同意记录

用于记录需要服务端强制校验的用户同意状态；首版使用 `GENERATION_THIRD_PARTY_PROCESSING` 类型。

| 字段 | 规则与职责 |
| --- | --- |
| `user_id` | `BIGINT UNSIGNED NOT NULL`，与 `consent_type` 组成联合主键。 |
| `consent_type` | `VARCHAR(64) NOT NULL`；首版固定为 `GENERATION_THIRD_PARTY_PROCESSING`。 |
| `policy_version` | `VARCHAR(64) NOT NULL`，用户已同意的规则版本。 |
| `policy_content_hash` | `CHAR(64) NOT NULL`，用户确认时所展示不可变规则文案的 SHA-256 十六进制摘要，用于追溯实际同意内容。 |
| `consented_at` | UTC `DATETIME(3) NOT NULL`，最后一次用户确认时间。 |

索引与约束：联合主键 `(user_id, consent_type)`；每个用户、每种同意类型仅保留当前有效确认状态。重新确认时更新规则版本、文案摘要与 `consented_at`。

服务端必须维护当前有效规则版本及其不可变完整文案。前端先查询当前版本、完整文案和用户同意状态，确认时回传实际展示的 `policyVersion`；后端只接受当前有效版本，并更新该用户、该同意类型的当前记录。创建生成任务前，服务端必须校验该记录的版本等于当前版本；未同意或版本落后时，拒绝创建任务且不写入会话、任务、消息、用量或 Outbox 数据。规则文案应明确提示：为生成图片，用户本次正向提示词、负向提示词及生成所需会话上下文将发送至阿里云百炼；生成内容会被项目保存，并可在后续资产功能中删除；用户不应输入密码、身份证号、联系方式等敏感个人信息。

#### 外键与删除规则

`generation_sessions.user_id`、`generation_messages.session_id`、`generation_tasks.user_id / session_id / source_message_id`、`generation_images.task_id / user_id`、`outbox_events.task_id`、`user_generation_daily_usage.user_id` 与 `user_consents.user_id` 均使用 MySQL 外键保障关联完整性，统一采用 `ON DELETE RESTRICT`。迭代 2 不提供会话、任务或图片的硬删除与数据库级联删除；后续个人资产删除必须单独设计数据库记录、私有 OSS 对象及公开副本的清理流程。

## 5. 普通文生图参数

当前创建任务接口只接受以下业务参数：

| 参数 | 规则 | 服务商映射 |
| --- | --- | --- |
| `prompt` | 必填；按当前模型能力校验。本期 `bailian/qwen-image-2.0` 配置为最多 1000 个字符。 | 正向提示词。 |
| `negativePrompt` | 可选；按当前模型能力校验。本期 `bailian/qwen-image-2.0` 配置为最多 500 个字符。 | `negative_prompt`。 |
| `aspectRatio` | 必填，默认 `1:1`。 | 映射为固定 `size`。 |
| `imageCount` | 必填；前端仅提供 `1`～`6` 的选择，服务端校验后映射至 `n`。 | `n`。 |

尺寸映射：

| `aspectRatio` | `size` |
| --- | --- |
| `1:1` | `2048*2048` |
| `16:9` | `2688*1536` |
| `9:16` | `1536*2688` |
| `4:3` | `2368*1728` |
| `3:4` | `1728*2368` |

普通模式固定 `prompt_extend = false`，以使已保存的原始提示词与实际普通模式输入一致。工作流模式未来如改写提示词，必须额外保存原始提示词、最终提示词与工作流版本。

每次调用分别以当前正向、负向提示词为核心，并在各自模型长度预算内按由近到远的优先级保留对应历史；最终发送顺序仍按时间从旧到新排列。后端使用与当前模型匹配的长度计算规则校验 `final_prompt` 与 `final_negative_prompt`。若仅本次输入已超限，返回“提示词过长”；若加上历史超限，则仅舍弃本次拼接中的最旧历史。

当前不向用户开放模型选择、随机种子、水印开关、任意宽高、自由工作流配置等参数。

## 6. OSS 存储与访问

首版仅区分本地开发与未来生产两个环境：分别配置独立的私有 OSS Bucket。本地应用只可访问开发 Bucket，生产应用只可访问生产 Bucket；两者不得混用或共享对象。每个 Bucket 与同环境后端部署在同一地域，所有访问使用 HTTPS；生产环境如受 OSS 中国内地接入要求影响，使用配置的自定义 HTTPS 访问域名及有效证书。Bucket 名称、Endpoint 与访问凭证均为环境配置，不写入业务数据库。对象默认继承 Bucket 的 `private` ACL，不为单个图片设置公开 ACL。

应用使用独立的 RAM 身份访问当前环境的 Bucket，不使用阿里云主账号凭证。首版仅授予 `users/*` 前缀的 `GetObject`、`PutObject` 和 `DeleteObject` 权限，分别满足读取/签发下载链接、上传转存图片，以及清理本次转存失败、取消或数据库提交失败时产生的孤儿对象；不授予全 Bucket 管理、列举或修改 ACL 的权限。`DeleteObject` 不是提前实现用户资产删除，后续用户主动删除仍需单独设计其业务流程。本地开发凭证保存在未提交的 `application-local.yaml`，生产凭证通过环境变量注入。

两个 Bucket 均启用 OSS 托管密钥的服务端加密（SSE-OSS），由 OSS 在写入时自动加密、读取时自动解密；首版不使用 KMS 自管密钥或客户端加密。当前用户生成图片长期保留，因此不配置图片自动删除、自动归档或降冷存储的生命周期规则；后续资产删除按用户操作单独设计对象删除流程。

生成图片必须在服务端完成转存：

1. 工作器从百炼返回临时 URL 后，先将 `provider_request_id`、临时 URL 列表和内部 `source_index` 写入任务快照并提交；服务商 URL 有效期为 24 小时。
2. 工作器再从临时 URL 下载图像。前端、SSE、普通日志和数据库其他字段均不得获得该 URL。
3. 仅请求 HTTPS 且命中真实百炼联调确认后的结果域名白名单的地址；禁止重定向，拒绝本地、内网和云元数据地址，防止 SSRF。
4. 流式下载，连接超时固定 5 秒、读取超时固定 60 秒，单图片最大 30 MiB；校验 HTTP 成功、响应 `Content-Type`、文件魔数、真实图片格式、宽高与总像素数。首版仅接受 PNG，响应类型、魔数与实际格式必须一致，实际宽高不得超过本次任务请求尺寸及当前模型像素上限；不符合时视为该图片转存失败，避免将错误响应、超大文件或伪造内容误存为图片。
5. 上传至当前环境配置的私有 OSS Bucket；对象键完全由服务端生成，不接受用户输入的路径片段。首版对象统一使用 `.png` 后缀。
6. 成功后写入 `generation_images`；全部图片转存成功时置为 `SUCCEEDED`，至少一张成功但未全部成功时置为 `PARTIALLY_SUCCEEDED`。只有该终态事务提交后，才通知前端展示图片。

若图片已上传但随后发现任务被取消、转存失败或数据库提交失败，工作器立即对该对象执行一次尽力删除。删除失败只记录安全日志与监控指标，不新建持久化清理表或额外状态机；数据库任务与图片记录始终是业务真相。此类罕见孤儿对象的持续清理与用户资产删除一并在后续迭代设计。

工作器重投或崩溃恢复时，按 `(task_id, source_index)` 查询 `generation_images`：已有记录则跳过上传；没有记录则向固定对象键重新上传并写入记录，因此正常重试不会产生重复对象。首版不在任务详情或消息查询时逐张调用 OSS `HEAD`，也不扫描整个 Bucket；若出现“数据库记录存在而 OSS 对象被外部删除”等基础设施异常，以监控和安全日志排查，不新增业务状态机。

对象键规范：

```text
users/{userId}/generations/{taskId}/{sourceIndex}.png
```

该路径只用于归档与转存幂等：同一任务同一来源图片重试时写回同一对象键，避免创建重复对象；按用户前缀也为后续个人资产清理提供定位基础。Bucket 保持私有，对象键不对客户端返回，也不作为访问控制依据。

迭代 2 只保存私有 OSS 原图，客户端不直接获得永久 OSS 地址，任务详情接口按权限返回固定 10 分钟有效的签名 URL。签名 URL 不得写入日志、数据库或 SSE 事件。返回签名 URL 的 API 响应使用 `Cache-Control: private, no-store`；OSS 图片使用不超过签名 URL 有效期的 `Cache-Control: private, max-age=...`。前端仅在 SPA 内存中缓存 URL 及过期时间，过期后重新查询任务详情，不写入本地存储，并设置 `Referrer-Policy: no-referrer`，错误监控和埋点不得上报完整 URL。迭代 4 用户发布图片时，基于私有原图生成公开展示副本或缩略图，保存到公开 Bucket 或公开前缀，并通过固定 CDN/OSS URL 供灵感社区浏览；撤销发布时移除或禁止该公开副本访问，不影响私有原图。

## 7. API 与认证契约

所有接口使用 `/api` 前缀、统一响应结构和 Bearer Access Token 认证。

| 功能 | 方法与路径 | 认证 | 说明 |
| --- | --- | --- | --- |
| 获取会话列表 | `GET /api/generation-sessions?cursor=&limit=20` | Bearer Token | 游标分页返回当前用户全部已提交内容的会话，包含存在活动任务和仅含终态任务的会话；按 `last_message_at DESC, id DESC` 排序。默认 `limit=20`，最大 50；超出 1～50 返回 `422` 参数校验失败。响应返回 `items`、不透明 `nextCursor` 和 `hasMore`，不提供页码或总页数。 |
| 获取会话消息 | `GET /api/generation-sessions/{sessionId}/messages?before=&limit=30` | Bearer Token | 游标分页获取会话消息与关联任务结果。数据库与游标按 `sequence_no` 定位，响应中的消息按时间正序排列，便于前端向上加载更早历史。默认 `limit=30`，最大 100；超出 1～100 返回 `422` 参数校验失败。 |
| 修改会话标题 | `PATCH /api/generation-sessions/{sessionId}` | Bearer Token | 仅会话所有者可修改标题。 |
| 查询第三方数据处理规则 | `GET /api/users/me/consents/generation` | Bearer Token | 返回当前规则版本、完整展示文案、当前用户是否已同意该版本及同意时间。 |
| 确认第三方数据处理规则 | `POST /api/users/me/consents/generation` | Bearer Token | 请求必须携带前端实际展示的 `policyVersion`；后端只接受当前有效版本。首版仅支持确认，不提供撤回后继续生成的能力。 |
| 创建普通文生图任务 | `POST /api/generation-tasks` | Bearer Token | 要求 `Idempotency-Key` 请求头。`sessionId` 可选：传入时使用已有会话；不传时原子创建会话和任务，并在 `202 Accepted` 响应中返回 `sessionId` 与 `taskId`。 |

### 7.1 创建普通文生图任务

`POST /api/generation-tasks` 仅接收用户可选择的生成参数；模型标识、模型能力上限、最终拼接提示词及百炼调用参数均由服务端决定，不得由客户端传入。

请求必须携带 `Authorization: Bearer <accessToken>` 与 `Idempotency-Key: <UUID v4>`。请求体如下：

```json
{
  "sessionId": "456",
  "prompt": "一座漂浮在云海上的未来城市，日落，电影感",
  "negativePrompt": "模糊，低清晰度",
  "aspectRatio": "16:9",
  "imageCount": 2
}
```

字段约定：

- `sessionId` 可选。传入时必须是当前用户已有会话；不传时，服务端在同一事务中创建会话、消息、任务和执行 Outbox 事件。
- `prompt` 必填，`negativePrompt` 可选。两者先按当前模型能力配置校验本次输入，再分别与会话历史拼接；客户端不得提交已拼接的历史提示词。
- `aspectRatio` 必填，为当前模型能力配置支持的预定义宽高比标识；服务端将其映射为百炼实际尺寸，客户端不得直接传递宽度或高度。
- `imageCount` 必填，当前模型支持 `1`～`6`；服务端仍以任务模型能力配置为准校验。

成功时统一返回 `202 Accepted`：

```json
{
  "code": 0,
  "message": "accepted",
  "data": {
    "taskId": "123",
    "sessionId": "456",
    "status": "QUEUED",
    "taskVersion": 0,
    "requestedImageCount": 2,
    "createdAt": "2026-07-24T10:00:00Z"
  }
}
```

同一用户使用相同 `Idempotency-Key` 且请求指纹一致时，始终返回已有任务的当前状态（HTTP 仍为 `202 Accepted`），不重复创建会话、消息、任务或占用额度；请求指纹不同则返回 `409 IDEMPOTENCY_KEY_CONFLICT`。请求指纹由当前用户、`sessionId`（或“新会话”标识）、正负提示词、宽高比和图片数量的规范化值构成。

### 7.2 获取会话列表

`GET /api/generation-sessions?cursor=&limit=20` 返回当前用户已提交内容的会话摘要，按 `lastMessageAt DESC, sessionId DESC` 稳定排序。不返回完整消息、任务详情、提示词或图片签名 URL；前端进入某一会话后再按需查询。

`cursor` 可选且不透明，由服务端根据最后一项的 `lastMessageAt` 与 `sessionId` 生成；首次请求不传。`limit` 默认 `20`，范围 `1`～`50`。游标不能被解析、已过期或与本接口查询条件不匹配时返回 `422 INVALID_CURSOR`。

```json
{
  "code": 0,
  "message": "ok",
  "data": {
    "items": [
      {
        "sessionId": "456",
        "title": "一座漂浮在云海上的未来城市",
        "lastMessageAt": "2026-07-24T10:00:00Z",
        "latestTask": {
          "taskId": "123",
          "status": "RUNNING",
          "taskVersion": 2
        }
      }
    ],
    "nextCursor": "opaque-cursor",
    "hasMore": true
  }
}
```

`latestTask` 为该会话最近一次提交消息关联的任务摘要；旧会话数据不完整时可为 `null`。`nextCursor` 仅在 `hasMore=true` 时返回非空值。会话标题、`lastMessageAt` 和最新任务摘要足以驱动侧栏展示；任务状态变化通过 SSE 到达后，前端以 `taskId` 和 `taskVersion` 局部更新对应摘要，必要时重新请求列表对账。

### 7.3 获取会话消息

`GET /api/generation-sessions/{sessionId}/messages?before=&limit=30` 用于按历史顺序恢复会话。消息以 `sequenceNo` 为游标：服务端先查询 `before` 之前的较早消息并按 `sequenceNo DESC` 取一页，再将该页按 `sequenceNo ASC` 返回，因此前端可将返回结果追加到现有消息列表顶部。首次请求不传 `before`，从最新消息开始加载。

`before` 可选且不透明，服务端按消息 `sequenceNo` 生成；`limit` 默认 `30`，范围 `1`～`100`。非法、过期或不属于该会话的游标返回 `422 INVALID_CURSOR`。

```json
{
  "code": 0,
  "message": "ok",
  "data": {
    "items": [
      {
        "message": {
          "messageId": "8001",
          "sequenceNo": 1,
          "prompt": "一座漂浮在云海上的未来城市",
          "negativePrompt": "模糊，低清晰度",
          "createdAt": "2026-07-24T10:00:00Z"
        },
        "generation": {
          "taskId": "123",
          "status": "SUCCEEDED",
          "taskVersion": 4,
          "requestedImageCount": 2,
          "completedImageCount": 2,
          "failedImageCount": 0,
          "cancelledImageCount": 0,
          "failureCode": null,
          "failureMessage": null,
          "images": [
            {
              "imageId": "9001",
              "url": "https://example.com/signed-object-url",
              "urlExpiresAt": "2026-07-24T10:10:00Z",
              "width": 2048,
              "height": 1152
            }
          ],
          "createdAt": "2026-07-24T10:00:00Z",
          "completedAt": "2026-07-24T10:03:00Z"
        }
      }
    ],
    "nextCursor": "opaque-cursor",
    "hasMore": true
  }
}
```

每一项固定由 `message` 与 `generation` 组成：前者是一次用户提交，后者是该消息唯一关联任务的完整可展示快照，字段与[任务快照响应](#710-任务快照响应)一致，终态任务包含成功图片的短期签名 URL。前端按数组顺序渲染 `message`，再渲染紧随其后的 `generation` 卡片，无需自行关联两份接口数据。

服务端先按游标取得本页消息，再按本页 `messageId` 批量查询 `source_message_id` 对应任务、按任务 ID 批量查询图片并组装响应，避免 N+1 查询；三个查询处于同一只读数据库事务中。消息接口不得返回最终拼接提示词、模型标识、服务商字段、服务商临时 URL 或密钥。收到 SSE 后，前端按 `generation.taskId` 与 `taskVersion` 更新卡片；签名 URL 过期、需要单任务刷新或需要补齐状态时，调用任务详情接口。`nextCursor` 仅在 `hasMore=true` 时返回非空值。

### 7.4 修改会话标题

`PATCH /api/generation-sessions/{sessionId}` 仅允许修改当前用户会话的标题。请求体：

```json
{
  "title": "新的会话标题"
}
```

服务端去除 `title` 首尾空白后校验长度为 `1`～`100` 个字符；空白标题、超长标题或缺失字段返回 `422 VALIDATION_ERROR`。成功时返回 `200 OK`：

```json
{
  "code": 0,
  "message": "ok",
  "data": {
    "sessionId": "456",
    "title": "新的会话标题",
    "createdAt": "2026-07-24T09:58:00Z",
    "lastMessageAt": "2026-07-24T10:00:00Z"
  }
}
```

会话不存在或不属于当前用户时统一返回 `404 GENERATION_RESOURCE_NOT_FOUND`。标题修改只更新数据库技术字段 `updated_at`，不改变业务字段 `last_message_at`，因此不会改变会话列表排序。

### 7.5 查询未完成任务

`GET /api/generation-tasks?active=true` 返回当前用户处于 `QUEUED`、`RUNNING` 或 `CANCEL_REQUESTED` 的全部任务，用于生成页首次加载和 SSE 重连后的权威状态对账。单用户未完成任务上限为 `4`，因此该接口不分页；它不承担会话侧栏或历史任务浏览职责。

响应 `data` 为任务快照数组，数组元素与[任务快照响应](#710-任务快照响应)使用相同字段和安全边界：

```json
{
  "code": 0,
  "message": "ok",
  "data": [
    {
      "taskId": "123",
      "sessionId": "456",
      "status": "RUNNING",
      "taskVersion": 2,
      "requestedImageCount": 2,
      "completedImageCount": 0,
      "failedImageCount": 0,
      "cancelledImageCount": 0,
      "failureCode": null,
      "failureMessage": null,
      "images": [],
      "createdAt": "2026-07-24T10:00:00Z",
      "completedAt": null
    }
  ]
}
```

不传 `active=true` 或传入其他值返回 `422 VALIDATION_ERROR`，避免该接口被误用为全量历史任务查询。结果按 `createdAt ASC, taskId ASC` 返回，保证客户端恢复时以创建顺序重建活动任务状态。

### 7.6 取消任务

`POST /api/generation-tasks/{taskId}/cancel` 不需要请求体。服务端通过带状态前置条件和 `taskVersion` 的更新处理取消与工作器之间的竞争，并返回更新后的任务快照：

- 任务处于 `QUEUED`，或工作器已领取但 `provider_call_started_at` 仍为空时，直接进入 `CANCELLED`；随后到达的 MQ 消息只确认，不得调用百炼。
- 任务已开始百炼调用时，立即进入 `CANCEL_REQUESTED` 并返回该状态。调用无法由首版服务端中断，工作器仍等待响应、超时或异常；结束后先检查任务状态，若仍为 `CANCEL_REQUESTED`，则丢弃服务商响应，不保存临时 URL、不下载、不上传 OSS、不写入图片记录，最后收敛为 `CANCELLED`。
- 已处于 `CANCEL_REQUESTED` 或 `CANCELLED` 时重复请求为幂等成功，返回当前任务快照。
- `SUCCEEDED`、`PARTIALLY_SUCCEEDED` 或 `FAILED` 为不可取消终态，返回 `409 TASK_ALREADY_FINISHED`。

取消不会返还已在任务创建时占用的每日图片额度；对于已发起百炼调用的任务，也不会减少已占用的实际调用资源，直至工作器结束该调用。任务最终状态变化通过 SSE 推送，客户端也可通过任务详情或未完成任务查询完成状态对账。

### 7.7 第三方数据处理同意

`GET /api/users/me/consents/generation` 返回当前生效规则及当前用户对该版本的同意状态：

```json
{
  "code": 0,
  "message": "ok",
  "data": {
    "policyVersion": "2026-07-24",
    "policyContent": "完整的第三方数据处理规则文案",
    "hasConsented": false,
    "consentedAt": null
  }
}
```

`POST /api/users/me/consents/generation` 请求体仅包含前端实际展示的版本：

```json
{
  "policyVersion": "2026-07-24"
}
```

后端仅接受当前生效版本；成功时创建或更新该用户、该同意类型唯一的一条当前记录，并返回与查询接口相同的状态结构（其中 `hasConsented=true`，`consentedAt` 为本次确认时间）。客户端传入已过期或非当前版本时，返回 `409 GENERATION_CONSENT_VERSION_OUTDATED`，前端必须重新查询并展示最新完整文案。该接口不接受客户端传入规则正文、哈希或同意时间，三者均由服务端维护；首版不提供撤回后继续生成的能力。

### 7.8 SSE 实时通知与断线恢复

SSE 适用于本模块的服务端单向任务状态通知；创建、取消、查询和恢复仍使用 REST，不引入 WebSocket。`GET /api/events` 必须使用支持 `Authorization` 请求头的 Fetch 流式 SSE 实现，浏览器不得将 Access Token 写入 URL 查询参数。每个生成页维持一条连接，同一用户最多三条；离开生成页时主动关闭。Access Token 即将到期时，客户端刷新 Token 后重新建立连接。

事件示例：

```text
event: generation.task.updated
id: 928
data: {"sessionId":"456","taskId":"123","taskVersion":4,"status":"SUCCEEDED"}
```

`id` 为 Outbox 事件 ID，仅用于事件标识和排障；首版不实现基于 `Last-Event-ID` 的历史回放，也不把事件视为任务真相。事件不携带提示词、图片签名 URL、服务商临时 URL或完整任务信息。客户端仅接受 `taskVersion` 大于本地版本的同任务事件；收到后调用任务详情接口获得完整快照与短期签名 URL。

首次连接或重连成功后，客户端必须同时执行两项 REST 对账：查询当前用户全部未完成任务，恢复跨会话仍在执行的状态；重新查询当前打开会话的消息列表，恢复断线期间已经完成、因此不再属于未完成任务的生成结果。SSE 只负责连接存续期间的实时通知，断线、重复或乱序事件均不得影响上述 REST 权威恢复结果。

服务端每 15 秒发送一次 SSE 注释心跳以发现断连；连接写入失败时清理该连接。Nginx / 网关必须关闭 SSE 响应缓冲并配置足够长的读取超时。

幂等规则：前端每次用户主动提交时生成新的 UUID v4；同一次网络重试、超时重试或重复点击必须复用原键。幂等范围为当前用户与幂等键。同一用户使用相同 `Idempotency-Key` 且请求指纹一致时，返回已有任务的当前状态和首次创建的 `sessionId`，不重复占用未完成任务名额或每日额度；若指纹不同，返回 `409 IDEMPOTENCY_KEY_CONFLICT`，不得创建新任务。用户主动再次生成时，即使参数相同，也必须使用新键。

### 7.9 HTTP 状态与错误语义

| 场景 | HTTP 状态 | 稳定错误标识 |
| --- | --- | --- |
| 成功查询、修改或确认规则 | `200` | `0` |
| 成功接收生成任务 | `202` | `0` |
| 参数格式或范围错误 | `422` | `VALIDATION_ERROR` |
| 游标非法、过期或与查询条件不匹配 | `422` | `INVALID_CURSOR` |
| 未携带或 Access Token 无效 | `401` | `UNAUTHORIZED` |
| 会话、任务或图片不存在，或不属于当前用户 | `404` | `GENERATION_RESOURCE_NOT_FOUND` |
| 会话已有活动任务 | `409` | `SESSION_ACTIVE_TASK_EXISTS` |
| 用户未完成任务总量达到上限 | `409` | `USER_GENERATION_CONCURRENCY_LIMIT` |
| 相同幂等键对应不同请求指纹 | `409` | `IDEMPOTENCY_KEY_CONFLICT` |
| 未同意当前第三方规则 | `409` | `GENERATION_CONSENT_REQUIRED` |
| 确认的规则版本已过期或不是当前版本 | `409` | `GENERATION_CONSENT_VERSION_OUTDATED` |
| 任务已经进入不可取消终态 | `409` | `TASK_ALREADY_FINISHED` |
| 每日 12 张额度已用尽 | `429` | `DAILY_GENERATION_QUOTA_EXCEEDED` |
| 生成服务暂时不可用 | `503` | `GENERATION_SERVICE_UNAVAILABLE` |

所有接口继续使用统一 `code`、`message`、`data` 响应；错误标识对应的具体数字值在实现错误码表时统一分配。访问其他用户的私有生成资源与资源不存在统一返回 `404`，不得通过 `403` 或差异化消息泄露资源是否存在。每日额度错误应返回下一次北京时间重置时间；部署层网关如返回 `429`，由网关按其规则决定是否附带 `Retry-After`。服务商原始错误、异常栈和内部执行字段不得返回客户端。异步任务的最终失败通过任务状态和安全失败分类表达，不改变创建请求已经被 `202` 接收的事实。

### 7.10 任务快照响应

`GET /api/generation-tasks/{taskId}` 用于返回任务当前的权威状态和可展示结果，供 SSE 事件后的状态对账、页面刷新或重连恢复，以及私有 OSS 签名 URL 过期后的重新获取。该接口不是数据库任务记录的完整映射。

响应 `data` 使用紧凑任务快照：

```json
{
  "taskId": "123",
  "sessionId": "456",
  "status": "PARTIALLY_SUCCEEDED",
  "taskVersion": 6,
  "requestedImageCount": 2,
  "completedImageCount": 1,
  "failedImageCount": 1,
  "cancelledImageCount": 0,
  "failureCode": "IMAGE_TRANSFER_PARTIAL_FAILURE",
  "failureMessage": "部分图片保存失败",
  "images": [
    {
      "imageId": "9001",
      "url": "https://example.com/signed-object-url",
      "urlExpiresAt": "2026-07-23T12:10:00Z",
      "width": 2688,
      "height": 1536
    }
  ],
  "createdAt": "2026-07-23T12:00:00Z",
  "completedAt": "2026-07-23T12:03:00Z"
}
```

规则：

- `status` 与 `taskVersion` 用于权威状态对账，客户端不得用更旧版本覆盖本地新状态。
- `requestedImageCount`、`completedImageCount` 为持久化数量；`failedImageCount`、`cancelledImageCount` 由接口根据任务状态计算。`QUEUED`、`RUNNING`、`CANCEL_REQUESTED` 三个数量均为 `0`；`SUCCEEDED` 为“请求数、0、0”；`PARTIALLY_SUCCEEDED` 为“成功数、请求数减成功数、0”；`FAILED` 为“0、请求数、0”；`CANCELLED` 为“0、0、请求数”。只有成功转存的图片进入 `images`。
- 非终态任务的 `images` 为空，`failureCode`、`failureMessage` 和 `completedAt` 为 `null`。
- `failureCode` 是持久化的稳定失败分类；`failureMessage` 由接口按错误码生成安全的用户提示，不暴露服务商原始错误。
- `urlExpiresAt` 供前端判断签名 URL 是否仍可复用；图片宽高用于图片加载前预留布局空间。
- 响应必须使用 `Cache-Control: private, no-store`。
- 普通任务快照不得返回模型标识、原始提示词、负向提示词、`finalPrompt`、`finalNegativePrompt`、服务商请求 ID、调用开始时间或临时 URL。原始提示词由会话消息接口恢复，内部追溯信息只保留在服务端。

## 8. 权限、可靠性与安全

- 用户只能创建、查询和取消自己的任务，不能读取其他用户的任务、结果或 OSS 签名 URL。
- 创建请求必须使用 UUID v4 幂等键；同一用户与同一幂等键只允许对应一个任务。相同键和相同规范化请求指纹返回已有任务当前状态，相同键但不同请求指纹必须拒绝。资源所有权查询必须同时使用资源 ID 与当前用户 ID，避免差异化响应泄露其他用户资源。
- Outbox 分发器以数据库条件更新或行锁领取事件；RabbitMQ 收到 Publisher Confirm 后才将执行命令事件标记为已发布。工作器仅在图片记录、任务终态和 SSE Outbox 事件提交后确认 RabbitMQ 消息；消息重投时根据任务终态或转存快照决定直接确认或继续转存，并通过任务状态条件更新和 `(task_id, source_index)` 唯一约束保证幂等。
- Outbox 投递 RabbitMQ 采用有限自动重试；最终失败时将 Outbox 事件标记为 `FAILED` 并将对应任务更新为 `FAILED`，不配置 DLQ、人工重放或运维处理。执行队列使用 Durable Quorum Queue，显式设置 `x-queue-type=quorum`；消费者并发初始为 30、每消费者 `prefetch=1`，不使用全局 QoS。RabbitMQ 消费失败达到上限时同样将任务更新为 `FAILED` 后确认该消息。百炼仅对短时限流 `429 Throttling`、`Throttling.RateQuota`、`Throttling.BurstRate` 最多自动重试 5 次并按指数退避加随机抖动重新排队；调用结果未知时直接失败，避免重复生成与计费。
- 应用定期扫描 `QUEUED` 超过 5 分钟的任务，并通过带前置状态的条件更新将其收敛为 `FAILED + QUEUE_TIMEOUT`，在同一事务中写入 SSE Outbox 事件。队列不设置消息 TTL；超时任务遗留的消息被工作器再次投递时，工作器发现任务已终态后直接 ACK，不执行模型调用。
- 生成限制均为后端配置：每用户跨所有会话、页面和设备最多 4 个未完成任务；每日按请求 `imageCount` 累计最多 12 张，按 `Asia/Shanghai` 自然日结算；全局工作器并发初始配置为 30，全局实际模型调用上限初始也为 30。百炼全局出口初始每秒最多 2 次。相同幂等键且请求指纹一致的重放不消耗限额。前端的单空白新对话规则仅用于体验引导，后端仍以未完成任务数和每日用量为准。`imageCount` 仅允许模型支持的 1～6。
- 首版百炼出口令牌桶只存在单个应用实例内，应用重启后令牌桶重置。迁移到 Redis 等共享限流和跨实例事件分发能力之前，不得水平扩容或拆分独立工作器。扩容后百炼出口限流和 SSE 事件分发都必须使用跨实例共享能力。请求级防刷与 DDoS 防护由可信网关负责配置；应用不直接信任客户端提交的转发 IP 头。
- 百炼模型调用只在后台工作器中进行，前端创建任务后立即收到 `202 Accepted`，后续状态通过 REST 查询与 SSE 获取。工作器连接百炼超过 5 秒仍未建立连接时，视为确认未发出请求，最多自动重试 5 次；已建立连接但等待服务商结果超过 300 秒时，结果状态未知，任务进入 `FAILED + PROVIDER_RESULT_UNKNOWN`，不得自动重试。HTTP 客户端禁止自动重定向和自动重试，必须校验证书与主机名；最大并发连接数初始为 30，并且不得记录完整请求体、响应临时 URL 或 API Key。
- 用户首次生成前必须读取并明确同意当前版本的第三方数据处理规则；后端提供当前版本、完整文案和同意状态查询，确认请求携带版本且只接受当前有效版本。创建任务前校验该同意记录，未同意或版本过期时拒绝创建，且不写入会话、任务、消息、用量或 Outbox 数据。规则需明确告知：生成所需的提示词和会话上下文会发送至阿里云百炼，生成内容会被保存并可在后续资产功能中删除，同时提示用户不要输入敏感个人信息。
- 百炼 API Key 和 OSS 凭证在本地仅允许写入未提交的 `application-local.yaml`，在生产环境仅通过环境变量注入。服务商 API Key、OSS 密钥、临时图片 URL、签名 URL、完整提示词和原始服务商错误不得写入客户端响应或普通业务日志。
- 迭代 2 的私有生成仅依赖服务商内容安全能力；服务商内容安全拒绝应映射为稳定、面向用户的失败分类，不得暴露服务商原始错误。当前不做本地关键词拦截或独立内容审核。后续如有需要，再为发布记录设计独立审核状态，并使公开灵感流仅展示审核通过的作品。注册、登录以及请求级防刷限流属于后续认证安全与部署层网关职责；用户级并发任务数以数据库状态为准。
- Nginx / 网关必须关闭 SSE 响应缓冲并设置足够长的读取超时。
- SSE 定期发送心跳；Spring MVC 通过 `SseEmitter` 支持 SSE，断线恢复依赖任务状态对账而不是假设事件绝不丢失。参见[Spring SSE 文档](https://docs.spring.io/spring-framework/reference/web/webmvc/mvc-ann-async.html)。
- 迭代 2 的提示词、会话历史和图片均为私有生成数据。迭代 4 发布时，用户必须明确确认：将公开该图片所属会话中截至本次生成的完整用户正向、负向提示词历史；不得公开服务商参数、工作流配置或系统提示词。
- 迭代 2 默认长期保存图片、会话、提示词和任务记录，不提供删除入口；迭代 3 必须交付会话或生成资产删除能力，不得继续无限延期。存在活动任务的会话不得删除。删除时应清理私有图片、消息和任务业务数据；OSS 对象采用异步删除并记录清理状态，直到对象清理完成。已发布图片必须先撤销发布或由删除流程自动撤销公开副本，具体语义在迭代 4 确认。
- 用户生成图片、会话、提示词和任务记录首版不自动过期。生产数据库必须具备加密备份与可验证恢复能力，备份保留 7 天；具体实施可使用 RDS 自动备份，或自建 MySQL 的定时全量备份加二进制日志或等效方案。正式上线前必须完成至少一次恢复演练。备份副本的自然淘汰不改变应用库中的业务保留语义。

### 8.1 基础技术监控

首版使用 Micrometer 与 Spring Boot Actuator 记录和暴露技术指标，不要求在本迭代搭建 Prometheus、Grafana 或完整告警平台。监控接口必须限制为内网、运维网络或受认证保护的访问，不能直接暴露至公网。

至少记录以下聚合指标：

- `QUEUED`、`RUNNING`、`FAILED` 与 `PROVIDER_RESULT_UNKNOWN` 的任务数量，以及最久排队时间；
- Outbox `PENDING`、`FAILED` 数量及最久等待时间；
- 百炼调用耗时、调用次数和按安全失败分类聚合的失败次数；
- OSS 下载、校验和上传失败次数；
- 并发上限与每日额度拒绝次数。

指标标签不得使用 `userId`、`taskId`、会话 ID、提示词或图片 URL，避免高基数和敏感数据泄露。排障日志可记录 `taskId`、会话 ID、服务商请求 ID 和安全失败分类，但不得记录提示词、URL 或密钥。业务/产品行为埋点不属于本模块首版范围。

## 9. 实施顺序与验收

实施顺序：

1. 确认会话、消息、任务、图片、用户行锁和每日用量之间的数据关系，以及 MySQL 表、索引、Flyway 迁移与 Outbox 事件模型。
2. 配置私有 OSS，完成安全下载、文件校验、上传、短期签名 URL 与最小权限访问控制。
3. 配置 RabbitMQ 和任务/事件分发基础设施；首版单实例不部署 Redis Pub/Sub，多实例部署时再补充。
4. 实现首版文生图执行器与百炼 Qwen-Image 调用。
5. 实现会话行锁、`users` 行锁与最多 4 个未完成任务的约束，任务状态机、`provider_call_started_at`、UUID v4 幂等、有限安全重试、取消竞态、失败分类、任务级图片结果汇总和服务商结果转存恢复。
6. 实现会话历史恢复、最终模型输入保存，以及 SSE 连接、心跳、事件分发与断线状态对账。
7. 补充单元测试、MySQL 集成测试、RabbitMQ/OSS 集成测试、安全下载测试、工作器崩溃恢复测试、并发名额与每日额度原子性测试、基础监控指标测试和真实百炼联调验收；首版不要求 Redis 集成测试。

验收标准：

- 同一幂等键不会创建或执行重复任务。
- 用户可创建 1～6 张的普通文生图任务，并仅能读取自己的结果。
- 用户进入生成页可使用空白创作框；首次提交会创建并返回会话，已提交会话可恢复、改名并按最近活动时间显示在不限数量的侧栏列表中。空白创作框成功提交后才可开启下一个新会话。
- 同一会话不会同时存在两个未完成任务；同一用户跨所有会话、页面和设备最多同时存在 4 个未完成任务；由会话行锁、`users` 行锁和任务状态统计提供数据库级保护。关闭会话页面只会断开 SSE，不影响已创建任务的执行；下次加载会话和任务详情可恢复请求与结果。
- 并发创建请求不会突破单用户 4 个未完成任务或每日 12 张请求图片额度；任务创建后立即占用每日额度，幂等重放不重复占用，任务终态不返还每日额度。
- 百炼出口初始为全局每秒 2 次。明确的短时百炼限流 `429 Throttling`、`Throttling.RateQuota`、`Throttling.BurstRate` 最多自动重试 5 次，并按指数退避加随机抖动重新排队；`AllocationQuota` 等配额/计费问题不自动重试。
- 用户重新打开页面后可恢复会话历史、任务状态和已生成图片；SSE 事件可定位到对应会话与任务。
- 每次生成任务均保留最终调用的正向、负向提示词；会话消息可恢复历史提示词，满足模型输入追溯需要。
- 所有请求图片均成功转存私有 OSS 后，任务进入 `SUCCEEDED`；至少一张成功转存但未全部成功时，任务进入 `PARTIALLY_SUCCEEDED`；没有任何图片成功转存时进入 `FAILED`。客户端只展示成功图片及任务级成功、失败、取消数量，不展示失败图片槽位；已有至少一张成功图片的任务不得进入 `FAILED`。
- 用户在百炼调用已发出后取消任务时，任务先进入 `CANCEL_REQUESTED`，调用结束后进入 `CANCELLED`；该次调用返回的图片不得写入结果记录或个人资产。
- 任务状态变化可通过 SSE 通知在线客户端；断线重连后客户端通过“未完成任务查询 + 当前会话消息重载”恢复正确状态。
- 可重试与不可重试失败得到不同处理；最终失败对用户返回安全的失败说明。
- 调用结果未知时不会自动重试，避免重复生成与重复计费。
- 工作器将任务条件更新为 `RUNNING` 后，在实际 HTTP 调用前提交写入 `provider_call_started_at`。消息重投时：调用开始时间为空的任务可重新领取；调用开始时间非空但未保存服务商结果快照的任务进入 `FAILED + PROVIDER_RESULT_UNKNOWN`，不得再次调用模型。百炼明确返回短时限流 `Throttling`、`Throttling.RateQuota` 或 `Throttling.BurstRate` 时，清空该时间、增加 `attempt_count` 并重新排队。
- 服务商已返回图片临时结果时，先保存仅供转存恢复使用的 `request_id + source_index + 临时 URL` 快照，再执行 OSS 下载与上传；工作器崩溃后只允许恢复转存，不得再次调用模型。全部结果转存完成后立即清除该快照；URL 超过 24 小时仍无法恢复时，保留已成功图片并将未完成部分收敛为失败。
- 同一幂等键且参数相同不会创建重复任务；相同幂等键但参数不同返回冲突。
- 任务、图片、Outbox 事件和任务状态变更具备可追溯性；签名 URL、密钥和完整提示词不出现在普通日志或 SSE 事件中。
- Outbox、RabbitMQ 和 SSE 事件仅传递最小标识；工作器根据 `taskId` 从 MySQL 读取任务详情，不在消息载荷中复制提示词、上下文或 URL。
- 基础技术指标可在受保护的 Actuator 监控端点读取，且不包含用户、任务或提示词等高基数/敏感标签。
