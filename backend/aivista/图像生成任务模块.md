# AiVista 图像生成任务模块

> 对应迭代：迭代 2——文生图与局部重绘任务  
> 当前状态：详细设计中  
> 本文档只记录已确认的设计决策与待确认项；未实现内容不得视为已交付。

## 1. 目标与范围

本模块负责创建、执行、追踪和通知 AI 图像生成任务，并将生成结果可靠存储到项目自有的阿里云 OSS。

当前优先实现的能力：

- 用户创建普通文生图任务。
- 支持用户创建、切换和恢复多个持久化生成会话。
- 同一会话同一时间仅允许一个活动生成任务；不同会话可独立排队或生成。
- 持久化会话内的对话历史，并为后续模型调用保留上下文记忆。
- 后端异步调用阿里云百炼 Qwen-Image。
- 一次任务可生成的图片数量由实际调用模型的能力决定。
- 将图片从服务商临时地址转存至私有 OSS。
- 用户通过 SSE 获得任务状态变化通知，并通过 REST 查询任务详情。

后续在同一任务模型中扩展：

- 局部重绘任务。
- 用户主动选择的 Agent / 工作流执行模式。

不属于当前实现范围：

- 局部重绘具体输入、遮罩和编辑 API。
- Agent / 工作流定义、编排、版本管理和界面。
- 社区发布、收藏、点赞、关注和完整生成历史浏览。
- 用户计费、订单和付费配额。

## 2. 已确认设计决策

| 主题 | 决策 |
| --- | --- |
| 图像服务 | 使用阿里云百炼 Qwen-Image API。 |
| 默认模型 | 当前固定使用 `qwen-image-2.0-pro`；每个任务保存实际调用的模型标识。 |
| 存储 | 服务商返回的临时图片 URL 由后端下载并转存至私有阿里云 OSS，不作为长期资产地址。 |
| 图片格式 | 首版仅接收和保存 PNG，不做格式转换。下载时响应 `Content-Type`、文件魔数和实际解析格式必须一致且均为 PNG；不符合时视为该图片转存失败。 |
| 百炼 HTTP 调用 | 前端创建任务后立即获得 `202 Accepted`；后台工作器再通过 HTTP 同步调用百炼。工作器连接百炼超过 5 秒仍未建立连接时，视为确认未发出请求，可按有限次数重试；已建立连接但等待结果超过 300 秒时，视为服务商结果未知，任务进入 `FAILED + PROVIDER_RESULT_UNKNOWN`，绝不自动重试。工作器最大同时发起 6 个模型调用；真正调用百炼前还必须取得全局出口令牌，初始限制为每分钟 2 次。 |
| 第三方数据处理同意 | 用户首次创建生成任务前，必须读取并明确同意“本次提示词及生成所需会话上下文将发送至阿里云百炼处理”的当前规则版本；后端返回当前版本、完整文案和用户同意状态，并记录用户实际确认的版本、文案摘要和同意时间。确认请求必须携带版本，且只接受当前有效版本；规则升级后必须重新确认。 |
| 任务类型 | `TEXT_TO_IMAGE`、`INPAINT`；当前仅开放 `TEXT_TO_IMAGE`。 |
| 执行模式 | `DIRECT`、`WORKFLOW`；当前仅开放 `DIRECT`。`WORKFLOW` 未来由用户主动选择。 |
| 执行方式 | 所有模式统一由后端异步任务系统执行；`DIRECT` 仅表示直接调用模型服务，不表示 HTTP 请求同步等待结果。 |
| 会话规则 | 会话列表展示用户所有已提交内容的持久化会话，不限制数量。进入生成页默认打开一个不持久化的空白创作框；首次提交时原子创建会话和任务，成功提交后才可再从侧栏开启下一个空白创作框。默认标题截取首条正向提示词，用户可后续修改。单个会话只允许一个活动任务，单个用户跨所有会话和浏览器页面同时最多 2 个活动任务。 |
| 任务创建限流 | 首版对新建任务启用应用内令牌桶限流：单个已登录用户每分钟最多 2 个、单个 IP 每分钟最多 6 个。相同幂等键且请求参数一致的重放不消耗限额；触发时返回 `429 Too Many Requests` 和 `Retry-After`。注册、登录限流不属于本迭代。 |
| 百炼出口限流 | 首版单实例使用应用内全局令牌桶，初始每分钟最多调用百炼 2 次。任务可以先进入 RabbitMQ；工作器取得出口令牌后才调用百炼。百炼明确返回 `429/Throttling` 且请求未被接受时，按配置化退避重新排队，不重复占用用户额度；任务总排队时间仍不得超过 5 分钟。 |
| 每日额度时区 | 每日 12 张图片额度按 `Asia/Shanghai`（北京时间）自然日统计，每日 00:00 重置；数据库生命周期时间仍统一以 UTC 存储。 |
| 历史与上下文 | 当前 `DIRECT` 文生图的一轮历史由用户提示词消息和 AI 生成结果消息组成；结果消息不含文本，通过关联任务取得状态和图片。同一会话始终继承历史正向提示词，历史负向提示词不继承；模型调用前由应用按既定长度规则拼接历史正向提示词和本次正向提示词。前端必须提示用户一个会话只围绕一个图片主题创作，更换主题时新建会话。 |
| 实时通知 | 使用 SSE 将任务状态变化主动推送给在线用户；REST 接口负责详情查询和断线后的状态对账。 |
| 执行队列 | 首版使用 Durable Quorum Queue `generation.task.execute`，声明时显式设置 `x-queue-type=quorum`。单节点部署时以 1 个成员运行；需要节点级高可用时扩展为 3 节点、3 成员。消费者并发固定为 6、每消费者 `prefetch=1`、使用手动 ACK。 |
| 排队超时 | 任务处于 `QUEUED` 超过 5 分钟时，由应用主动收敛为 `FAILED + QUEUE_TIMEOUT`，释放活动任务名额并通知客户端；不使用 RabbitMQ 消息 TTL，避免消息过期后任务真相无法同步。 |
| 数据保留与备份 | 用户生成图片、会话、提示词、上下文快照和任务记录首版不自动过期。服务商临时 URL 快照全部转存后立即清除，最长保留 24 小时；已发布 Outbox 事件保留 7 天，失败 Outbox 事件保留 14 天。生产数据库必须具备加密备份与可验证恢复能力，备份保留 7 天；使用 RDS 时采用其自动备份，自建 MySQL 时采用定时全量备份加二进制日志或等效方案，并在正式上线前完成至少一次恢复演练。 |
| 首版部署 | 首版按单个 Spring Boot 应用实例设计：同一进程处理 HTTP/SSE 请求并消费 RabbitMQ 任务，应用内直接向 SSE 连接推送事件，并使用应用内用户/IP创建限流器和百炼出口限流器，不启用 Redis。迁移到 Redis 等共享限流和跨实例事件分发能力之前，不得水平扩容或拆分独立工作器。 |
| 任务真相来源 | MySQL 任务与结果记录是权威状态；SSE 仅为实时通知通道。 |
| 服务商密钥 | 本地开发可在未提交的 `application-local.yaml` 填写开发密钥；环境变量始终可覆盖配置。首版生产环境统一通过环境变量注入百炼 API Key 与 OSS 凭证，绝不下发客户端、写入代码仓库或记录到日志。 |

首版固定使用 Qwen-Image 2.0 Pro，向用户提供 1～6 张的数量选择和五种预定义尺寸；服务端也必须校验该请求范围和尺寸映射，防止绕过前端直接调用 API 传入非法参数。不提供模型选择，也不自动降级或切换模型。模型调用本身为同步方式，因此由后台工作器包装为异步任务执行。编码前必须在目标地域和业务空间完成真实百炼联调，确认模型可用性、单次最大图片数、支持尺寸、`negative_prompt`、`prompt_extend=false`、并发限制、计费口径和错误码分类；若结果与本文档不一致，应先更新配置与文档再实施。详见[阿里云 Qwen-Image API 文档](https://help.aliyun.com/zh/model-studio/qwen-image-api)。

### 2.1 千问 Qwen-Image 调用参考

> 官方文档：[千问 Qwen-Image 文生图 API](https://help.aliyun.com/zh/model-studio/qwen-image-api)（访问与实现前应再次核对模型可用地域、版本、限流和计费规则）。

当前 `DIRECT` 执行器调用百炼的同步多模态生成接口；项目层面的异步由 RabbitMQ 工作线程负责，不应将本项目任务状态机与百炼异步接口混用。

| 项目 | 约定 |
| --- | --- |
| 默认模型 | `qwen-image-2.0-pro`；当前固定使用该模型，每次调用均保存实际模型标识。该模型支持同步接口。 |
| 北京地域端点 | `POST https://{WorkspaceId}.cn-beijing.maas.aliyuncs.com/api/v1/services/aigc/multimodal-generation/generation` |
| 鉴权 | 请求头 `Authorization: Bearer ${DASHSCOPE_API_KEY}`；API Key 与地域对应，严禁写入客户端、代码仓库或日志。 |
| 请求头 | `Content-Type: application/json`。 |
| 正向提示词 | `input.messages` 必须且只能包含一条 `user` 消息；其 `content` 必须且只能包含一个 `text`。`qwen-image-2.0` 系列上限为 1300 Token，服务端应在提交前校验项目自身长度限制。 |
| 反向提示词 | `parameters.negative_prompt` 可选，最长 500 字符；项目接口字段 `negativePrompt` 映射至该参数。 |
| 尺寸 | `parameters.size` 格式为 `宽*高`。`qwen-image-2.0` 系列总像素范围为 512×512 至 2048×2048；本项目仅允许预定义的宽高比映射值。 |
| 生成数量 | `parameters.n` 为 1～6；前端提供该范围选择，服务端执行同范围校验。 |
| 提示词改写 | `parameters.prompt_extend` 官方默认 `true`；当前普通模式固定传 `false`，以保证保存的原始提示词与实际输入一致。 |
| 返回图片 | 服务商返回临时公开 URL；工作线程必须下载、校验并转存至私有 OSS 后，才可将任务置为 `SUCCEEDED`。 |

同步请求体结构如下。实现时只替换模型、提示词和已校验的业务参数：

```json
{
  "model": "qwen-image-2.0-pro",
  "input": {
    "messages": [
      {
        "role": "user",
        "content": [{ "text": "正向提示词" }]
      }
    ]
  },
  "parameters": {
    "negative_prompt": "反向提示词",
    "size": "2048*2048",
    "n": 1,
    "prompt_extend": false,
    "watermark": false
  }
}
```

注意：百炼的异步文生图接口当前仅适用于 `qwen-image-plus` 和 `qwen-image`，不适用于 `qwen-image-2.0-pro`。若未来切换到百炼异步接口，需重新设计服务商任务 ID 的轮询、超时与重试策略，不复用当前同步执行器的调用逻辑。

## 3. 核心架构

```text
创建任务 API
    ↓
MySQL：任务记录 + Outbox 事件（同一事务）
    ↓
RabbitMQ：生成任务队列
    ↓
任务工作器：DIRECT 执行器调用百炼 Qwen-Image
    ↓
模型返回临时 URL → MySQL：保存转存快照
    ↓
下载临时结果 → 上传私有 OSS
    ↓
MySQL：图片记录 + 任务终态 + SSE Outbox 事件（同一事务）
    ↓
RabbitMQ：工作器 ACK 执行消息
    ↓
应用内 SSE 分发：向所属用户推送任务状态变化
    ↓
多实例部署时：Redis Pub/Sub 跨实例分发
```

组件职责：

| 组件 | 职责 |
| --- | --- |
| MySQL | 保存任务、图片、Outbox 事件；是唯一权威状态来源。 |
| RabbitMQ | 使用 Durable Quorum Queue 承载生成任务、确认与有限重试。执行队列显式声明 `x-queue-type=quorum`，消费者并发为 6、每消费者 `prefetch=1`、使用手动 ACK。 |
| 阿里云百炼 | 生成图像。 |
| 阿里云 OSS | 保存项目拥有的生成结果文件。 |
| Redis Pub/Sub | 向持有 SSE 连接的应用节点分发实时通知；首版单实例不启用，多实例部署时启用；不作为持久化消息来源。 |
| SSE | 向浏览器单向推送任务变化。 |

采用 Transactional Outbox：任务状态与待分发事件必须在同一数据库事务内写入。创建任务时记录“投递 RabbitMQ 执行命令”；任务状态变化时记录“通知所属用户”的事件。执行事件仅携带 `eventId`、`taskId` 与 `taskVersion`，工作器必须按 `taskId` 从 MySQL 读取任务参数、提示词与上下文；SSE 事件仅携带会话 ID、任务 ID、版本和状态。事件载荷不得包含提示词、上下文快照、临时 URL、签名 URL 或密钥。任务投递事件采用有限自动重试；超过上限则将对应任务收敛为 `FAILED`，提示用户重新发起，不引入人工重放或运维处理链路。SSE 事件发送失败不改变任务真相，客户端通过 REST 状态对账恢复。

工作器使用手动 ACK：模型返回的临时 URL 快照、图片记录、任务终态和 SSE Outbox 事件均已提交后，才 ACK RabbitMQ 执行消息。工作器在此之前崩溃时，RabbitMQ 重投同一任务；新工作器读取任务状态，已有快照则只恢复 OSS 转存，终态任务则直接 ACK，不重复调用模型。

## 4. 任务模型

### 4.1 类型与模式

```text
task_type
- TEXT_TO_IMAGE
- INPAINT

execution_mode
- DIRECT
- WORKFLOW
```

当前允许的组合只有 `TEXT_TO_IMAGE + DIRECT`。未来扩展组合时不修改任务基础生命周期与结果模型。

### 4.2 会话与任务规则

- 会话是用户可持久化恢复的生成对话单元；用户离开页面、关闭浏览器或切换会话不得中断已创建的后台任务。
- 每个生成任务必须归属于一个会话。会话列表展示全部已提交内容的会话，不限制数量。页面初始空白创作框不落库；其首次提交在同一事务中创建会话、任务、消息与任务 Outbox 事件。仅当该空白创作框成功提交后，前端才允许侧栏“新建会话”开启下一个空白创作框，避免无内容会话。
- 首次提交时由后端截取首条正向提示词生成默认会话标题；用户可后续修改标题。会话列表仅展示已提交内容的会话，按最近消息时间倒序。
- 同一会话在 `QUEUED`、`RUNNING` 或 `CANCEL_REQUESTED` 状态下，不得创建第二个活动任务；同一用户跨所有会话、浏览器页面和设备同时最多 2 个活动任务。前端默认仅保留一个空白新对话，且该对话成功提交后才允许新建下一个空白对话；后端仍必须以数据库约束为准，不能仅依赖前端按钮状态。创建任务时，在同一事务内锁定用户生成门闩、占用活动槽位和当日图片额度，再创建任务与执行 Outbox 事件。达到并发上限时返回 `409 Conflict` 与错误码 `USER_GENERATION_CONCURRENCY_LIMIT`；任一任务进入终态即释放活动槽位。
- 会话历史是对话记忆的权威来源。当前 `DIRECT` 文生图的一轮历史由用户提示词消息和 AI 生成结果消息构成；结果消息关联任务，由任务再关联状态和结果图片。每次调用模型时，应用需要保存该次任务使用的上下文快照和最终调用提示词，以便历史恢复和结果追溯。
- 当前文生图模型调用为单轮调用，不会天然记住上一轮内容，也不产生 AI 文本内容；AI 生成结果消息只承载任务引用和图片结果。每次生成均拼接会话中的历史正向提示词与当前正向提示词；超过当前模型提示词长度限制时，仅从本次拼接结果中依次排除最久远的历史提示词，绝不删除历史消息或截断当前提示词。当前提示词自身超长时，直接返回参数校验错误，不创建任务、不调用模型。历史负向提示词不拼接，本次负向提示词仅传服务商 `negative_prompt` 参数。
- Agent / 工作流模式未来可产生思考过程或文本回复；相关消息类型和内容在该模式落地时启用，不影响当前 `DIRECT` 文生图的记录方式。

### 4.3 状态机

```text
QUEUED
  ├── CANCELLED
  └── RUNNING
        ├── SUCCEEDED
        ├── PARTIALLY_SUCCEEDED
        ├── FAILED
        └── CANCEL_REQUESTED → CANCELLED / CANCELLED_WITH_RESULTS
```

规则：

- 状态只允许按既定方向迁移，终态不可再次执行。工作器领取与所有状态变更均使用带前置状态和 `task_version` 的数据库条件更新；受影响行数为 `1` 才代表成功，受影响行数为 `0` 的工作器不得执行模型调用。不得将悲观锁或数据库事务跨越百炼调用、图片下载或 OSS 上传。
- 请求数量的全部图片均成功转存 OSS 时，任务进入 `SUCCEEDED`；至少一张成功转存、但未全部转存时进入 `PARTIALLY_SUCCEEDED`；没有任何图片成功转存时进入 `FAILED`。
- 用户取消时：仍在队列中、工作器尚未领取或已经领取但尚未向百炼发出请求的任务直接进入 `CANCELLED`；百炼请求已经发出但尚未返回时进入 `CANCEL_REQUESTED`，且可能已经产生费用。百炼已经返回但尚未下载图片时不得开始下载；已有部分图片成功转存时停止处理剩余图片并进入 `CANCELLED_WITH_RESULTS`，否则进入 `CANCELLED`。取消与工作器领取通过带前置状态、`task_version` 和执行阶段的条件更新竞争。重复取消 `CANCEL_REQUESTED`、`CANCELLED` 或 `CANCELLED_WITH_RESULTS` 属于幂等成功；对 `SUCCEEDED`、`PARTIALLY_SUCCEEDED` 或 `FAILED` 再取消返回 `409 TASK_ALREADY_FINISHED`。取消不返还每日额度，`CANCELLED_WITH_RESULTS` 的成功图片仍进入个人历史资产。
- 仅“确认请求尚未发送到服务商”的网络失败可自动重试。请求可能已发出但发生超时、连接中断等结果未知情况时，不得自动重试，以避免重复生成与重复计费；任务进入 `FAILED`，内部失败码记为 `PROVIDER_RESULT_UNKNOWN`，对用户仅返回安全的失败说明。
- 参数错误、内容违规等不可重试失败直接进入 `FAILED`。
- 每次状态变化递增 `task_version`，客户端仅接受比本地版本新的事件。

### 4.4 核心数据表

以下资源主键统一使用 `BIGINT UNSIGNED` 自增；对外 HTTP 响应统一序列化为字符串。所有时间使用 `DATETIME`，创建和更新时间由数据库维护。

#### `generation_sessions`：生成会话

| 字段 | 规则与职责 |
| --- | --- |
| `id` | 会话 ID。 |
| `user_id` | 所属用户 ID，外键关联 `users.id`。 |
| `title` | 会话名称，`VARCHAR(100)`；首次提交时由后端截取首条正向提示词生成默认标题，用户可修改。 |
| `context_summary` | 当前会话的上下文摘要，`LONGTEXT`，可空；具体内容格式随上下文编排策略确定。 |
| `context_version` | 上下文版本号，`INT UNSIGNED`，用于识别摘要更新。 |
| `active_task_id` | 当前活动任务 ID，可空；创建任务时原子设置，任务进入终态时清空，用于保证同一会话只有一个活动任务。 |
| `last_message_at` | 最近一条消息时间，用于用户会话列表排序。 |
| `created_at` / `updated_at` | 会话创建与更新时间。 |

索引与约束：`idx_generation_sessions_user_last_message(user_id, last_message_at)`；`active_task_id` 仅可指向该会话的活动任务，具体外键与事务实现随迁移脚本确定。

#### `generation_messages`：会话历史消息

| 字段 | 规则与职责 |
| --- | --- |
| `id` | 消息 ID。 |
| `session_id` | 所属生成会话 ID，外键关联 `generation_sessions.id`。 |
| `sequence_no` | 会话内递增序号，`INT UNSIGNED`；用于稳定恢复消息顺序。 |
| `role` | 当前 `DIRECT` 模式使用 `USER` 和 `ASSISTANT`：前者表示用户提示词，后者表示 AI 生成结果；`SYSTEM` 为系统消息预留。 |
| `message_type` | 当前使用 `PROMPT` 和 `GENERATION_RESULT`；`TEXT`、`TASK_STATUS` 等类型为后续 Agent / 工作流预留。 |
| `content` | 文本内容，`LONGTEXT`，可空；`PROMPT` 保存用户提示词，当前 `GENERATION_RESULT` 可为空。Agent / 工作流模式的 AI 文本回复未来保存在此字段。 |
| `task_id` | 关联生成任务 ID，可空；当前 AI 生成结果消息必须关联一个任务。图片 ID 通过 `task_id → generation_images` 获取，不重复存储。 |
| `created_at` | 消息创建时间；历史消息创建后不修改。 |

索引与约束：唯一索引 `uk_generation_messages_session_sequence(session_id, sequence_no)`；索引 `idx_generation_messages_session_created(session_id, created_at)`。

创建任务时，同一事务中写入用户 `PROMPT` 消息、关联任务和一条 `GENERATION_RESULT` AI 消息。结果消息在生成中即存在，客户端通过其 `task_id` 展示任务状态；任务完成后，通过 `task_id → generation_images` 取得 1～6 张图片。消息表不保存 OSS 签名 URL；图片记录只保存私有对象定位信息，客户端再按权限获取短期 URL。

#### `generation_tasks`：异步生成任务

| 字段 | 规则与职责 |
| --- | --- |
| `id` | 任务 ID。 |
| `user_id` / `session_id` | 所属用户与会话；`session_id` 关联 `generation_sessions.id`。 |
| `source_message_id` | 触发本次生成的用户提示词消息 ID。 |
| `task_type` / `execution_mode` | 图像任务类型与执行模式。 |
| `provider` / `model` | 服务商标识与实际调用模型标识。 |
| `status` / `task_version` | 当前任务状态，以及乐观锁和 SSE 事件去重版本。 |
| `activity_slot` | 活动任务槽位，`TINYINT UNSIGNED`，可空；活动任务只能为 `1` 或 `2`，终态任务必须为 `NULL`。唯一索引 `(user_id, activity_slot)` 从数据库层保证单用户最多两个活动任务。 |
| `execution_stage` / `lease_expires_at` / `worker_id` / `attempt_count` | 内部执行阶段、工作器租约、执行工作器标识和可安全重试次数；不直接暴露给客户端。 |
| `original_prompt` / `negative_prompt` | 用户本次原始正向、负向提示词快照。 |
| `context_snapshot` | 本次生成实际保留并参与拼接的历史正向提示词快照，`JSON`；被长度限制排除的历史仍保留在会话消息中，但不出现在该快照。 |
| `final_prompt` | 最终传给图像模型的正向提示词，`LONGTEXT`；用于生成追溯。 |
| `width` / `height` | 实际请求的输出尺寸。 |
| `requested_image_count` / `completed_image_count` / `failed_image_count` / `cancelled_image_count` | 请求数量、实际成功转存数量、最终失败数量与因取消不再处理数量；不预创建失败图片槽位。 |
| `provider_request_id` | 服务商请求追踪标识。 |
| `provider_result_snapshot` | 服务商成功响应的短期转存快照：整次 `provider_request_id`、临时 URL 列表及各 URL 的内部 `source_index`。仅用于 OSS 转存恢复；仅允许后端读写、不返回客户端、不写入普通日志，全部转存后立即清除，最长不超过服务商 URL 的 24 小时有效期。生产 RDS 必须开启静态加密，并使用最小权限数据库账号与网络隔离。 |
| `idempotency_key` / `request_fingerprint` | 创建请求幂等键与请求参数摘要。幂等键必须为 UUID v4；指纹使用 SHA-256，覆盖规范化后的 `sessionId`（包括未传状态）、`prompt`、`negativePrompt`、`aspectRatio`、`imageCount`、任务类型和执行模式。两者随任务长期保存。 |
| `failure_code` / `failure_message` | 对用户安全的失败分类与说明。 |
| `created_at` / `updated_at` / `started_at` / `completed_at` | 任务生命周期时间。 |

索引与约束：唯一索引 `uk_generation_tasks_user_idempotency(user_id, idempotency_key)` 和 `uk_generation_tasks_user_activity_slot(user_id, activity_slot)`；索引 `idx_generation_tasks_session_created(session_id, created_at)` 和 `idx_generation_tasks_user_status(user_id, status)`。MySQL 唯一索引允许多个 `NULL`，因此历史终态任务可同时保留，活动任务只能分别占用槽位 `1`、`2`。

#### `generation_images`：生成结果图片

| 字段 | 规则与职责 |
| --- | --- |
| `id` | 图片 ID。 |
| `task_id` / `user_id` | 所属任务与用户。会话归属通过任务关联获得，不重复存储 `session_id`。 |
| `oss_bucket` / `object_key` | 私有 OSS 对象定位信息，永不保存短期签名 URL。 |
| `content_type` / `file_size` | 文件元数据；首版 `content_type` 固定为 `image/png`。 |
| `width` / `height` | 实际图像尺寸。 |
| `source_index` | 服务商结果的内部来源标识，仅用于崩溃恢复时避免重复转存；不作为用户可见的失败槽位或顺序。 |
| `created_at` | 保存时间。 |

索引与约束：唯一索引 `uk_generation_images_task_source(task_id, source_index)`；索引 `idx_generation_images_user_created(user_id, created_at)`，为迭代 3 的个人资产查询提供基础。仅在图片成功校验并转存 OSS 后创建记录；单张失败仅汇总至任务字段。

#### `outbox_events`：可靠事件分发

| 字段 | 规则与职责 |
| --- | --- |
| `id` | 事件 ID。 |
| `aggregate_type` / `aggregate_id` | 聚合类型与 ID；本模块主要为生成任务。 |
| `event_type` | 事件类型，例如任务状态更新。 |
| `destination` | 分发目的地：`RABBITMQ_TASK` 或 `SSE_NOTIFICATION`。 |
| `payload` | 最小事件载荷，`JSON`。`RABBITMQ_TASK` 仅包含 `eventId`、`taskId`、`taskVersion`；`SSE_NOTIFICATION` 仅包含 `sessionId`、`taskId`、`taskVersion`、`status`。不得保存提示词、上下文、临时 URL、签名 URL 或密钥。 |
| `payload_version` | 事件载荷版本，供后续兼容演进。 |
| `status` / `retry_count` | 分发状态（`PENDING`、`PROCESSING`、`PUBLISHED`、`FAILED`）与重试次数。 |
| `available_at` / `locked_at` / `locked_by` / `published_at` | 下次可分发时间、分发租约与成功分发时间。 |
| `last_error` | 最近一次分发失败的内部错误摘要，不包含密钥、临时 URL 或完整提示词。 |
| `created_at` | 事件创建时间。 |

保留规则：已发布事件保留 7 天，失败事件保留 14 天后由定时清理任务删除；任务和图片记录仍是业务追溯来源，不依赖 Outbox 长期保存。

#### `user_generation_gates`：用户生成门闩

每个用户一行，用于串行化同一用户的任务创建事务；不保存冗余活动任务计数。

| 字段 | 规则与职责 |
| --- | --- |
| `user_id` | 用户 ID，主键，关联 `users.id`。 |
| `updated_at` | 最近一次任务创建锁定时间。 |

创建任务时先对当前用户门闩行执行 `SELECT ... FOR UPDATE`，再从 `1`、`2` 中选择未被活动任务占用的槽位。没有空槽位时返回 `409 USER_GENERATION_CONCURRENCY_LIMIT`。任务进入任一终态时，在同一终态事务中将 `activity_slot` 置为 `NULL`。应用启动后及定时任务应检查“活动状态必须持有槽位、终态不得持有槽位”，异常时告警并按任务权威状态修复；正常业务不依赖定时校准。

#### `user_generation_daily_usage`：用户每日图片用量

每个用户每天一行，用于原子控制每日请求图片额度。

| 字段 | 规则与职责 |
| --- | --- |
| `user_id` / `usage_date` | 用户 ID 与按 `Asia/Shanghai` 计算的业务自然日，共同唯一。 |
| `requested_image_count` | 当日已创建任务的请求图片数量总和。 |
| `updated_at` | 最近一次额度占用时间。 |

创建任务时在同一事务中按 `imageCount` 原子占用额度，条件为占用后不超过 12。`usage_date` 由服务端按 `Asia/Shanghai` 计算，不直接使用数据库 UTC 日期。任务创建成功即占用，后续成功、失败、部分成功或取消均不返还；同一幂等键重放不重复占用。

#### `user_consents`：用户规则同意记录

用于记录需要服务端强制校验的用户同意状态；首版使用 `GENERATION_THIRD_PARTY_PROCESSING` 类型。

| 字段 | 规则与职责 |
| --- | --- |
| `user_id` | 用户 ID，与 `consent_type` 共同唯一。 |
| `consent_type` | 同意类型；首版固定为 `GENERATION_THIRD_PARTY_PROCESSING`。 |
| `policy_version` | 用户已同意的规则版本。 |
| `policy_content_hash` | 用户确认时所展示不可变规则文案的摘要，用于追溯实际同意内容。 |
| `consented_at` | 用户确认时间。 |
| `updated_at` | 最近一次确认时间。 |

服务端必须维护当前有效规则版本及其不可变完整文案。前端先查询当前版本、完整文案和用户同意状态，确认时回传实际展示的 `policyVersion`；后端只接受当前有效版本。创建生成任务前，服务端必须校验用户已同意当前版本；未同意或版本落后时，拒绝创建任务且不占用活动槽位、每日额度或创建限流。规则文案应明确提示：为生成图片，用户本次正向提示词、负向提示词及生成所需会话上下文将发送至阿里云百炼；生成内容会被项目保存，并可在后续资产功能中删除；用户不应输入密码、身份证号、联系方式等敏感个人信息。

## 5. 普通文生图参数

当前创建任务接口只接受以下业务参数：

| 参数 | 规则 | 服务商映射 |
| --- | --- | --- |
| `prompt` | 必填，最多 1000 个字符。 | 正向提示词。 |
| `negativePrompt` | 可选，最多 500 个字符。 | `negative_prompt`。 |
| `aspectRatio` | 必填，默认 `1:1`。 | 映射为固定 `size`。 |
| `imageCount` | 必填；前端仅提供 `1`～`6` 的选择，服务端校验后映射至 `n`。 | `n`。 |

尺寸映射：

| `aspectRatio` | `size` |
| --- | --- |
| `1:1` | `2048*2048` |
| `16:9` | `2688*1536` |
| `9:16` | `1536*2688` |
| `4:3` | `2368*1728` |
| `3:4` | `1728*2368` |

普通模式固定 `prompt_extend = false`，以使已保存的原始提示词与实际普通模式输入一致。工作流模式未来如改写提示词，必须额外保存原始提示词、最终提示词与工作流版本。

每次调用以当前正向提示词为核心，并在模型长度预算内按由近到远的优先级保留历史正向提示词；最终发送顺序仍按时间从旧到新排列。后端使用与当前模型匹配的长度计算规则校验 `final_prompt`。若仅当前提示词已超限，返回“提示词过长”；若加上历史超限，则仅舍弃本次拼接中的最旧历史。

当前不向用户开放模型选择、随机种子、水印开关、任意宽高、自由工作流配置等参数。

## 6. OSS 存储与访问

生成图片必须在服务端完成转存：

1. 工作器从百炼返回临时 URL 后，先将 `provider_request_id`、临时 URL 列表和内部 `source_index` 写入任务快照并提交；服务商 URL 有效期为 24 小时。
2. 工作器再从临时 URL 下载图像。前端、SSE、普通日志和数据库其他字段均不得获得该 URL。
3. 仅请求 HTTPS 且命中服务商域名白名单的地址；禁止重定向，拒绝本地、内网和云元数据地址，防止 SSRF。
4. 流式下载并限制连接/读取超时与文件大小；校验 HTTP 成功、响应 `Content-Type`、文件魔数、真实图片格式、宽高与总像素数。首版仅接受 PNG，三者必须一致；不符合时视为该图片转存失败，避免将错误响应、超大文件或伪造内容误存为图片。
5. 上传至私有 OSS；对象键完全由服务端生成，不接受用户输入的路径片段。首版对象统一使用 `.png` 后缀。
6. 成功后写入 `generation_images`；全部图片转存成功时置为 `SUCCEEDED`，至少一张成功但未全部成功时置为 `PARTIALLY_SUCCEEDED`。只有该终态事务提交后，才通知前端展示图片。

对象键规范：

```text
users/{userId}/generations/{taskId}/{sourceIndex}.png
```

迭代 2 只保存私有 OSS 原图，客户端不直接获得永久 OSS 地址，任务详情接口按权限返回短期签名 URL。签名 URL 有效期建议为 5～10 分钟，且不得写入日志、数据库或 SSE 事件。返回签名 URL 的 API 响应使用 `Cache-Control: private, no-store`；OSS 图片使用不超过签名 URL 有效期的 `Cache-Control: private, max-age=...`。前端仅在 SPA 内存中缓存 URL 及过期时间，过期后重新查询任务详情，不写入本地存储，并设置 `Referrer-Policy: no-referrer`，错误监控和埋点不得上报完整 URL。迭代 4 用户发布图片时，基于私有原图生成公开展示副本或缩略图，保存到公开 Bucket 或公开前缀，并通过固定 CDN/OSS URL 供灵感社区浏览；撤销发布时移除或禁止该公开副本访问，不影响私有原图。

## 7. API 与认证契约

所有接口使用 `/api` 前缀、统一响应结构和 Bearer Access Token 认证。

| 功能 | 方法与路径 | 认证 | 说明 |
| --- | --- | --- | --- |
| 获取会话列表 | `GET /api/generation-sessions?cursor=&limit=20` | Bearer Token | 游标分页返回当前用户全部已提交内容的会话，包含存在活动任务和仅含终态任务的会话；按 `last_message_at DESC, id DESC` 排序。默认 `limit=20`，最大 50；超出 1～50 返回 `422` 参数校验失败。响应返回 `items`、不透明 `nextCursor` 和 `hasMore`，不提供页码或总页数。 |
| 获取会话详情 | `GET /api/generation-sessions/{sessionId}` | Bearer Token | 仅返回当前用户的会话基础信息，不全量返回消息和任务历史。 |
| 获取会话消息 | `GET /api/generation-sessions/{sessionId}/messages?before=&limit=30` | Bearer Token | 游标分页获取会话消息与关联任务结果。数据库与游标按 `sequence_no` 定位，响应中的消息按时间正序排列，便于前端向上加载更早历史。默认 `limit=30`，最大 100；超出 1～100 返回 `422` 参数校验失败。 |
| 修改会话标题 | `PATCH /api/generation-sessions/{sessionId}` | Bearer Token | 仅会话所有者可修改标题。 |
| 查询第三方数据处理规则 | `GET /api/users/me/consents/generation` | Bearer Token | 返回当前规则版本、完整展示文案、当前用户是否已同意该版本及同意时间。 |
| 确认第三方数据处理规则 | `POST /api/users/me/consents/generation` | Bearer Token | 请求必须携带前端实际展示的 `policyVersion`；后端只接受当前有效版本。首版仅支持确认，不提供撤回后继续生成的能力。 |
| 创建普通文生图任务 | `POST /api/generation-tasks` | Bearer Token | 要求 `Idempotency-Key` 请求头。`sessionId` 可选：传入时使用已有会话；不传时原子创建会话和任务，并在 `202 Accepted` 响应中返回 `sessionId` 与 `taskId`。 |
| 查询任务详情 | `GET /api/generation-tasks/{taskId}` | Bearer Token | 仅任务所属用户可查询；成功结果返回短期 OSS 签名 URL。 |
| 查询活动任务 | `GET /api/generation-tasks?active=true` | Bearer Token | 仅用于页面初始化与 SSE 重连后的状态对账，不承担会话侧边栏或历史展示；单用户最多 2 个活动任务，因此无需分页。 |
| 取消任务 | `POST /api/generation-tasks/{taskId}/cancel` | Bearer Token | 取消非终态任务；对取消中或已取消任务重复调用属于幂等成功，对其他终态返回 `409 TASK_ALREADY_FINISHED`。 |
| 建立事件流 | `GET /api/events` | Bearer Token | `text/event-stream`；每个生成页维持一条连接，同一用户最多 3 条。离开生成页时前端主动关闭连接，不影响后台任务执行。 |

SSE 事件示例：

```text
event: generation.task.updated
id: 928
data: {"sessionId":"456","taskId":"123","taskVersion":4,"status":"SUCCEEDED"}
```

事件不携带图片签名 URL 或完整任务信息。客户端收到事件后请求任务详情；首次连接和重连后查询活动任务完成状态对账。

浏览器侧不得把 Access Token 写入 SSE URL 查询参数。由于当前认证使用 Bearer Token，事件流应使用支持 `Authorization` 请求头的 Fetch 流式 SSE 实现。Access Token 即将到期时，客户端刷新 Token 并重新连接事件流。

幂等规则：前端每次用户主动提交时生成新的 UUID v4；同一次网络重试、超时重试或重复点击必须复用原键。幂等范围为当前用户与幂等键。同一用户使用相同 `Idempotency-Key` 且请求指纹一致时，返回已有任务的当前状态和首次创建的 `sessionId`，不重复消耗创建限流、活动槽位或每日额度；若指纹不同，返回 `409 IDEMPOTENCY_KEY_CONFLICT`，不得创建新任务。用户主动再次生成时，即使参数相同，也必须使用新键。

### 7.1 HTTP 状态与错误语义

| 场景 | HTTP 状态 | 稳定错误标识 |
| --- | --- | --- |
| 成功查询、修改或确认规则 | `200` | `0` |
| 成功接收生成任务 | `202` | `0` |
| 参数格式或范围错误 | `422` | `VALIDATION_ERROR` |
| 游标非法、过期或与查询条件不匹配 | `422` | `INVALID_CURSOR` |
| 未携带或 Access Token 无效 | `401` | `UNAUTHORIZED` |
| 会话、任务或图片不存在，或不属于当前用户 | `404` | `GENERATION_RESOURCE_NOT_FOUND` |
| 会话已有活动任务 | `409` | `SESSION_ACTIVE_TASK_EXISTS` |
| 用户两个活动槽位均被占用 | `409` | `USER_GENERATION_CONCURRENCY_LIMIT` |
| 相同幂等键对应不同请求指纹 | `409` | `IDEMPOTENCY_KEY_CONFLICT` |
| 未同意当前第三方规则 | `409` | `GENERATION_CONSENT_REQUIRED` |
| 任务已经进入不可取消终态 | `409` | `TASK_ALREADY_FINISHED` |
| 用户创建频率超限 | `429` | `USER_GENERATION_RATE_LIMIT` |
| IP 创建频率超限 | `429` | `IP_GENERATION_RATE_LIMIT` |
| 每日 12 张额度已用尽 | `429` | `DAILY_GENERATION_QUOTA_EXCEEDED` |
| 生成服务暂时不可用 | `503` | `GENERATION_SERVICE_UNAVAILABLE` |

所有接口继续使用统一 `code`、`message`、`data` 响应；错误标识对应的具体数字值在实现错误码表时统一分配。访问其他用户的私有生成资源与资源不存在统一返回 `404`，不得通过 `403` 或差异化消息泄露资源是否存在。`429` 响应必须包含 `Retry-After`；每日额度错误还应返回下一次北京时间重置时间。服务商原始错误、异常栈和内部执行字段不得返回客户端。异步任务的最终失败通过任务状态和安全失败分类表达，不改变创建请求已经被 `202` 接收的事实。

### 7.2 任务快照响应

`GET /api/generation-tasks/{taskId}` 用于返回任务当前的权威状态和可展示结果，供 SSE 事件后的状态对账、页面刷新或重连恢复，以及私有 OSS 签名 URL 过期后的重新获取。该接口不是数据库任务记录的完整映射。

响应 `data` 使用紧凑任务快照：

```json
{
  "taskId": "123",
  "sessionId": "456",
  "status": "PARTIALLY_SUCCEEDED",
  "taskVersion": 6,
  "requestedImageCount": 2,
  "completedImageCount": 1,
  "failedImageCount": 1,
  "cancelledImageCount": 0,
  "failureCode": "IMAGE_TRANSFER_PARTIAL_FAILURE",
  "failureMessage": "部分图片保存失败",
  "images": [
    {
      "imageId": "9001",
      "url": "https://example.com/signed-object-url",
      "urlExpiresAt": "2026-07-23T12:10:00Z",
      "width": 2688,
      "height": 1536
    }
  ],
  "createdAt": "2026-07-23T12:00:00Z",
  "completedAt": "2026-07-23T12:03:00Z"
}
```

规则：

- `status` 与 `taskVersion` 用于权威状态对账，客户端不得用更旧版本覆盖本地新状态。
- 四个数量字段用于展示完整成功、部分成功、失败和取消结果；只有成功转存的图片进入 `images`。
- 非终态任务的 `images` 为空，`failureCode`、`failureMessage` 和 `completedAt` 为 `null`。
- `failureCode` 与 `failureMessage` 只返回稳定、安全的用户可见分类，不暴露服务商原始错误。
- `urlExpiresAt` 供前端判断签名 URL 是否仍可复用；图片宽高用于图片加载前预留布局空间。
- 响应必须使用 `Cache-Control: private, no-store`。
- 普通任务快照不得返回 `taskType`、`executionMode`、模型标识、原始提示词、负向提示词、`finalPrompt`、`contextSnapshot`、服务商请求 ID、执行阶段、工作器 ID或临时 URL。原始提示词由会话消息接口恢复，内部追溯信息只保留在服务端。

## 8. 权限、可靠性与安全

- 用户只能创建、查询和取消自己的任务，不能读取其他用户的任务、结果或 OSS 签名 URL。
- 创建请求必须使用 UUID v4 幂等键；同一用户与同一幂等键只允许对应一个任务。相同键和相同规范化请求指纹返回已有任务当前状态，相同键但不同请求指纹必须拒绝。资源所有权查询必须同时使用资源 ID 与当前用户 ID，避免差异化响应泄露其他用户资源。
- Outbox 分发器以数据库条件更新或行锁领取事件；RabbitMQ 收到 Publisher Confirm 后才将执行命令事件标记为已发布。工作器仅在图片记录、任务终态和 SSE Outbox 事件提交后确认 RabbitMQ 消息；消息重投时根据任务终态或转存快照决定直接确认或继续转存，并通过任务状态条件更新和 `(task_id, source_index)` 唯一约束保证幂等。
- Outbox 投递 RabbitMQ 采用有限自动重试；最终失败时将 Outbox 事件标记为 `FAILED` 并将对应任务更新为 `FAILED`，不配置 DLQ、人工重放或运维处理。执行队列使用 Durable Quorum Queue，显式设置 `x-queue-type=quorum`；消费者并发固定为 6、每消费者 `prefetch=1`，不使用全局 QoS。RabbitMQ 消费失败达到上限时同样将任务更新为 `FAILED` 后确认该消息。工作器仅对确认尚未发送至服务商的临时故障有限重试；调用结果未知时直接失败，避免重复生成与计费。
- 应用定期扫描 `QUEUED` 超过 5 分钟的任务，并通过带前置状态的条件更新将其收敛为 `FAILED + QUEUE_TIMEOUT`，在同一事务中释放活动任务名额并写入 SSE Outbox 事件。队列不设置消息 TTL；超时任务遗留的消息被工作器再次投递时，工作器发现任务已终态后直接 ACK，不执行模型调用。
- 生成限制均为后端配置：每用户跨所有会话、页面和设备同时最多 2 个活动任务；每日按请求 `imageCount` 累计最多 12 张，按 `Asia/Shanghai` 自然日结算；全局工作器并发初始配置为 6。首版对新建任务启用应用内令牌桶限流：单用户每分钟最多 2 个、单 IP 每分钟最多 6 个；百炼全局出口初始每分钟最多 2 次。相同幂等键且请求指纹一致的重放不消耗限额。前端的单空白新对话规则仅用于体验引导，后端仍以活动槽位、每日用量和限流器为准。`imageCount` 仍仅允许模型支持的 1～6。
- 首版限流状态只存在单个应用实例内，应用重启后令牌桶重置。用户与 IP 两个创建令牌桶必须同时通过；客户端 IP 只能取自受控 Nginx 写入的可信转发头，无可信代理头时使用 TCP 对端地址，不得直接信任公网客户端提交的 `X-Forwarded-For`。迁移到 Redis 等共享限流器前不得水平扩容或拆分独立工作器。扩容后用户/IP创建限流、百炼出口限流和 SSE 事件分发都必须使用跨实例共享能力。
- 百炼模型调用只在后台工作器中进行，前端创建任务后立即收到 `202 Accepted`，后续状态通过 REST 查询与 SSE 获取。工作器连接百炼超过 5 秒仍未建立连接时，视为确认未发出请求，可按有限次数重试；已建立连接但等待服务商结果超过 300 秒时，结果状态未知，任务进入 `FAILED + PROVIDER_RESULT_UNKNOWN`，不得自动重试。HTTP 客户端禁止自动重定向和自动重试，必须校验证书与主机名；最大并发连接数为 6，并且不得记录完整请求体、响应临时 URL 或 API Key。
- 用户首次生成前必须读取并明确同意当前版本的第三方数据处理规则；后端提供当前版本、完整文案和同意状态查询，确认请求携带版本且只接受当前有效版本。创建任务前校验该同意记录，未同意或版本过期时拒绝创建，且不写入会话、任务、消息、用量或 Outbox 数据，也不消耗创建限流。规则需明确告知：生成所需的提示词和会话上下文会发送至阿里云百炼，生成内容会被保存并可在后续资产功能中删除，同时提示用户不要输入敏感个人信息。
- 百炼 API Key 和 OSS 凭证在本地仅允许写入未提交的 `application-local.yaml`，在生产环境仅通过环境变量注入。服务商 API Key、OSS 密钥、临时图片 URL、签名 URL、完整提示词、上下文快照和原始服务商错误不得写入客户端响应或普通业务日志。
- 迭代 2 的私有生成仅依赖服务商内容安全能力；服务商内容安全拒绝应映射为稳定、面向用户的失败分类，不得暴露服务商原始错误。当前不做本地关键词拦截或独立内容审核。后续如有需要，再为发布记录设计独立审核状态，并使公开灵感流仅展示审核通过的作品。注册、登录接口限流留待后续认证安全迭代；生成创建接口已按本模块规则启用用户/IP 限流，用户级并发任务数以数据库状态为准。
- Nginx / 网关必须关闭 SSE 响应缓冲并设置足够长的读取超时。
- SSE 定期发送心跳；Spring MVC 通过 `SseEmitter` 支持 SSE，断线恢复依赖任务状态对账而不是假设事件绝不丢失。参见[Spring SSE 文档](https://docs.spring.io/spring-framework/reference/web/webmvc/mvc-ann-async.html)。
- 迭代 2 的提示词、上下文和图片均为私有生成数据。迭代 4 发布时，用户必须明确确认：将公开该图片所属会话中截至本次生成所使用的全部用户正向提示词历史，以及本次负向提示词；不得公开 `context_snapshot`、服务商参数、工作流配置或系统提示词。
- 迭代 2 默认长期保存图片、会话、提示词、上下文快照和任务记录，不提供删除入口；迭代 3 必须交付会话或生成资产删除能力，不得继续无限延期。存在活动任务的会话不得删除。删除时应清理私有图片、消息、上下文快照和任务业务数据；OSS 对象采用异步删除并记录清理状态，直到对象清理完成。已发布图片必须先撤销发布或由删除流程自动撤销公开副本，具体语义在迭代 4 确认。
- 用户生成图片、会话、提示词、上下文快照和任务记录首版不自动过期。生产数据库必须具备加密备份与可验证恢复能力，备份保留 7 天；具体实施可使用 RDS 自动备份，或自建 MySQL 的定时全量备份加二进制日志或等效方案。正式上线前必须完成至少一次恢复演练。备份副本的自然淘汰不改变应用库中的业务保留语义。

### 8.1 基础技术监控

首版使用 Micrometer 与 Spring Boot Actuator 记录和暴露技术指标，不要求在本迭代搭建 Prometheus、Grafana 或完整告警平台。监控接口必须限制为内网、运维网络或受认证保护的访问，不能直接暴露至公网。

至少记录以下聚合指标：

- `QUEUED`、`RUNNING`、`FAILED` 与 `PROVIDER_RESULT_UNKNOWN` 的任务数量，以及最久排队时间；
- Outbox `PENDING`、`FAILED` 数量及最久等待时间；
- 百炼调用耗时、调用次数和按安全失败分类聚合的失败次数；
- OSS 下载、校验和上传失败次数；
- 并发上限与每日额度拒绝次数。

指标标签不得使用 `userId`、`taskId`、会话 ID、提示词或图片 URL，避免高基数和敏感数据泄露。排障日志可记录 `taskId`、会话 ID、服务商请求 ID 和安全失败分类，但不得记录提示词、URL 或密钥。业务/产品行为埋点不属于本模块首版范围。

## 9. 实施顺序与验收

实施顺序：

1. 确认会话、消息、任务、图片、用户生成门闩、活动槽位和每日用量之间的数据关系，以及 MySQL 表、索引、Flyway 迁移与 Outbox 事件模型。
2. 配置私有 OSS，完成安全下载、文件校验、上传、短期签名 URL 与最小权限访问控制。
3. 配置 RabbitMQ 和任务/事件分发基础设施；首版单实例不部署 Redis Pub/Sub，多实例部署时再补充。
4. 实现 `DIRECT` 执行器与百炼 Qwen-Image 调用。
5. 实现会话活动位、用户生成门闩与两个活动槽位约束、任务状态机、执行租约、UUID v4 幂等、有限安全重试、取消竞态、失败分类、任务级图片结果汇总和服务商结果转存恢复。
6. 实现会话历史恢复、上下文快照保存，以及 SSE 连接、心跳、事件分发与断线状态对账。
7. 补充单元测试、MySQL 集成测试、RabbitMQ/OSS 集成测试、安全下载测试、工作器崩溃恢复测试、并发名额与每日额度原子性测试、基础监控指标测试和真实百炼联调验收；首版不要求 Redis 集成测试。

验收标准：

- 同一幂等键不会创建或执行重复任务。
- 用户可创建 1～6 张的普通文生图任务，并仅能读取自己的结果。
- 用户进入生成页可使用空白创作框；首次提交会创建并返回会话，已提交会话可恢复、改名并按最近活动时间显示在不限数量的侧栏列表中。空白创作框成功提交后才可开启下一个新会话。
- 同一会话不会同时存在两个活动任务；同一用户跨所有会话、页面和设备最多同时存在两个活动任务，并由 `(user_id, activity_slot)` 唯一索引提供数据库级保护。关闭会话页面只会断开 SSE，不影响已创建任务的执行；下次加载会话和任务详情可恢复请求与结果。
- 并发创建请求不会突破单用户 2 个活动任务或每日 12 张请求图片额度；任务创建后立即占用每日额度，幂等重放不重复占用，任务终态仅释放活动槽位而不返还每日额度。
- 用户和 IP 创建限流分别为每分钟 2 次和 6 次；百炼出口初始为全局每分钟 2 次。明确的百炼 `429/Throttling` 可退避重新排队，但总排队时间不超过 5 分钟。
- 用户重新打开页面后可恢复会话历史、任务状态和已生成图片；SSE 事件可定位到对应会话与任务。
- 每次生成任务均保留其上下文快照与最终调用提示词，满足模型生成上下文的可追溯性。
- 所有请求图片均成功转存私有 OSS 后，任务进入 `SUCCEEDED`；至少一张成功转存但未全部成功时，任务进入 `PARTIALLY_SUCCEEDED`；没有任何图片成功转存时进入 `FAILED`。客户端只展示成功图片及任务级成功、失败、取消数量，不展示失败图片槽位。
- 任务状态变化可通过 SSE 通知在线客户端；断线重连后客户端能通过活动任务查询恢复正确状态。
- 可重试与不可重试失败得到不同处理；最终失败对用户返回安全的失败说明。
- 调用结果未知时不会自动重试，避免重复生成与重复计费。
- 工作器领取任务时写入执行租约、工作器标识和内部执行阶段。租约到期后：确认尚未发送服务商请求的任务可重新排队；服务商请求结果未知的任务进入 `FAILED + PROVIDER_RESULT_UNKNOWN`，不得再次调用模型。
- 服务商已返回图片临时结果时，先保存仅供转存恢复使用的 `request_id + source_index + 临时 URL` 快照，再执行 OSS 下载与上传；工作器崩溃后只允许恢复转存，不得再次调用模型。全部结果转存完成后立即清除该快照；URL 超过 24 小时仍无法恢复时，保留已成功图片并将未完成部分收敛为失败。
- 同一幂等键且参数相同不会创建重复任务；相同幂等键但参数不同返回冲突。
- 任务、图片、Outbox 事件和任务状态变更具备可追溯性；签名 URL、密钥和完整提示词不出现在普通日志或 SSE 事件中。
- Outbox、RabbitMQ 和 SSE 事件仅传递最小标识；工作器根据 `taskId` 从 MySQL 读取任务详情，不在消息载荷中复制提示词、上下文或 URL。
- 基础技术指标可在受保护的 Actuator 监控端点读取，且不包含用户、任务或提示词等高基数/敏感标签。

16. **OSS 转存与删除状态**：确认 Bucket、地域、加密、生命周期、访问域名、安全下载限制、对象转存状态、异步删除状态、签名 URL 有效期及数据库与 OSS 不一致时的恢复方式。
