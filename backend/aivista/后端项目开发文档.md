## 2. 一期目标

一期后端围绕“普通用户生成、管理、发布并互动 AI 图片”构建最小完整闭环：

```text
用户提交文生图请求
        ↓
生成结果沉淀为个人历史资产
        ↓
用户收藏图片，或收藏并发布
        ↓
公开图片进入灵感社区
        ↓
其他用户浏览提示词、点赞图片、关注作者
```

一期仅考虑普通用户，不包含付费订阅、权益、计费、订单或支付能力。推荐流（热门推荐、仅关注的人）不属于一期首版范围，后续单独规划。

一期客户端优先完成 Web 端。React Native App 在 Web 端完成后开发。后端认证与数据模型应保留 App 接入空间，但 App 的令牌传输、安全存储、应用完整性校验和设备管理不纳入当前 Web 端实现范围。

## 3. 已确认业务边界

| 主题 | 已确认规则 |
| --- | --- |
| 用户类型 | 一期仅支持普通用户。 |
| 单次文生图 | 当前固定使用 `qwen-image-2.0-pro`，用户可选择生成 1 到 6 张图片；服务端按模型能力校验数量。 |
| 局部重绘 | 局部重绘属于生成能力的一部分；输入、结果与原图的关系及生成数量在对应迭代文档中确定。 |
| 生成历史 | 所有成功生成的图片均保留在用户历史生成资产中，不以是否收藏作为保留条件。 |
| 收藏 | 收藏是用户对生成图片的标记；用户可在个人资产中筛选已收藏图片。 |
| 发布 | 用户可将自己的生成图片发布至灵感社区；发布时基于私有原图生成公开展示副本或缩略图，并使用固定公开 URL；撤销发布时移除或禁止公开副本访问，不影响私有原图和个人资产。 |
| 提示词公开 | 一期中，公开发布图片的历史提示词完整对外展示。 |
| 灵感流 | 首版提供全站公开内容浏览；“仅关注的人”“热门推荐”等能力留待后续迭代。 |
| 互动 | 一期包含图片点赞/取消点赞，以及用户关注/取消关注。用户可以点赞自己的或他人的已发布图片。 |

## 4. 迭代原则

1. 按可独立验收的最小闭环推进，每个迭代只解决一个明确目标。
2. 先建立私有生成资产，再开放公共发布与社交互动，保证内容来源和权限边界清晰。
3. 详细文档先行；开始任一迭代编码前，先补充该迭代文档。
4. 代码、测试和文档同步更新；接口契约或数据结构变化必须反映到对应文档。
5. 保持向后演进空间；一期不实现付费和推荐，但不把规则设计成阻碍未来扩展的硬限制。

### 5.2 迭代 1：用户模块

状态：已完成。

已完成：

- 用户注册、登录、刷新、退出。
- 当前用户资料查询和修改。
- JWT Access Token。

具体规则和实现见 [用户模块.md](用户模块.md)。

### 5.3 迭代 2：文生图与局部重绘任务

状态：详细设计中。

目标：

- 先实现普通文生图任务。
- 支持持久化多会话、会话历史恢复与应用层模型上下文记忆；同一会话同一时间仅允许一个活动任务，多个会话可独立排队或生成。
- 采用可靠异步任务、OSS 存储和 SSE 状态推送架构；首版按单个后端实例部署，不启用 Redis Pub/Sub，多实例部署时再启用跨实例事件分发。
- 后续在同一任务模型中扩展局部重绘和用户主动选择的工作流模式。

开始编码前必须新增详细文档，至少明确任务状态机、生成请求模型、结果模型、失败重试策略、接口契约和权限规则。

## 6. 后续迭代方向

以下方向不纳入当前一期首版排期：

- 普通用户之外的付费用户、权益和计费体系。
- 灵感社区的“仅关注的人”、热门排序和个性化推荐。
- 提示词可见范围、复制能力、脱敏策略等内容控制能力。
- 更丰富的生成参数、模型管理、任务调度和生成结果编辑能力。
- React Native App 接入，以及对应的令牌传输、安全存储、应用完整性校验和设备管理。
