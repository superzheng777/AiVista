# AiVista 后端项目开发文档

> 文档状态：迭代计划基线  
> 适用范围：AiVista 后端项目（普通用户与 AI 文生图社区第一期）  
> 维护规则：**每次开发必须同步更新代码和相关文档。**功能范围、业务规则、接口契约或数据结构发生变化时，应在同一次提交中更新对应的迭代设计文档；本文件仅在迭代计划或已确认业务边界调整时更新。

## 1. 文档目的

本文档记录 AiVista 后端第一期的迭代目标、已确认的业务边界和迭代顺序，作为后续开发与验收的共同基线。

本文档不展开具体技术选型、接口定义、数据库表结构、字段设计或实现细节。进入某个迭代前，应为该迭代补充独立的详细设计文档，明确其业务规则、接口、数据模型、权限、异常场景及验收标准后再开发。

## 2. 一期目标与范围

一期后端围绕“普通用户生成、管理、发布并互动 AI 图片”构建最小完整闭环：

```text
用户提交文生图请求
        ↓
生成结果沉淀为个人历史资产
        ↓
用户收藏图片，或收藏并发布
        ↓
公开图片进入灵感社区
        ↓
其他用户浏览提示词、点赞图片、关注作者
```

本期仅考虑普通用户，不包含付费订阅、权益、计费、订单或支付能力。推荐流（热门推荐、仅关注的人）不属于一期首版范围，后续单独规划。

一期客户端优先完成 Web 端；React Native App 在 Web 端完成后开发。后端认证与数据模型应保留 App 接入空间，但 App 的令牌传输、安全存储、应用完整性校验和设备管理不纳入当前 Web 端实现范围，须在 App 开发前单独确认。

## 3. 已确认业务约束

| 主题 | 已确认规则 |
| --- | --- |
| 用户类型 | 一期仅支持普通用户。 |
| 单次文生图 | 一次生成任务可产生 1～2 张图片。 |
| 局部重绘 | 局部重绘属于生成能力的一部分，一次可产生 1～4 张图片。其输入、结果与原始图片的关联方式在对应迭代详细设计中确定。 |
| 生成历史 | 所有成功生成的图片均保留在用户历史生成资产中，不以是否收藏作为保留条件。 |
| 收藏 | 收藏是用户对生成图片的标记；用户可在个人资产中筛选已收藏图片。 |
| 发布 | 用户可将自己的生成图片发布至灵感社区；发布后允许撤销发布，撤销不应影响原始生成历史和个人资产。 |
| 提示词公开 | 一期中，公开发布图片的历史提示词完整对外展示。 |
| 灵感流 | 首版提供全站公开内容浏览；“仅关注的人”“热门推荐”等能力留待后续迭代。 |
| 互动 | 一期包含图片点赞/取消点赞，以及用户关注/取消关注。用户可以点赞自己的或他人的已发布图片。 |

## 4. 迭代原则

1. 按可独立验收的最小闭环推进：每个迭代只解决一个明确目标，避免提前实现后续能力。
2. 先建立私有生成资产，再开放公共发布与社交互动，保证内容来源和权限边界清晰。
3. 详细设计先行：开始任一迭代编码前，先补充该迭代设计文档；完成后同步更新实现状态、验收结果和必要的接口说明。
4. 保持向后演进空间：一期不实现付费和推荐，但不将一期规则设计为阻碍未来扩展的硬性限制。

## 5. 迭代路线

### 迭代 0：工程基础（已完成）

已建立后端工程基础：Spring Boot 应用可启动，提供 `/api/health` 健康检查、统一响应与全局异常处理、跨域配置，以及 OpenAPI/Knife4j 接口文档。

工程可通过 `mvn test` 验证；Maven Wrapper（`mvnw.cmd`）也已验证可用，后续在 `backend/aivista` 目录优先执行 `.\mvnw.cmd test`。本迭代不涉及具体业务 API、数据库表或业务数据模型，相关设计在对应模块迭代中完成。

### 迭代 1：用户模块

**目标**：建立普通用户身份与个人资料能力。

**交付范围**：用户注册、登录或等价的身份认证能力；查询和修改个人资料；资料包含用户名、头像和个人简介。

**完成标志**：已认证用户能够维护并读取自己的个人资料。

### 迭代 2：文生图与局部重绘任务

**目标**：接收用户的图片生成请求，并追踪生成过程与结果。

**交付范围**：文生图任务创建与状态流转；一次任务产出 1～2 张图片；局部重绘任务创建与状态流转，一次产出 1～4 张图片；保留生成任务与结果之间的可追溯关系。

**完成标志**：用户可以提交文生图或局部重绘请求，并获取任务状态及生成结果。

### 迭代 3：个人生成历史与资产

**目标**：让用户可查询并管理自己的全部生成结果。

**交付范围**：保存生成图片、历史提示词及必要的生成上下文；分页查询个人历史生成图片；查看图片及其生成详情。

**完成标志**：用户可在个人资产中查阅每一张历史生成图片及其对应生成信息。

### 迭代 4：收藏与发布

**目标**：支持用户标记满意作品，并将个人资产转化为公开内容。

**交付范围**：收藏/取消收藏自己的生成图片；按收藏状态筛选个人资产；发布自己的生成图片；撤销已发布内容。

**完成标志**：生成历史始终保留，用户可独立管理收藏状态和公开发布状态。

### 迭代 5：灵感社区浏览

**目标**：建立全站公开发布内容的浏览与详情能力。

**交付范围**：按时间分页浏览公开发布图片；查看发布详情、发布者信息和完整历史提示词。

**完成标志**：任意用户均可浏览公开内容，并从详情中追溯该图片的历史提示词。

### 迭代 6：图片点赞

**目标**：支持围绕公开图片的基础互动与个人点赞足迹。

**交付范围**：点赞/取消点赞公开图片；展示点赞数量和当前用户点赞状态；查询当前用户已点赞的图片。

**完成标志**：用户可对自己的或他人的公开图片进行点赞，且点赞信息在列表、详情和个人记录中保持一致。

### 迭代 7：用户关注

**目标**：建立用户之间的关注关系。

**交付范围**：关注/取消关注用户；展示粉丝数、关注数和当前用户的关注状态；查询粉丝列表与关注列表。

**完成标志**：用户关系可被正确建立、解除、查询和统计。

### 迭代 8：一期闭环完善

**目标**：完善个人主页与一期的业务可靠性，形成可交付的第一版后端能力。

**交付范围**：聚合查询个人已发布图片和已点赞图片；补齐权限校验、数据一致性处理、必要的接口测试和运行保障。

**完成标志**：生成、资产、收藏、发布、浏览、点赞、关注和个人主页查询构成完整且可验证的一期闭环。

## 6. 后续迭代方向（不纳入当前排期）

- 普通用户之外的付费用户、权益和计费体系。
- 灵感社区的“仅关注的人”、热门排序和个性化推荐。
- 提示词可见范围、复制能力、脱敏策略等内容控制能力。
- 更丰富的生成参数、模型管理、任务调度和生成结果编辑能力。
- React Native App 接入，以及对应的令牌传输、安全存储、应用完整性校验和设备管理。

## 7. 详细设计文档约定

每个迭代启动前，在本项目文档目录（后续建立）中新增对应详细设计文档。至少应包含：

- 迭代目标、范围与非目标；
- 业务流程与状态规则；
- 数据模型与数据库变更；
- API 契约、鉴权与权限规则；
- 异常与边界场景；
- 验收标准、测试范围及文档同步项。

详细设计经确认后再实施代码；若实施中需求发生变化，应先更新详细设计，再同步修改代码和测试。
