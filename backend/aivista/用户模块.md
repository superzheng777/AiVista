## 1. 模块边界

本模块负责普通用户身份与个人资料。社交关系、图片资产、发布、点赞、会员、角色与账号注销均在后续独立设计。

## 2. 数据存储

- 数据库：`aivista`
- 用户表：`users`
- 初始化与迁移脚本：[sql/001_create_users.sql](sql/001_create_users.sql)
- 规范：InnoDB、utf8mb4、`utf8mb4_unicode_ci`；数据库字段使用下划线命名，Java 字段使用驼峰命名。

| 字段 | 数据库约束 | 开发规则 |
| --- | --- | --- |
| `id` | `BIGINT UNSIGNED`、`NOT NULL`、自增主键。 | 作为用户内部唯一标识。 |
| `login_name` | `VARCHAR(32)`、`NOT NULL`；唯一索引 `uk_users_login_name`；使用 `utf8mb4_unicode_ci` 比较，大小写不敏感。 | 登录账号，不对外展示；注册后不可修改。账号允许的字符范围与非空校验在注册接口中确定。 |
| `password_hash` | `VARCHAR(255)`、`NOT NULL`。 | 仅保存密码哈希，永不返回客户端；只有注册、修改密码等认证流程可写入。哈希算法与参数在认证设计中确定。 |
| `nickname` | `VARCHAR(32)`、`NOT NULL`。 | 公开展示昵称；允许重名。 |
| `avatar_url` | `VARCHAR(512)`、可空，默认 `NULL`。 | 仅保存头像地址；地址由通用媒体上传能力提供，更新时应校验文件归属。 |
| `bio` | `VARCHAR(500)`、可空，默认 `NULL`。 | 用户个人简介。 |
| `created_at` | `DATETIME`、`NOT NULL`、默认当前时间。 | 创建后不修改。 |
| `updated_at` | `DATETIME`、`NOT NULL`、默认当前时间；记录更新时自动更新。 | 不由客户端直接传入。 |

## 3. 索引与规则边界

| 主题 | 规范 |
| --- | --- |
| 账号唯一性 | `uk_users_login_name` 保证登录账号全局唯一；`Alice` 与 `alice` 不能同时注册。 |
| 字符比较 | 当前排序规则除大小写外，也可能将部分语言变体字符视为相等；注册接口应通过字符白名单限制账号字符范围，避免歧义。 |
| 应用层规则 | `login_name` 不可修改、`password_hash` 不可由资料接口写入、头像文件归属校验均为应用层规则，不是数据库约束。 |
| 非本期字段 | 表中不包含账号状态、角色、社交、资产、会员或注销字段；相关能力出现明确业务需求时再独立设计。 |

## 4. 开发约束

| 主题 | 规范 |
| --- | --- |
| 登录方式 | 注册与登录使用 `login_name + 密码`；其他登录方式后续通过独立身份绑定表扩展。 |
| 可修改资料 | 用户资料可修改 `nickname`、`avatar_url`、`bio`，不得修改 `login_name` 或直接写入 `password_hash`。 |
| 文件存储 | 头像、AI 生成图和发布图片共享通用文件存储能力，但按用途设置独立路径与访问策略。 |
| 接口规范 | 接口响应遵循项目统一 `code`、`message`、`data` 结构；认证、鉴权、限流和输入校验必须在接口设计中明确。 |

## 5. 认证令牌设计（已确认）

| 项目 | 规则 |
| --- | --- |
| Access Token 格式 | 使用 JWT，仅包含用户 ID、令牌类型、签发时间、过期时间和唯一标识；不得包含用户资料或密码信息。 |
| Access Token 使用方式 | 有效期 15 分钟；前端仅保存在内存中；通过 `Authorization: Bearer <token>` 访问受保护接口。 |
| Refresh Token 格式 | 使用密码学安全随机字符串，不使用 JWT；有效期 14 天。 |
| Refresh Token 使用方式 | 保存在 `HttpOnly Cookie` 中，仅用于刷新令牌和退出登录。 |
| 服务端存储与校验 | 服务端不保存 Refresh Token 明文，仅保存哈希和登录会话状态。Access Token 以签名与有效期校验为主；Refresh Token 必须校验会话是否存在、过期或已撤销。 |
| 配置管理 | Access Token 和 Refresh Token 有效期、Cookie 属性及 JWT 签名密钥必须统一配置管理，不得在业务代码中硬编码。 |

## 6. 会话、轮换与退出规则（已确认）

| 项目 | 规则 |
| --- | --- |
| Refresh Token 轮换 | 每次成功刷新后必须签发新令牌，并以新哈希替换当前认证会话中的旧哈希；旧令牌立即失效。 |
| 会话最终期限 | Refresh Token 自首次登录起有效 14 天；轮换不延长认证会话的最终过期时间。 |
| Cookie 属性 | 使用 `HttpOnly`；生产环境 `Secure=true`；`SameSite=Lax`；`Path=/api/auth`；不设置 `Domain`。首次登录的 `Max-Age` 为 14 天，轮换后使用认证会话的剩余有效期。 |
| 部署前提 | 生产环境由同一公开域名下的 Nginx 提供前端并代理 `/api` 请求；若改为跨站部署，必须重新评估 Cookie 策略。 |
| 多端登录 | 允许同一用户在多个设备或浏览器同时登录；每次登录创建独立认证会话。 |
| 普通退出登录 | 仅撤销当前认证会话，使当前 Refresh Token 立即失效，不影响其他设备。 |
| 全部设备退出 | 当前不实现；后续可增加接口，一次撤销该用户全部认证会话。 |
| Access Token 退出语义 | 不建立 Access Token 黑名单；退出后已签发的 Access Token 仍可使用，最迟在其 15 分钟有效期结束后失效。 |

## 7. Refresh Token 重用处理（已确认）

| 项目 | 规则 |
| --- | --- |
| 会话隔离 | 每个设备或浏览器登录创建独立认证会话与独立 Refresh Token。 |
| 会话记录 | 认证会话保存当前 Refresh Token 哈希、上一代 Refresh Token 哈希及上一代令牌宽限截止时间。 |
| 宽限期 | 宽限期为 5 秒，必须配置化，不得硬编码。上一代 Token 在此期间再次出现时，仅拒绝请求，不撤销会话。 |
| 重用检测 | 上一代 Token 超过宽限期再次出现时，撤销其所属的当前认证会话，撤销原因记为 `REFRESH_TOKEN_REUSE`。 |
| 无法匹配 | 无法匹配当前或上一代 Token 时，返回 `401`，不影响其他认证会话。 |
| 前端协作 | 前端必须实现单飞刷新；服务端宽限期仅用于并发、多标签页和网络边界的容错。 |
| 历史范围 | 一期仅保存当前与上一代 Token，不保存完整历史。上一代 Token 被下一次轮换替换后，不再具备重用检测能力。 |

> 宽限期内的旧 Token 请求会被拒绝，但不会得到新 Token；其作用是避免误撤销会话，而不是保证网络丢包后的刷新请求一定成功。
