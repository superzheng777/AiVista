# AiVista 用户模块

> 对应迭代：迭代 1——用户模块  
> 当前状态：已完成。Web 端用户认证与个人资料后端功能已实现，并已通过 OpenAPI/Knife4j 连接真实 MySQL 完成人工验收；自动化 Testcontainers 验收可在具备 Docker 的环境补充执行。  
> 文档目的：说明用户模块的业务边界、数据模型、认证机制、接口契约、实现位置和验收方式，使后续 AI 或开发者能快速接手维护。

## 1. 模块范围

用户模块负责普通用户身份、Web 端认证会话和当前用户个人资料。

当前包含：

- 用户注册、登录、刷新访问令牌、退出登录。
- 当前用户资料查询和修改。
- Access Token JWT 校验。
- Refresh Token HttpOnly Cookie、服务端哈希存储、轮换和会话清理。
- OpenAPI/Knife4j 接口说明与请求样例。

当前不包含：

- 角色、权限体系、管理员能力。
- 社交关系、关注、点赞、图片资产、发布、收藏。
- 账号注销、封禁、会员、付费订阅。
- Refresh Token 重用检测、上一代令牌宽限期、刷新幂等缓存。
- 接口限流。
- App 端令牌存储、设备管理和应用完整性校验。

## 2. 核心设计

认证采用“短期 Access Token + 长期 Refresh Token”的 Web 端方案：

- Access Token 使用 JWT，有效期 15 分钟，仅由前端保存在内存中。
- 业务请求通过 `Authorization: Bearer <accessToken>` 访问受保护接口。
- Refresh Token 使用密码学安全随机字符串，不使用 JWT，有效期 14 天。
- Web 端 Refresh Token 只保存在 `HttpOnly Cookie` 中，前端 JavaScript 不读取。
- 服务端只保存 Refresh Token 的 SHA-256 哈希，不保存明文。
- 每次刷新成功都会轮换 Refresh Token，旧 Refresh Token 立即失效。
- Access Token 校验只检查 JWT 签名、类型和有效期，不查询 `users` 或 `auth_sessions`。
- 退出登录删除当前 Refresh Token 对应会话并清 Cookie，不建立 Access Token 黑名单。

前端刷新策略：

- 业务请求因 Access Token 缺失或过期返回 `40100` 后，由前端响应拦截器触发刷新。
- 前端使用 Single Flight 合并并发刷新，同一时刻只发起一次 `/api/auth/refresh`。
- 刷新成功后更新内存中的 Access Token，并重放等待中的业务请求。
- 刷新返回 `40102` 表示认证会话失效，前端清空登录状态并跳转登录页。
- 网络异常不直接判定登录失效，前端保留当前状态并提示网络异常。

### 3.1 `users`

| 字段 | 规则 |
| --- | --- |
| `id` | `BIGINT UNSIGNED` 自增主键，用户内部唯一标识。 |
| `login_name` | 登录账号，`VARCHAR(32)`，唯一索引 `uk_users_login_name`，大小写不敏感；注册后不可修改。 |
| `password_hash` | Argon2id 密码哈希，永不返回客户端。 |
| `nickname` | 昵称，`VARCHAR(32)`，允许重名。 |
| `avatar_url` | 头像地址，`VARCHAR(512)`，可空；媒体模块落地前不接受新的非空外部地址。 |
| `bio` | 个人简介，`VARCHAR(500)`，可空。 |
| `created_at` | 创建时间，由数据库默认值维护。 |
| `updated_at` | 更新时间，由数据库自动更新。 |

账号规则：

- `loginName` 入库前去除首尾空白。
- 长度 4 到 32。
- 只能包含 ASCII 字母、数字和下划线。
- 必须以字母开头。
- 不自动改变大小写。
- 唯一性最终由数据库唯一索引保证。

密码规则：

- 按 Unicode 码点计算长度，8 到 64 个字符。
- 至少包含一个字母和一个数字。
- 可以包含空格和特殊字符。
- 不得仅由空白字符组成。
- 服务端不得去除空白、改变大小写、记录日志或回显。

昵称规则：

- 入库前去除首尾空白。
- 按 Unicode 码点计算长度，1 到 32 个字符。
- 允许重名和内部空格。
- 禁止控制字符。

### 3.2 `auth_sessions`

一条记录表示一个浏览器或设备上的独立登录会话。

| 字段 | 规则 |
| --- | --- |
| `id` | `BIGINT UNSIGNED` 自增主键。 |
| `user_id` | 所属用户 ID，外键关联 `users.id`。 |
| `client_type` | 当前固定为 `WEB`，预留 `APP`。 |
| `refresh_token_hash` | 当前 Refresh Token 的 SHA-256 哈希，`BINARY(32)`，唯一索引。 |
| `expires_at` | 会话最终过期时间，登录时确定，刷新不延长。 |
| `created_at` | 会话创建时间。 |
| `updated_at` | 最近一次轮换更新时间。 |

会话规则：

- 每次登录创建独立会话。
- 同一用户允许多浏览器、多设备同时登录。
- 刷新时按 `sessionId + oldHash + expiresAt > now` 原子替换为新哈希，更新行数必须为 1。
- 主动退出直接删除当前 Refresh Token 哈希对应会话。
- 自然过期会话失去认证能力，保留 7 天后由清理任务分批删除。

## 4. 接口契约

### 4.0 资源 ID 表示

数据库中的主键仍使用 `BIGINT UNSIGNED`；所有对外 HTTP 响应中的资源 ID 均序列化为字符串。当前 `UserProfileResponse.id` 为字符串，后续图片、生成任务等资源遵循相同规则，以避免 JavaScript 对大整数的精度丢失。

所有接口使用 `/api` 上下文路径。成功和失败响应均采用统一结构：

```json
{
  "code": 0,
  "message": "ok",
  "data": {}
}
```

### 4.1 接口列表

| 功能 | 方法与路径 | 认证要求 | 成功响应 |
| --- | --- | --- | --- |
| 注册 | `POST /api/auth/register` | 无 | `201`，返回 `UserProfileResponse`，不自动登录。 |
| 登录 | `POST /api/auth/login` | 无 | `200`，返回 `LoginResponse`，并写入 Refresh Token Cookie。 |
| 刷新 | `POST /api/auth/refresh` | `refresh_token` Cookie | `200`，返回 `TokenResponse`，并轮换 Refresh Token Cookie。 |
| 退出 | `POST /api/auth/logout` | Cookie 可缺失；不要求 Access Token | `200`，清除 Cookie，`data = null`。 |
| 查询个人资料 | `GET /api/users/me` | Bearer Access Token | `200`，返回 `UserProfileResponse`。 |
| 修改个人资料 | `PUT /api/users/me` | Bearer Access Token | `200`，返回更新后的 `UserProfileResponse`。 |

### 4.2 DTO

| DTO | 字段 | 说明 |
| --- | --- | --- |
| `RegisterRequest` | `loginName`、`password`、`nickname` | 注册请求，不包含头像、简介或认证字段。 |
| `LoginRequest` | `loginName`、`password` | 登录请求，账号去除首尾空白，密码保持原样。 |
| `TokenResponse` | `accessToken`、`tokenType`、`expiresIn` | 刷新接口响应，`tokenType = Bearer`，`expiresIn` 单位为秒。 |
| `LoginResponse` | `accessToken`、`tokenType`、`expiresIn`、`user` | 登录接口响应，比 `TokenResponse` 多当前用户资料。 |
| `UpdateProfileRequest` | `nickname`、`avatarUrl`、`bio` | 完整提交可修改资料；`avatarUrl` 和 `bio` 可为 `null`。 |
| `UserProfileResponse` | `id`（字符串）、`loginName`、`nickname`、`avatarUrl`、`bio`、`createdAt`、`updatedAt` | 当前用户资料响应，不包含密码或认证会话。 |

### 4.3 调用样例

注册：

```http
POST /api/auth/register
Content-Type: application/json

{
  "loginName": "alice_2026",
  "password": "Aivista2026",
  "nickname": "Alice"
}
```

登录：

```http
POST /api/auth/login
Content-Type: application/json

{
  "loginName": "alice_2026",
  "password": "Aivista2026"
}
```

刷新：

```http
POST /api/auth/refresh
Cookie: refresh_token=<HttpOnly Cookie，由浏览器自动携带>
```

退出：

```http
POST /api/auth/logout
Cookie: refresh_token=<HttpOnly Cookie，可缺省>
```

查询当前用户：

```http
GET /api/users/me
Authorization: Bearer <accessToken>
```

修改当前用户：

```http
PUT /api/users/me
Authorization: Bearer <accessToken>
Content-Type: application/json

{
  "nickname": "Alice",
  "avatarUrl": null,
  "bio": "AI image creator"
}
```

## 5. 错误码

| HTTP 状态 | 应用错误码 | 默认消息 | 使用场景 |
| --- | ---: | --- | --- |
| `400` | `40000` | 请求参数错误 | JSON 无法解析、字段类型错误或请求格式错误。 |
| `401` | `40100` | 未登录或登录已失效 | Access Token 缺失、无效、类型错误或过期。 |
| `401` | `40101` | 登录账号或密码错误 | 账号不存在或密码错误，不区分具体原因。 |
| `401` | `40102` | 登录会话已失效，请重新登录 | Refresh Token 缺失、无效、过期、会话不存在或轮换失败。 |
| `403` | `40300` | 无权限访问 | 已认证但没有目标资源权限。 |
| `403` | `40301` | 无权使用该媒体文件 | 头像文件不属于当前用户或不可用于头像。 |
| `404` | `40400` | 资源不存在 | 通用资源不存在。 |
| `409` | `40901` | 登录账号已存在 | 用户账号唯一索引冲突。 |
| `422` | `42200` | 请求参数校验失败 | JSON 语法有效但字段不符合规则。 |
| `429` | `42900` | 请求过于频繁，请稍后重试 | 后续限流预留，当前迭代不返回。 |
| `500` | `50000` | 系统繁忙，请稍后重试 | 未处理的服务端异常。 |

前端必须使用 `code` 判断业务分支，不得匹配 `message` 文本。

## 7. 运行配置

| 配置 | 默认值 | 说明 |
| --- | --- | --- |
| `access-token-ttl` | `15m` | Access Token 有效期。 |
| `refresh-token-ttl` | `14d` | Refresh Token 和认证会话最终有效期。 |
| `refresh-token-bytes` | `32` | Refresh Token 随机字节数。 |
| `jwt.secret` | 本地开发默认值 | 生产环境必须通过 `AIVISTA_JWT_SECRET` 覆盖，且不少于 32 字节。 |

| `cookie.secure` | `false` | 生产 HTTPS 环境应设为 `true`。 |

| `argon2.*` | 见配置文件 | 密码哈希参数。 |
| `session-cleanup.enabled` | `true` | 是否启用过期会话清理任务。 |

| `session-cleanup.batch-size` | `500` | 单批删除数量。 |
| `session-cleanup.max-batches` | `10` | 每轮最多删除批次数。 |
| `session-cleanup.fixed-delay-ms` | `3600000` | 清理任务固定延迟，默认 1 小时。 |

## 8. OpenAPI 与人工验收

接口文档地址：

- Knife4j：`http://localhost:8888/api/doc.html`
- Swagger UI：`http://localhost:8888/api/swagger-ui.html`
- OpenAPI JSON：`http://localhost:8888/api/v3/api-docs`

建议人工验收顺序：

1. 注册用户。
2. 重复注册同一 `loginName`，应返回 `40901`。
3. 登录成功，复制 `data.accessToken`，确认响应头存在 `Set-Cookie`。
4. 使用 `Authorization: Bearer <accessToken>` 调用 `GET /api/users/me`。
5. 调用 `PUT /api/users/me` 修改 `nickname` 和 `bio`，`avatarUrl` 传 `null`。
6. 调用 `POST /api/auth/refresh`，确认返回新 Access Token 并重新写 Cookie。
7. 使用新的 Access Token 再次调用 `GET /api/users/me`。
8. 调用 `POST /api/auth/logout`。
9. 退出后再次调用 `POST /api/auth/refresh`，应返回 `40102` 并清 Cookie。
