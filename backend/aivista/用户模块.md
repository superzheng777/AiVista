# AiVista 用户模块

> 对应迭代：迭代 1——用户模块  
> 当前阶段：数据访问、密码与令牌基础已实现；待实现安全过滤链和认证上下文

> **所有 MD 文档均定位为开发约束文档，而非设计说明书。仅记录必须遵循的业务规则、边界、约束和规范，不展开实现细节、接口流程和技术方案；内容应简洁、准确、便于快速阅读，以保证后续开发过程中设计一致、规范统一。文档确定后，再单独讨论具体业务流程和实现方案。**

## 1. 模块边界

本模块负责普通用户身份与个人资料。社交关系、图片资产、发布、点赞、会员、角色与账号注销均在后续独立设计。

## 2. 数据存储

- 数据库：`aivista`
- 用户表：`users`
- 认证会话表：`auth_sessions`
- Flyway 迁移脚本：
  - [src/main/resources/db/migration/V1__create_users.sql](src/main/resources/db/migration/V1__create_users.sql)
  - [src/main/resources/db/migration/V2__create_auth_sessions.sql](src/main/resources/db/migration/V2__create_auth_sessions.sql)
- 规范：InnoDB、utf8mb4、`utf8mb4_unicode_ci`；数据库字段使用下划线命名，Java 字段使用驼峰命名。

| 字段 | 数据库约束 | 开发规则 |
| --- | --- | --- |
| `id` | `BIGINT UNSIGNED`、`NOT NULL`、自增主键。 | 作为用户内部唯一标识。 |
| `login_name` | `VARCHAR(32)`、`NOT NULL`；唯一索引 `uk_users_login_name`；使用 `utf8mb4_unicode_ci` 比较，大小写不敏感。 | 登录账号，不对外展示；注册后不可修改。 |
| `password_hash` | `VARCHAR(255)`、`NOT NULL`。 | 仅保存密码哈希，永不返回客户端；只有注册、修改密码等认证流程可写入。 |
| `nickname` | `VARCHAR(32)`、`NOT NULL`。 | 公开展示昵称；允许重名。 |
| `avatar_url` | `VARCHAR(512)`、可空，默认 `NULL`。 | 仅保存头像地址；地址由通用媒体上传能力提供，更新时应校验文件归属。 |
| `bio` | `VARCHAR(500)`、可空，默认 `NULL`。 | 用户个人简介。 |
| `created_at` | `DATETIME`、`NOT NULL`、默认当前时间。 | 创建后不修改。 |
| `updated_at` | `DATETIME`、`NOT NULL`、默认当前时间；记录更新时自动更新。 | 不由客户端直接传入。 |

## 3. 索引与规则边界

| 主题 | 规范 |
| --- | --- |
| 账号唯一性 | `uk_users_login_name` 保证登录账号全局唯一；`Alice` 与 `alice` 不能同时注册。 |
| 字符比较 | 当前排序规则除大小写外，也可能将部分语言变体字符视为相等；注册接口必须通过字符白名单限制账号字符范围。 |
| 应用层规则 | `login_name` 不可修改、`password_hash` 不可由资料接口写入、头像文件归属校验均为应用层规则。 |
| 非本期字段 | 表中不包含账号状态、角色、社交、资产、会员或注销字段；相关能力出现明确业务需求时再独立设计。 |

## 4. 开发约束

| 主题 | 规范 |
| --- | --- |
| 登录方式 | 注册与登录使用 `login_name + 密码`；其他登录方式后续通过独立身份绑定表扩展。 |
| 可修改资料 | 用户资料可修改 `nickname`、`avatar_url`、`bio`，不得修改 `login_name` 或直接写入 `password_hash`。 |
| 文件存储 | 头像、AI 生成图和发布图片共享通用文件存储能力，但按用途设置独立路径与访问策略。 |
| 接口规范 | 接口响应遵循项目统一 `code`、`message`、`data` 结构；认证、鉴权和输入校验必须在接口设计中明确。限流留待后续安全优化。 |

## 5. 认证令牌设计（已确认）

| 项目 | 规则 |
| --- | --- |
| Access Token 格式 | 使用 JWT，仅包含用户 ID、令牌类型、签发时间、过期时间和唯一标识；不得包含认证会话 ID、用户资料或密码信息。 |
| Access Token 使用方式 | 有效期不得超过 15 分钟；前端仅保存在内存中；通过 `Authorization: Bearer <token>` 访问受保护接口。业务请求只校验 JWT 签名、令牌类型和有效期，不查询认证会话。 |
| Refresh Token 格式 | 使用密码学安全随机字符串，不使用 JWT；有效期 14 天。 |
| Refresh Token 使用方式 | 当前 Web 端保存在 `HttpOnly Cookie` 中，仅用于刷新令牌和退出登录。 |
| 服务端存储与校验 | 服务端不长期保存 Refresh Token 明文，仅保存 SHA-256 哈希和认证会话状态。Access Token 以签名与有效期校验为主；Refresh Token 必须校验会话是否存在且未过期。 |
| 配置管理 | 令牌有效期、Cookie 属性、JWT 签名密钥及密码哈希参数必须统一配置管理，不得在业务代码中硬编码。 |

## 6. 会话、轮换与退出规则（已确认）

| 项目 | 规则 |
| --- | --- |
| Refresh Token 轮换 | 每次成功刷新后必须签发新令牌，并以新哈希替换当前认证会话中的旧哈希；被替换的旧令牌立即失效。 |
| 会话最终期限 | Refresh Token 自首次登录起有效 14 天；轮换不延长认证会话的最终过期时间。 |
| Cookie 属性 | 使用 `HttpOnly`；生产环境 `Secure=true`；`SameSite=Lax`；`Path=/api/auth`；不设置 `Domain`。首次登录的 `Max-Age` 为 14 天，轮换后使用会话剩余有效期。 |
| 部署前提 | 生产环境由同一公开域名下的 Nginx 代理 `/api`；开发环境由 Vite 或其他前端开发服务器代理 `/api`。浏览器默认通过当前页面同源地址访问后端。 |
| 多端登录 | 允许同一用户在多个设备或浏览器同时登录；每次登录创建独立认证会话。 |
| 普通退出登录 | 直接删除当前 Refresh Token 对应的会话记录并清除 Cookie；不影响其他会话；接口保持幂等。 |
| Access Token 退出语义 | 不建立 Access Token 黑名单；退出后已签发的 Access Token 最迟在其 15 分钟有效期结束后失效。 |
| 当前边界 | 当前不实现上一代令牌、重用检测、宽限期或刷新结果幂等缓存。 |

## 7. 认证会话表（已确认）

一条记录表示一个设备或浏览器的独立登录会话。当前只保存正在使用的 Refresh Token 哈希。

| 字段 | 数据库约束 | 开发规则 |
| --- | --- | --- |
| `id` | `BIGINT UNSIGNED`、`NOT NULL`、自增主键。 | 认证会话内部唯一标识。 |
| `user_id` | `BIGINT UNSIGNED`、`NOT NULL`；外键关联 `users.id`，删除规则为 `RESTRICT`。 | 标识会话所属用户。 |
| `client_type` | `VARCHAR(16)`、`NOT NULL`。 | 当前固定为 `WEB`；预留 `APP`，不得作为权限判断依据。 |
| `refresh_token_hash` | `BINARY(32)`、`NOT NULL`；唯一索引 `uk_auth_sessions_refresh_token_hash`。 | 保存 Refresh Token 的 SHA-256 哈希。 |
| `expires_at` | `DATETIME`、`NOT NULL`。 | 登录时确定最终过期时间；刷新不得延长。 |
| `created_at` | `DATETIME`、`NOT NULL`、默认当前时间。 | 会话创建时间。 |
| `updated_at` | `DATETIME`、`NOT NULL`、默认当前时间；记录更新时自动更新。 | 最近一次令牌轮换或其他会话变更时间。 |

| 主题 | 规范 |
| --- | --- |
| 会话索引 | 除主键和 Refresh Token 哈希唯一索引外，为 `user_id`、`expires_at` 建立普通索引。 |
| 登录创建 | 每次登录生成独立 Refresh Token，保存其 SHA-256 哈希，并创建独立会话记录。 |
| 刷新轮换 | 以会话 ID、旧哈希、未过期条件原子替换为新哈希；必须确认恰好更新一条记录。 |
| 主动退出 | 删除当前 Refresh Token 哈希对应的会话记录；无论记录是否存在，均清除 Cookie 并按成功处理。 |
| 自然过期 | 过期会话立即失去认证能力，记录保留 7 天后由清理任务分批删除。 |
| 当前非目标 | 不记录撤销时间、撤销原因、上一代令牌或设备详情。 |

## 8. 注册、登录、刷新与退出流程（已确认）

### 8.1 注册

| 项目 | 规则 |
| --- | --- |
| 输入 | `login_name`、密码、`nickname`。`avatar_url`、`bio` 不作为注册必填项。 |
| 账号格式 | 服务端先去除 `login_name` 首尾空白，再以处理后的值完成校验和入库。长度为 4～32，仅允许 ASCII 字母、数字和下划线，必须以字母开头；不自动改变字母大小写。 |
| 密码规则 | 按 Unicode 码点计算长度，限制为 8～64 个字符，至少包含一个字母和一个数字，可包含空格和特殊字符；不得仅由空白字符组成。密码是原样认证凭证，服务端不得去除空白或改变大小写，不得记录、回显或写入日志。 |
| 密码存储 | 使用 Argon2id 哈希；算法参数统一配置，哈希结果保存到 `password_hash`。 |
| 昵称规则 | 服务端先去除 `nickname` 首尾空白，再以处理后的值入库；按 Unicode 码点计算长度，限制为 1～32 个字符，允许重名和内部空格，禁止控制字符。 |
| 重复账号 | 以数据库唯一索引作为最终约束；冲突统一返回账号已存在，不暴露数据库异常。 |
| 事务边界 | 注册事务只创建 `users` 记录，不创建认证会话。 |
| 注册后行为 | 注册成功后不自动登录；客户端进入登录流程。 |

### 8.2 登录

| 项目 | 规则 |
| --- | --- |
| 输入与查询 | 使用 `login_name + 密码`；`login_name` 去除首尾空白后查询，查询遵循数据库大小写不敏感规则；密码保持原样。 |
| 失败响应 | 账号不存在与密码错误使用统一错误，不暴露账号是否存在。 |
| 密码校验 | 使用统一 `PasswordEncoder` 校验，不直接比较哈希字符串。 |
| 成功处理 | 生成 Refresh Token 及其 SHA-256 哈希，创建 `auth_sessions` 记录，再签发 Access Token。认证会话仅用于 Refresh Token 生命周期，不参与业务请求的 Access Token 校验。 |
| 原子性 | 用户校验、会话创建和令牌签发过程失败时不得遗留不完整会话；数据库事务提交后再向客户端写入凭证。 |
| 响应 | Access Token 放在响应体；Refresh Token 通过 HttpOnly Cookie 写入。 |

### 8.3 刷新

| 项目 | 规则 |
| --- | --- |
| 输入 | 仅从约定的 HttpOnly Cookie 读取 Refresh Token，不接受请求体或查询参数中的 Web Refresh Token。 |
| 定位 | 对令牌计算 SHA-256，通过唯一索引定位当前会话。 |
| 校验 | 会话必须存在且 `expires_at > 当前时间`；用户记录必须存在。 |
| 轮换 | 生成新 Refresh Token，并以旧哈希和未过期条件原子替换为新哈希；更新行数必须为 1。 |
| 成功响应 | 返回新 Access Token，并以会话剩余有效期重新写入 Refresh Token Cookie。 |
| 认证失效 | Cookie 缺失、令牌无效、会话不存在、已过期或原子更新失败时统一返回 `401`，并清除 Cookie。 |
| 并发控制 | 前端使用 Single Flight 合并并发刷新；认证接口的 `401` 不得再次触发刷新。 |
| 网络异常 | 网络或服务异常不直接清空前端登录状态；若服务端已完成轮换但响应丢失，后续旧令牌刷新失败时需要重新登录。 |

### 8.4 退出

| 项目 | 规则 |
| --- | --- |
| 输入 | 从 HttpOnly Cookie 读取 Refresh Token；不依赖 Access Token 是否仍然有效。 |
| 服务端处理 | Cookie 存在时计算哈希并删除对应会话；记录不存在也按成功处理。 |
| Cookie 清除 | 响应必须使用与写入时一致的 `Path`、`SameSite`、`Secure` 等关键属性，并设置 `Max-Age=0` 清除 Cookie。 |
| 幂等性 | 缺少 Cookie、重复退出或会话已过期均返回成功。 |
| 多端影响 | 仅退出当前会话，不删除该用户的其他会话。 |
| Access Token | 前端同时清除内存中的 Access Token；已签发 JWT 不进入黑名单。 |

## 9. 清理规则（已确认）

- 清理对象：`expires_at < 当前时间 - 7 天` 的自然过期会话。
- 清理方式：定时任务按主键或过期时间分批删除，不执行无界单次删除。
- 清理任务失败不得影响认证主流程，下次调度继续处理。
- 主动退出的会话已直接删除，不进入自然过期保留期。

## 10. App 兼容边界

- 当前迭代只实现 Web 端认证，React Native App 在 Web 端完成后开发。
- `client_type` 预留 `APP`，但客户端提交的类型不构成可信身份或权限依据。
- App 后续应使用 iOS Keychain、Android Keystore 等系统安全存储保存 Refresh Token。
- App 的 Refresh Token 输入输出、应用完整性校验和设备管理在 App 开发前单独设计。
- Web 端继续使用 HttpOnly Cookie，不因 App 接入而允许 Web JavaScript 读取 Refresh Token。

## 11. REST API 契约（已确认）

- 所有接口沿用服务端 `/api` 上下文路径；当前为内部一期 API，暂不增加 `/v1` 版本前缀。
- 请求与响应 JSON 字段使用驼峰命名；数据库字段继续使用下划线命名。
- 成功和失败响应均使用统一 `code`、`message`、`data` 结构；注册成功返回 `201`，其余成功接口返回 `200`。
- Access Token 通过 `Authorization: Bearer <token>` 传递；Refresh Token Cookie 名称固定为 `refresh_token`。

| 功能 | 方法与路径 | 认证要求 | 成功响应 |
| --- | --- | --- | --- |
| 注册 | `POST /api/auth/register` | 无 | `201`，返回 `UserProfileResponse`；不自动登录。 |
| 登录 | `POST /api/auth/login` | 无 | `200`，返回 `LoginResponse` 并写入 Refresh Token Cookie。 |
| 刷新 | `POST /api/auth/refresh` | `refresh_token` Cookie | `200`，返回 `TokenResponse` 并轮换 Refresh Token Cookie。 |
| 退出 | `POST /api/auth/logout` | Cookie 可缺失；不要求 Access Token | `200`，清除 Cookie，`data` 为 `null`。 |
| 查询个人资料 | `GET /api/users/me` | Bearer Access Token | `200`，返回 `UserProfileResponse`。 |
| 修改个人资料 | `PUT /api/users/me` | Bearer Access Token | `200`，返回更新后的 `UserProfileResponse`。 |

个人资料修改使用 `PUT` 完整提交全部可修改字段，以明确区分保留值与清空值；不使用需要区分“未提交”和“显式 `null`”的局部更新语义。

## 12. DTO 约束（已确认）

| DTO | 字段 | 规则 |
| --- | --- | --- |
| `RegisterRequest` | `loginName`、`password`、`nickname` | 校验和规范化遵循注册流程；不得包含头像、简介或认证字段。 |
| `LoginRequest` | `loginName`、`password` | 账号去除首尾空白；密码保持原样。 |
| `UpdateProfileRequest` | `nickname`、`avatarUrl`、`bio` | 三个字段均须出现在请求中；`nickname` 不可为 `null`，`avatarUrl` 和 `bio` 可为 `null`，表示清空。不得包含其他用户字段。 |
| `TokenResponse` | `accessToken`、`tokenType`、`expiresIn` | `tokenType` 固定为 `Bearer`；`expiresIn` 单位为秒；不得包含 Refresh Token。 |
| `LoginResponse` | `accessToken`、`tokenType`、`expiresIn`、`user` | `user` 使用 `UserProfileResponse`；不得包含 Refresh Token 或密码信息。 |
| `UserProfileResponse` | `id`、`loginName`、`nickname`、`avatarUrl`、`bio`、`createdAt`、`updatedAt` | 仅用于当前用户本人；不得返回 `passwordHash` 或认证会话信息。 |

| 资料字段 | 校验规则 |
| --- | --- |
| `nickname` | 去除首尾空白后按 Unicode 码点计算 1～32 个字符，禁止控制字符。 |
| `avatarUrl` | 可为 `null`；非空时最长 512 个字符，且必须指向当前用户有权使用的平台媒体文件。媒体能力落地前，不得接受未经归属校验的任意外部地址。 |
| `bio` | 可为 `null`；非空时按 Unicode 码点计算不超过 500 个字符，禁止控制字符。 |

## 13. HTTP 状态与统一错误码（已确认）

| HTTP 状态 | 应用错误码 | 默认消息 | 使用场景 |
| --- | ---: | --- | --- |
| `400` | `40000` | 请求参数错误 | JSON 无法解析、字段类型错误或请求格式错误。 |
| `401` | `40100` | 未登录或登录已失效 | Access Token 缺失、无效、类型错误或过期。 |
| `401` | `40101` | 登录账号或密码错误 | 账号不存在或密码错误；不得通过响应区分。 |
| `401` | `40102` | 登录会话已失效，请重新登录 | Refresh Token 缺失、无效、过期、会话不存在或原子轮换失败。 |
| `403` | `40300` | 无权限访问 | 已认证但没有目标资源权限。 |
| `403` | `40301` | 无权使用该媒体文件 | 头像文件不属于当前用户或不可用于头像。 |
| `404` | `40400` | 资源不存在 | 通用资源不存在。 |
| `409` | `40901` | 登录账号已存在 | 用户账号唯一索引冲突。 |
| `422` | `42200` | 请求参数校验失败 | 语法有效但字段未通过输入校验。 |
| `429` | `42900` | 请求过于频繁，请稍后重试 | 为后续接口限流预留；当前迭代不返回。 |
| `500` | `50000` | 系统繁忙，请稍后重试 | 未处理的服务端异常。 |

- 前端必须使用应用错误码判断业务分支，不得匹配错误消息文本。
- 参数校验统一使用 `42200`，当前不为单条字段规则增加错误码。
- 数据库约束或内部异常必须转换为稳定错误码，不得暴露 SQL、堆栈或令牌信息。
- Refresh Token 认证失败返回 `40102` 时必须清除 Cookie；退出接口的幂等场景不返回错误。

## 14. 后续安全优化：接口限流

当前迭代不实现接口限流，不引入限流组件、计数存储或来源地址识别逻辑；以下内容仅作为后续安全优化基线，不纳入当前代码、OpenAPI 和测试验收范围。后端对公网开放前必须重新评估并完成登录防暴力破解与刷新接口保护。

后续实现时，限流阈值必须配置化，不得写死在 Controller 或业务代码中；任一限流条件命中应返回 `42900` 和 `Retry-After` 响应头。

| 接口 | 限流维度 | 初始阈值 |
| --- | --- | --- |
| 注册 | 来源地址 | 5 次/小时。 |
| 登录 | 规范化后的 `loginName` | 5 次/5 分钟。 |
| 登录 | 来源地址 | 30 次/5 分钟。 |
| 刷新 | Refresh Token 哈希 | 30 次/分钟。 |
| 刷新 | 来源地址 | 60 次/分钟。 |

日志与限流键不得保存或输出密码、Access Token、Refresh Token 或可还原的敏感凭证。

- 单实例可使用进程内计数；部署多个实例前必须使用 Redis 等共享存储。
- 来源地址只能从服务端直接连接信息或受控反向代理提供的可信转发头中提取，不得无条件信任客户端传入的 `X-Forwarded-For`。
- Nginx 与 Spring 的转发头处理配置必须配套；未处于受信代理链路时忽略转发头。

## 15. OpenAPI 描述（已确认）

- 使用 `用户认证` 与 `个人资料` 两个接口分组标签。
- 全局声明名称为 `bearerAuth` 的 HTTP Bearer/JWT 安全方案；个人资料接口必须声明该安全要求。
- 注册和登录接口不声明认证；刷新和退出不声明 Bearer 认证，并分别将 `refresh_token` Cookie 标记为必需和可选。
- 每个接口必须提供摘要、请求 DTO、成功响应示例、统一响应结构及实际可能出现的错误响应。
- 认证接口文档必须明确：注册不自动登录；Refresh Token 仅通过 HttpOnly Cookie 传递；刷新会轮换令牌；退出保持幂等。

### 16.1 数据访问框架选择

- 当前项目统一使用 MyBatis-Flex，不同时引入 MyBatis-Plus 或其他 MyBatis 增强框架。
- 使用适配 Spring Boot 4 的 MyBatis-Flex Starter，并显式引入 Spring JDBC/DataSource 自动配置所需依赖。
- 使用 MyBatis-Flex APT 生成表定义；接入时必须验证其注解处理器与项目现有 Lombok 注解处理器能够同时完成主代码和测试代码编译。
- 普通单表增删改查可使用 MyBatis-Flex Mapper 与 QueryWrapper；涉及认证安全边界、并发条件或更新行数判断的操作必须保持 SQL 条件明确且可测试。
- Refresh Token 原子轮换、过期会话分批清理等关键操作允许使用 Mapper 注解 SQL 或 XML SQL，不为追求链式写法牺牲语义清晰度。
- 当前不启用逻辑删除、Active Record、多租户、自动关联查询等未被本迭代需要的扩展能力。

### 16.2 数据库迁移与时间规范

- 使用 Flyway 作为数据库结构的唯一迁移入口；正式接入时将现有用户表和认证会话表 SQL 转换为按版本命名的迁移脚本，不再保留并行的手工初始化流程。
- 开发、测试和部署环境启动时均通过相同迁移脚本建立或升级结构；已经执行的迁移不得修改，变更必须新增后续版本脚本。
- 认证时间统一以 UTC 为基准；Java 认证领域字段使用 `Instant`，JDBC 连接时区固定为 UTC，对外时间使用带时区的 ISO 8601 格式。
- Token 签发、会话过期和数据库条件更新必须使用同一时间基准，不依赖服务器本地时区。

### 16.3 JWT 与密码参数

- Access Token 当前使用 `HS256` 签名；签名密钥至少为 32 字节的密码学安全随机值，通过环境变量 `AIVISTA_JWT_SECRET` 或等价的外部安全配置提供。
- 生产环境不得在代码或配置文件中提供默认 JWT 密钥；密钥缺失或不符合长度要求时，应用必须启动失败。测试环境使用独立测试密钥。
- Argon2id 初始采用 Spring Security 提供的安全默认参数，盐长度、哈希长度、并行度、内存成本和迭代次数统一配置，不在业务代码中硬编码。
- 上线前必须在目标部署规格上测试密码哈希耗时，并在安全性与登录吞吐之间调整参数；调整不得破坏已有密码哈希的校验能力。

### 16.4 当前运行边界

- 通用媒体归属校验落地前，个人资料接口允许 `avatarUrl` 为 `null`，但不得接受新的非空头像地址；不得临时放开任意外部 URL。

### 16.5 测试环境

- 数据访问和认证集成测试使用真实 MySQL 行为，不使用 H2 替代 `utf8mb4_unicode_ci`、`BINARY(32)`、无符号主键、约束及时间行为。

- 集成测试必须执行与应用相同的 Flyway 迁移，并覆盖重复账号、令牌轮换并发、过期判断、退出幂等和用户资料越权场景。
- 快速单元测试使用 `mvnw.cmd test` 执行，不得依赖数据库或 Docker；MySQL 集成测试使用 `mvnw.cmd verify` 单独执行。认证 API 完成后，在快速测试层补充不依赖外部数据库的接口契约测试，在集成测试层保留真实 MySQL 端到端验证。
