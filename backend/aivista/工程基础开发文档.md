# AiVista 工程基础开发文档

> 对应迭代：迭代 0——工程基础（已完成）

## 1. 项目入口与常用命令

项目为 Spring Boot 后端，默认地址为 `http://localhost:8888`，所有业务接口统一使用 `/api` 前缀。

```powershell
# 当前可用：运行测试
mvn test

# 当前可用：启动应用
mvn spring-boot:run
```

`mvnw.cmd` 是项目 Maven Wrapper。请在 `backend/aivista` 目录优先使用 `.\mvnw.cmd test` 和 `.\mvnw.cmd spring-boot:run`，以统一团队使用的 Maven 版本。Wrapper 已验证可运行；首次使用会按其配置下载 Maven 发行版和缺失依赖。

## 2. 已有基础能力

| 能力 | 使用方式 / 约定 |
| --- | --- |
| 健康检查 | `GET /api/health`；用于本地启动后确认服务可用。 |
| 接口文档 | Knife4j：`/api/doc.html`；Swagger UI：`/api/swagger-ui.html`；OpenAPI JSON：`/api/v3/api-docs`。新增 Controller 接口时应补充 OpenAPI 注解。 |
| 统一响应 | Controller 成功响应使用 `ResponseUtils.success(data)`，结构固定为 `code`、`message`、`data`。 |
| 业务异常 | 可预期的业务失败使用 `BusinessException` 和 `ErrorCode` 抛出，由全局异常处理器统一转换为 HTTP 响应。 |
| 参数校验 | 请求 DTO 使用 Jakarta Validation 注解；校验失败由全局异常处理器统一返回。 |
| 请求代理 | 生产环境由同一公开域名下的 Nginx 代理 `/api`；开发环境优先由 Vite 或其他前端开发服务器代理 `/api`，浏览器通过页面同源地址访问后端。 |
| 跨域 | 仅在无法使用开发代理时直接跨域访问后端；允许来源在 `application.yaml` 的 `app.cors.allowed-origins` 中维护。本地 Next.js 默认允许 `http://localhost:3000` 与 `http://127.0.0.1:3000`，可由 `AIVISTA_CORS_ALLOWED_ORIGINS` 覆盖。携带认证 Cookie 时必须使用明确来源并显式允许凭证，不得使用通配来源。 |
| 日志 | 未处理异常由全局异常处理器记录；业务代码应使用 SLF4J 记录必要上下文，避免记录密码、令牌等敏感信息。 |

## 3. 代码位置

| 位置 | 职责 |
| --- | --- |
| 各业务模块的 `api` 包 | HTTP 接口层，例如 `auth/api`、`user/api`、`health/api`。 |
| `common/response` | 统一响应对象与创建工具。 |
| `common/exception` | 错误码、业务异常和全局异常处理。 |
| `common/request` | 通用请求 DTO 基类。 |
| `config` | 跨域和 OpenAPI 等工程配置。 |
| `application.yaml` | 服务端口、上下文路径、跨域和接口文档配置。 |

## 4. 开发约定

- 每次开发同步修改代码和相关文档。
- 新模块开始前，先补充该模块的详细设计文档；数据库表、API、权限和测试规则在该文档中确定。
- 新增接口遵循统一响应和异常处理约定，不在 Controller 中自行拼装错误响应。
- 迭代状态变化时，同步更新《后端项目开发文档》。

## 5. 部署与联调约定

- 生产环境由 Nginx 提供前端静态资源并将同一域名下的 `/api` 请求代理至 Spring Boot 服务，客户端不直接访问后端内部地址。
- 开发环境由 Next.js 将 `/api` 请求 rewrite 至 `http://localhost:8888` 并保留原始 `/api` 路径；代理应保留 Cookie 和必要的转发头。
- Spring Boot 应只信任受控代理写入的转发头；生产环境的 HTTPS、真实客户端地址和协议由 Nginx 统一传递。
- 若部署方式改为前后端跨站，必须重新评估 CORS、Cookie `SameSite`/`Secure` 属性及 CSRF 防护，不得直接沿用当前认证配置。
