## 2. 项目结构

```text
frontend/aivista/
├── src/app/                 # App Router 路由与页面
├── src/widgets/             # 跨页面界面区块，例如应用壳层
├── src/features/            # 业务能力，例如认证、资料编辑
├── src/entities/            # 前端领域模型，例如用户
├── src/shared/              # HTTP、统一响应等无业务语义能力
├── src/components/          # 已初始化的 shadcn/ui 与待迁移展示组件
├── src/lib/                 # 通用工具
├── public/                  # 静态资源
├── components.json          # shadcn/ui 配置
├── next.config.ts           # Next.js 与开发代理配置
└── .env.example             # 本地环境变量示例
```

### 2.1 分层与依赖约束

业务代码采用轻量 Feature-first 结构，不预建无内容目录；实际产生代码时按以下职责归位：

```text
src/
├── app/                    # App Router 路由、根布局与页面组合
├── widgets/                # 跨页面的完整界面区块，例如 app-shell
├── features/               # 用户可执行的业务能力，例如 auth
│   └── <feature>/{api,model,ui,lib}
├── entities/               # 业务实体及前端领域模型，例如 user
├── shared/                 # 无业务语义的 API、通用 UI、工具与配置
└── lib/                    # 现有通用工具，后续按需迁入 shared/lib
```

- 依赖只允许从上向下：`app → widgets → features → entities → shared`；同层业务功能不直接互相导入。
- `shared/api` 只处理 HTTP 与统一响应；`features/*/api` 定义接口 DTO 并映射为领域模型；页面和 UI 只使用 `entities` 中的领域模型。
- 所有对外资源 ID 使用字符串。当前用户模型拆为 `CurrentUser`（含登录账号、简介等私有资料）与 `PublicUser`（可公开展示的信息），禁止新增万能 `User` 类型。
- 认证 UI 通过 `useSession` 与 `useAuthDialog` 消费稳定接口；Zustand、Axios 等具体实现仅位于认证功能内部。

当前已落地的基础目录：`entities/user/model`、`features/auth/{api,model,ui}`、`shared/api`、`widgets/app-shell/ui`。

## 3. 启动与验证

在 `frontend/aivista` 目录执行：

```powershell
pnpm dev       # 启动开发服务，默认 http://localhost:3000
pnpm lint      # 执行代码检查
pnpm build     # 生成生产构建
pnpm start     # 启动生产构建
```

当前环境若因 PowerShell 执行策略无法直接运行 `pnpm`，使用 `pnpm.cmd` 替代。

## 4. 后端接口与代理约定

浏览器始终调用同域相对路径 `/api/*`，不在业务代码中写死 Spring Boot 地址。

```text
开发环境：浏览器 → Next.js /api/* rewrite → Spring Boot http://localhost:8888/api/*
生产环境：浏览器 → Nginx /api/* proxy → Spring Boot
```

- 开发代理地址由 `BACKEND_ORIGIN` 配置，默认值为 `http://localhost:8888`，且不带 `/api` 后缀。
- 复制 `.env.example` 为 `.env.local` 后可覆盖本地代理地址；`.env.local` 不提交到仓库。
- Next.js 不实现业务 API；Spring Boot 是唯一业务后端。
- 服务端渲染的公开页面通过仅服务端可见的后端地址读取数据；浏览器客户端仍调用相对路径 `/api/*`。

## 5. 渲染与认证约定

- 灵感流、公开作品详情和公开用户主页等公开页面优先使用服务端渲染或静态生成，以支持 SEO。
- 个人资产、生成、个人资料等已登录页面以客户端请求为主。
- Access Token 只保存在浏览器内存；业务请求使用 `Authorization: Bearer <accessToken>`。
- 后端返回的资源 ID 一律按字符串处理，避免 MySQL `BIGINT UNSIGNED` 在 JavaScript 中发生精度丢失。
- Refresh Token 由后端以 HttpOnly Cookie 管理；收到后端 `40100` 时执行单飞刷新并重放请求，收到 `40102` 时清除会话并打开登录卡片。
- 认证行为必须遵循后端 [用户模块.md](../../backend/aivista/用户模块.md) 的接口与错误码约定。

## 6. 当前页面与交互

详细布局、路由、交互状态与验收标准见 [首页与导航设计.md](首页与导航设计.md)。当前已具备：

- 固定侧栏：灵感、生成、资产三个入口；移动端切换为底部导航。
- 首页：创作入口区与灵感作品展示区的布局骨架。
- 登录：右下角登录入口以及未登录的受保护操作均打开当前页面上的登录/注册卡片；不存在独立登录路由；支持 Refresh Cookie 会话恢复。
- 登录后：侧栏登录入口显示头像或昵称首字，点击进入 `/profile`；个人主页可编辑昵称、个人简介并退出登录。头像修改等待媒资上传能力。
- 生成与资产：已有独立路由和页面骨架，等待后端迭代 2 至 4 接入真实数据。

## 8. App 规划

Web 一期完成后再开发 React Native App。App 使用 Expo + TypeScript，复用接口类型、校验规则和业务模型，不强行复用 Web UI 组件。
