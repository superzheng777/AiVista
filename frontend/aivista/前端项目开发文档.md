> 项目状态：前端迭代 0、迭代 1 已完成；生成、资产与灵感真实数据仍待对应后端迭代接入。 
> 维护规则：每次开发必须同步修改代码和相关文档；页面、接口契约或前端架构发生变化时，在同一次提交中更新对应文档。

### 1.1 主题颜色约定

页面结构颜色必须使用语义 token，而非直接使用 `white`、`slate-*` 等中性色。常用映射如下：

| UI 用途 | Tailwind token |
| --- | --- |
| 页面背景与正文 | `bg-background`、`text-foreground` |
| 卡片与卡片正文 | `bg-card`、`text-card-foreground` |
| 弹出层 | `bg-popover`、`text-popover-foreground` |
| 弱化区域与次级文字 | `bg-muted`、`text-muted-foreground` |
| 边框与输入框 | `border-border`、`bg-background` |
| 主操作按钮 | `bg-primary`、`text-primary-foreground` |

浅色与深色的实际颜色由 `globals.css` 的 CSS 变量统一定义。天空蓝品牌色、作品占位渐变、警示色和危险色具有明确语义，可继续使用具体颜色类。

## 2. 项目结构

```text
frontend/aivista/
├── src/app/                 # App Router 路由与页面
├── src/widgets/             # 跨页面界面区块，例如应用壳层
├── src/features/            # 业务能力，例如认证、资料编辑
├── src/entities/            # 前端领域模型，例如用户
├── src/shared/              # HTTP、统一响应等无业务语义能力
├── src/components/          # 已初始化的 shadcn/ui 与待迁移展示组件
├── src/lib/                 # 通用工具
├── public/                  # 静态资源
├── components.json          # shadcn/ui 配置
├── next.config.ts           # Next.js 与开发代理配置
└── .env.example             # 本地环境变量示例
```

### 2.1 分层与依赖约束

业务代码采用轻量 Feature-first 结构，不预建无内容目录；实际产生代码时按以下职责归位：

```text
src/
├── app/                    # App Router 路由、根布局与页面组合
├── widgets/                # 跨页面的完整界面区块，例如 app-shell
├── features/               # 用户可执行的业务能力，例如 auth
│   └── <feature>/{api,model,ui,lib}
├── entities/               # 业务实体及前端领域模型，例如 user
├── shared/                 # 无业务语义的 API、通用 UI、工具与配置
└── lib/                    # 现有通用工具，后续按需迁入 shared/lib
```

- 依赖只允许从上向下：`app → widgets → features → entities → shared`；同层业务功能不直接互相导入。
- `shared/api` 只处理 HTTP 与统一响应；`features/*/api` 定义接口 DTO 并映射为领域模型；页面和 UI 只使用 `entities` 中的领域模型。
- 所有对外资源 ID 使用字符串。当前用户模型拆为 `CurrentUser`（含登录账号、简介等私有资料）与 `PublicUser`（可公开展示的信息），禁止新增万能 `User` 类型。
- 认证 UI 通过 `useSession` 与 `useAuthDialog` 消费稳定接口；Zustand、Axios 等具体实现仅位于认证功能内部。

当前已落地的基础目录：`entities/user/model`、`features/auth/{api,model,ui}`、`features/theme/model`、`features/account-menu/ui`、`shared/api`、`widgets/app-shell/ui`。

## 3. 启动与验证

在 `frontend/aivista` 目录执行：

```powershell
pnpm dev       # 启动开发服务，默认 http://localhost:3000
pnpm lint      # 执行代码检查
pnpm build     # 生成生产构建
pnpm start     # 启动生产构建
```

当前环境若因 PowerShell 执行策略无法直接运行 `pnpm`，使用 `pnpm.cmd` 替代。

## 4. 后端接口与代理约定

浏览器始终调用同域相对路径 `/api/*`，不在业务代码中写死 Spring Boot 地址。

```text
开发环境：浏览器 → Next.js /api/* rewrite → Spring Boot http://localhost:8888/api/*
生产环境：浏览器 → Nginx /api/* proxy → Spring Boot
```

- 开发代理地址由 `BACKEND_ORIGIN` 配置，默认值为 `http://localhost:8888`，且不带 `/api` 后缀。
- 复制 `.env.example` 为 `.env.local` 后可覆盖本地代理地址；`.env.local` 不提交到仓库。
- Next.js 不实现业务 API；Spring Boot 是唯一业务后端。
- 服务端渲染的公开页面通过仅服务端可见的后端地址读取数据；浏览器客户端仍调用相对路径 `/api/*`。

## 5. 渲染与认证约定

- 灵感流、公开作品详情和公开用户主页等公开页面优先使用服务端渲染或静态生成，以支持 SEO。
- 个人资产、生成、个人资料等已登录页面以客户端请求为主。
- `SessionProvider` 在应用启动时加载当前用户资料；受保护路由的匿名跳转由 `AppShell` 统一处理，不引入页面级认证边界。
- Access Token 只保存在浏览器内存；业务请求使用 `Authorization: Bearer <accessToken>`。
- 后端返回的资源 ID 一律按字符串处理，避免 MySQL `BIGINT UNSIGNED` 在 JavaScript 中发生精度丢失。
- Refresh Token 由后端以 HttpOnly Cookie 管理；收到后端 `40100` 时执行单飞刷新并重放请求。刷新接口明确返回 `40102` 时才清除会话；网络超时、断网和服务端异常不判定为退出登录。
- 认证行为必须遵循后端 [用户模块.md](../../backend/aivista/用户模块.md) 的接口与错误码约定。

### 5.1 会话恢复与失效处理

浏览器完整刷新后，`SessionProvider` 只调用 `GET /api/users/me` 恢复侧边栏用户资料。浏览器内存没有 Access Token 时，该请求返回 `40100`，由 Axios 响应拦截器的唯一 Single Flight 调用 `POST /api/auth/refresh`，成功后为原请求补充新 Token 并重放。初始化代码不直接调用刷新接口。

认证状态与页面行为：

- `loading`：受保护页面保留 `AppShell` 并显示加载占位；私有数据请求交由全局拦截器完成刷新与重放。
- `authenticated`：展示受保护页面内容并请求业务数据。
- `anonymous`：仅表示刷新接口返回 `40102` 或用户主动退出。`AppShell` 发现当前为受保护路由时使用 `router.replace("/")` 返回首页，不打开登录弹窗。
- `error`：表示网络、超时或服务端异常，保留当前页面并提供重新连接，不清空为匿名状态。

公共页面立即加载公开数据。点赞、关注等单个受保护动作在确认会话失效后停留当前页面，并由对应动作打开登录弹窗；匿名用户首次访问首页不自动打开登录弹窗。所有 `40100` 的刷新与重放均由 Axios 拦截器的唯一 Single Flight 负责，包括首次恢复用户资料和已登录期间 Token 过期。

## 6. 当前页面与交互

详细布局、路由、交互状态与验收标准见 [首页与导航设计.md](首页与导航设计.md)。当前已具备：

- 固定侧栏：灵感、生成、资产三个入口；移动端切换为底部导航。
- 首页：创作入口区与灵感作品展示区的布局骨架。
- 登录：右下角登录入口以及未登录的受保护操作均打开当前页面上的登录/注册卡片；不存在独立登录路由；支持 Refresh Cookie 会话恢复。
- 登录后：侧栏登录入口显示头像或昵称首字，点击进入 `/profile`；个人主页可编辑昵称、个人简介并退出登录。头像修改等待媒资上传能力。
- 账户菜单与主题：桌面端头像悬停展示个人主页、主题、AiVista APP 与退出登录菜单；主题偏好仅保存于浏览器本地，退出登录统一经过二次确认弹窗。
- 生成与资产：已有独立路由和页面骨架，等待后端迭代 2 至 4 接入真实数据。

### 7.1 前端迭代 2 对接约定

- 当前仅接入普通文生图；`imageCount` 仅允许 `1`～`6`，服务端仍是最终校验方。局部重绘在后端开放后另行设计输入与结果交互。
- 进入生成页先获取全部已持久化会话列表，列表不设数量上限；默认展示不属于列表的空白创作框。该空白框首次提交时不传 `sessionId`，后端原子创建会话和任务并在 `202 Accepted` 响应中返回 `sessionId`；前端据此更新侧栏并切换当前会话。仅当该空白框成功提交后，才允许用户从侧栏点击“新建会话”开启下一个空白创作框。会话默认标题由首条提示词截取生成，可通过会话标题接口修改。
- 同一会话会自动继承历史正向提示词，历史负向提示词不继承。生成工作台必须明确提示用户“建议一个会话只围绕一个图片主题创作；更换主题时请新建会话”。
- 创建任务时为每次用户主动提交生成新的 UUID v4 幂等键；网络重试、超时重试和重复点击必须复用原键。成功接收后按 `202 Accepted` 返回的任务概要展示排队状态并禁用该会话的生成按钮；同一会话存在活动任务时不允许重复提交。前端允许从会话列表中的不同会话发起最多 2 个活动任务；后端通过活动槽位对跨页面和设备的上限作最终校验。达到上限时，前端提示等待任一任务完成；后端返回 `409 Conflict` 与 `USER_GENERATION_CONCURRENCY_LIMIT`。
- 每日图片额度为 12 张；任务创建后按请求张数立即占用，失败、取消或部分成功均不返还。新建任务按单用户每分钟 2 次、同一公网 IP 下全部账号合计每分钟 6 次限流；`429` 时前端读取 `Retry-After` 并展示可再次尝试的时间。相同幂等请求重放不重复消耗限流或额度。
- 首次生成前，前端调用 `GET /api/users/me/consents/generation` 获取当前第三方处理规则版本、完整文案和同意状态；用户主动确认后携带该 `policyVersion` 调用 `POST /api/users/me/consents/generation`。规则升级后必须重新确认。文案需说明提示词和上下文会发送至阿里云百炼、生成内容会被保存并可在后续资产功能中删除。
- 任务结果只展示成功转存的图片；若部分失败或取消，展示后端返回的成功、失败、取消数量，不展示失败图片占位或顺序。
- 每个生成页使用携带 `Authorization: Bearer <accessToken>` 的 Fetch 流式 SSE 建立一条事件流，不将 Access Token 放入 URL；同一用户最多保留 3 条连接。离开生成页时主动关闭连接，关闭页面不取消任务。收到事件后仅依据更大的 `taskVersion` 更新本地状态。
- SSE 事件不携带签名 URL 或完整任务详情。首次进入、重连以及收到终态事件后，调用任务查询接口以 MySQL 状态为准并获取短期图片访问地址。任务详情响应不做浏览器 HTTP 缓存；前端在 SPA 内存中缓存已加载任务、签名 URL 和过期时间，在 URL 有效时重复打开同一会话直接复用，临近或超过过期时间才重新查询。签名 URL 不得写入本地存储、错误监控、埋点或页面 URL；页面设置 `Referrer-Policy: no-referrer`。
- 图片、会话和生成资产的删除能力属于前端迭代 3、4；生成工作台不实现删除入口。
- 发布能力属于前端迭代 4。发布确认界面必须清晰说明：将公开该图片所用的完整用户正向提示词历史及本次负向提示词；不得展示内部上下文、服务商参数、工作流配置或系统提示词。

## 8. App 规划

Web 一期完成后再开发 React Native App。App 使用 Expo + TypeScript，复用接口类型、校验规则和业务模型，不强行复用 Web UI 组件。
